# Java Developer Guide 知识库改造方案

## 一、现状诊断

### 1.1 内容规模

| 模块 | 文件数 | 粒度 |
|------|--------|------|
| 10_基础 | 282 | 每个知识点 = 1个文件夹(专题.md + QA.md) |
| 11_并发 | 386 | 同上 |
| 12_JVM | 49 | 同上，但未完成 |
| 15_框架(SpringBoot) | 232 | 文件平铺，无子文件夹 |
| **合计** | **~950** | |

### 1.2 核心问题

**问题1：知识索引是"名词表"，不是"问题链"**

目录按技术概念平铺（线程与进程、volatile、CAS……），但面试是按**问题场景**考察的。面试官不会问"说一下synchronized锁升级"，他会问"线上服务TPS上不去，你排查发现是锁竞争导致的，怎么定位和优化的？"——这个问题横跨锁升级、AQS、线程池、JVM多个模块。

**问题2：专题文档和QA内容高度重叠**

同一个"锁升级过程"，专题文档里写了400行，QA里又写了380行，表格一样、代码一样、结论一样。两份文档都变成了"半深不浅"的状态。

**问题3：模块间颗粒度不一致**

- 并发模块：18个子模块 × 每个知识点一个文件夹 = 极细
- SpringBoot模块：4个子模块 × 文件平铺 = 极粗
- 复习时体验割裂

**问题4：缺少"串联层"**

没有跨模块的场景文档把知识点串成线。现在的知识地图是清单，适合"查漏"，不适合"建立理解"。

## 二、改造方案

### 2.1 总体思路

```

```

### 2.3 各层职责（核心区别）

| 层级 | 职责 | 写什么 | 不写什么 |
|------|------|--------|----------|
| **场景文档** `_面试场景/` | 模拟面试实战 | 面试官考察意图、跨模块知识串联、回答思路、追问链、项目经验切入 | 不贴大段源码，不重复基础概念，通过链接引用知识点文档 |
| **主题文档** `XX_模块/` | 建立深入理解 | 设计动机、原理架构、关键决策、与其他概念的关系、易错点 | 不写"面试怎么答"，聚焦"为什么这么设计" |
| **面试题文档** `XX_模块/QA_*.md` | 面试输出训练 | 一句话定义、追问链、常见错误答法、自己的回答模板 | 不重复贴源码和原理，只引用主题文档 |

### 2.4 关键改造动作

#### 动作1：新增 `_面试场景/` 目录

按**面试高频考察方向**组织场景文档（约15-20篇），每篇结构：

```markdown
---
title: 并发安全排查实战
tags: [面试场景, 并发, 性能调优]
difficulty: 难
frequency: 极高
---

# 并发安全排查实战

## 面试官想考察什么
> 这个问题表面是"排查"，实际在考察你能不能把 volatile、synchronized、CAS、线程池、
> JVM 这些知识点串联起来，形成完整的排查思路。

## 标准回答思路（STAR法）
1. **现象**：描述问题表现（TPS下降、响应变慢、CPU飙高……）
2. **定位**：用什么工具/命令定位（jstack、jstat、Arthas……）
3. **分析**：从哪些角度分析原因（锁竞争？线程饥饿？死锁？……）
4. **解决**：怎么解决，为什么这个方案有效

## 涉及的知识点
- [[01_synchronized原理]] ← 核心依赖
- [[02_CAS与原子类]] ← 并发安全基础
- [[03_线程池调优]] ← 常见优化手段
- [[04_JVM线程监控]] ← 排查工具

## 可能的追问
1. "你怎么判断是锁竞争而不是GC导致的？"
   → 参考 [[JVM-GC分析]]
2. "锁升级到重量级锁后，对性能影响有多大？"
   → 参考 [[synchronized锁升级]]
3. "如果用ReentrantLock替代synchronized，会有什么区别？"
   → 参考 [[Lock vs synchronized]]

## 我的回答模板
> （自己写一遍，比看十遍管用）
> 
> "在我们项目中，有一次订单服务TPS突然下降……我用jstack发现了大量BLOCKED线程……
>  定位到是XX接口的synchronized锁粒度太大……改成ReentrantLock+细粒度锁后TPS恢复了……"

## 踩坑记录
> （面试中实际遇到的问题和反思）
```

#### 动作3：专题文档和QA彻底分家

**专题文档**只讲"为什么"：
- 设计动机和背景
- 架构/原理深度解析
- 与其他概念的关系
- ASCII图解

**QA文档**只讲"怎么答"：
- 一句话核心答案
- 3-5个高频追问
- 对比表格（精炼版）
- 常见错误答法（❌ vs ✅）
- 链接到专题文档获取详细原理

#### 动作4：统一SpringBoot模块粒度

SpringBoot模块目前文件平铺（232个文件），需要和并发模块一样按主题合并：

## 四、改造前后对比

### 对比1：复习一个主题

**改造前**：打开 `04_synchronized/05_锁升级过程/` 文件夹 → 里面有一篇500行的专题文档 + 一篇500行的QA，内容重叠60% → 看完晕了，记不住重点

**改造后**：打开 `02_锁升级全过程.md`（300行，只讲原理）→ 看不懂的地方点链接跳到具体代码 → 面试前翻 `QA_synchronized.md`（150行，只讲怎么答）→ 每个回答都是精炼的要点

### 对比2：准备一场面试

**改造前**：把10_基础、11_并发、12_JVM、15_框架每个模块的知识地图过一遍 → 900多个知识点 → 不知道该重点看什么

**改造后**：打开 `_面试场景/` 目录 → 按场景复习（10-15篇）→ 每篇场景文档已经帮你串联好了跨模块的知识点 → 不会的知识点点击链接去主题文档深入 → 面试前只看场景文档的"我的回答模板"
