
# 直接内存（Direct Memory）

## 先说结论

**直接内存（Direct Memory）** 不是 JVM 运行时数据区的一部分，而是**本地内存**，通过 `ByteBuffer` 分配。用于 NIO 零拷贝，性能高但不受 GC 管理，需手动设置大小，**易忽略导致 OOM**。

## 深度解析

### 直接内存 vs 堆内存

| 特性 | 直接内存 | 堆内存 |
|------|----------|--------|
| **位置** | 本地内存（堆外） | JVM 堆 |
| **GC 管理** | 不受 GC 管理 | 受 GC 管理 |
| **分配/释放** | 手动（Unsafe/JNI） | 自动 |
| **IO 性能** | 零拷贝，高性能 | 需要拷贝 |
| **使用方式** | ByteBuffer.allocateDirect() | new byte[] |

### 工作原理

```
传统 IO（堆内存）:
用户缓冲区 ← 堆内存 ←→ 内核缓冲区 ←→ 磁盘
              (两次拷贝)

NIO（直接内存）:
用户缓冲区 = 内核缓冲区 ←→ 磁盘
(零拷贝，无需堆内存中转)
```

### NIO 零拷贝流程

```java
// 使用直接内存
ByteBuffer directBuffer = ByteBuffer.allocateDirect(1024);

// 直接内存 -> 内核缓冲区 -> 网卡
// 避免数据在堆内外复制
```

### 配置参数

```bash
# 直接内存大小（JDK8+）
-XX:MaxDirectMemorySize=256m

# 默认等于 -Xmx（最大堆）
# 实际限制受限于物理内存和 swap
```

## 易错点/踩坑

- ❌ 直接内存不受 GC 管理，需手动释放
- ❌ `-Xmx` 不包含直接内存大小
- ❌ 直接内存不足会导致 OOM（DirectMemory OOM）
- ❌ 忘记 `close()` 或 `clear()` 会导致内存泄漏
- ❌ NIO 面试必问：零拷贝原理

## 代码示例

```java
import java.nio.ByteBuffer;

public class DirectMemoryDemo {
    public static void main(String[] args) {
        // 分配直接内存
        ByteBuffer buffer = ByteBuffer.allocateDirect(1024 * 1024);
        
        // 写入数据
        buffer.put("Hello Direct Memory".getBytes());
        buffer.flip();
        
        // 读取数据
        byte[] data = new byte[buffer.remaining()];
        buffer.get(data);
        
        // 释放（依赖 GC 或手动 clean）
        // 实际需要通过反射调用 Cleaner
    }
}

// 模拟 Netty 直接内存使用
// ServerBootstrap b = new ServerBootstrap();
// b.option(ChannelOption.SO_BACKLOG, 1024);
```

```bash
# 查看直接内存使用（JMX）
jconsole
# 或
java -XX:MaxDirectMemorySize=512m YourApp
```
