# 伪共享（缓存行竞争导致性能问题）

## 先说结论

伪共享（False Sharing）是多线程下一种隐蔽的性能杀手：多个线程修改**同一缓存行**中不同变量的数据，CPU 缓存一致性协议会反复使该缓存行失效，导致本应命中缓存的操作退化为跨核内存访问，性能可能下降数倍。解决方案是使用**缓存行填充**（Padding）或 `@Contended` 注解，将竞争变量分散到不同缓存行。

## 深度解析

### 什么是缓存行

CPU 不会按字节读写内存，而是以**缓存行（Cache Line）**为单位（通常 64 字节）与主内存交换数据。当 CPU 读取一个变量时，该变量所在整个缓存行（64 字节）会被加载到 CPU 缓存中。

### 伪共享的产生

当两个独立变量恰好位于同一缓存行时：

```
缓存行（64字节）: |---- 变量 A (8B) ----|---- 变量 B (8B) ----|---- 未使用 ----|
```

- 线程 1（CPU Core 0）修改变量 A → 整个缓存行失效
- 线程 2（CPU Core 1）修改变量 B → 整个缓存行失效
- 两个线程本不共享数据，却因缓存行级失效而互相干扰

```
CPU Core 0                     CPU Core 1
  │                              │
  │ 修改变量 A                    │ 修改变量 B
  ▼                              ▼
Cache Line: [A | B | ...]    Cache Line: [A | B | ...]
  │                              │
  └─────── 竞争同一个缓存行 ────────┘
         (MESI 协议反复失效)
```

### 解决方案

#### 方案一：缓存行填充（Padding）

在变量之间填充无用字段，使不同变量落入不同缓存行：

```java
public class PaddedLong {
    public volatile long value;
    // 填充 7 个 long（56 字节），使 value 独占一个缓存行
    public long p1, p2, p3, p4, p5, p6, p7;
}
```

#### 方案二：@Contended 注解（JDK 8+）

```java
public class ContendedLong {
    @jdk.internal.vm.annotation.Contended
    public volatile long value;
}
```

使用时需添加 JVM 参数：`-XX:-RestrictContended`

> Java 8 提供了 `@Contended` 注解，默认只在 JDK 内部类中可用。用户代码需配合 `-XX:-RestrictContended` 参数使用。Java 9+ 可以用 `@jdk.internal.vm.annotation.Contended`，但注意这是内部 API。

#### 方案三：LongAdder（JDK 8 推荐）

`LongAdder` 内部使用 Cell 数组 + 分段 CAS，本质就是避免多线程竞争同一个缓存行：

```java
// Cell 类源码（简化）
@jdk.internal.vm.annotation.Contended
static final class Cell {
    volatile long value;
    Cell(long x) { value = x; }
    // CAS 操作
    final boolean cas(long cmp, long val) {
        return UNSAFE.compareAndSwapLong(this, VALUE, cmp, val);
    }
}
```

### Disruptor 框架：缓存行填充的工业级实践

Disruptor 是 LMAX 开源的高性能并发框架，通过精心设计的缓存行填充彻底解决了伪共享问题：

```java
// Disruptor 中 RingBuffer 的 Sequence 字段设计
// 每个 Sequence（槽指针）都是独立缓存行
abstract class RingBufferPad {
    // 前置填充：7 个 long（56字节），确保字段从缓存行边界开始
    protected long p1, p2, p3, p4, p5, p6, p7;
}

abstract class RingBufferFields extends RingBufferPad {
    // 真正的 Sequence 字段，每个独占缓存行
    protected long value;
}

class SingleProducerSequencerFields extends RingBufferFields {
    // 后置填充：同样 7 个 long，确保字段独占缓存行
    protected long p1, p2, p3, p4, p5, p6, p7;
}

// Disruptor 的 RingBuffer 使用了总计 14 个填充 long：
// 7 (前置) + 8 (Sequence) + 7 (后置) = 22 * 8 = 176 字节
// 远超 64 字节缓存行，保证每个 Sequence 完全独立
```

**Disruptor 的设计哲学**：
1. **前置填充 + 后置填充**：确保被保护字段独占一个缓存行，避免前后字段侵入
2. **无锁设计**：通过缓存行填充的原子性 + CAS 实现高吞吐量
3. **内存预分配**：RingBuffer 预先分配所有对象，避免 GC 开销

> **面试亮点**：Disruptor 在金融交易系统（高频交易）中广泛使用，其缓存行填充设计是"理论到工程"的经典案例。

### 缓存行大小因架构而异

文档说"通常 64 字节"，但实际不同架构的缓存行大小不同：

| 架构 | 缓存行大小 | 说明 |
|------|-----------|------|
| x86/x64 | 64 字节 | 最常见 |
| ARM Cortex-A 系列 | 64 字节 | 移动端主流 |
| Apple M1/M2 | **128 字节** | ARM64 定制，缓存行更大 |
| SPARC | 64 字节 | 服务端 |
| IBM POWER | 128 字节 | 大型机/高端服务器 |

```java
// Java 中获取缓存行大小（近似值）
int cacheLineSize = 64; // 无法直接通过 API 获取，可通过 jol (Java Object Layout) 工具查看

// jol 使用示例
// 添加依赖：org.openjdk.jmh:jol-core
// System.out.println(VM.current().details()); // 打印对象布局
```

> **实践建议**：在设计高性能数据结构时，使用 jol 工具实际测量对象布局，而非假设 64 字节。

## 易错点/踩坑

- ❌ 误以为 synchronized 或 volatile 能解决伪共享问题 → 它们只保证可见性/原子性，不解决缓存行竞争
- ❌ 忘记加 `-XX:-RestrictContended` 参数导致 `@Contended` 不生效
- ✅ 伪共享是**逻辑上不共享**但**物理上在同一缓存行**导致的，与真正的数据共享不同
- ✅ 在对象头占 12~16 字节（压缩指针时 12 字节），计算填充时需考虑对象头

## 代码示例

### 演示伪共享的性能影响

```java
public class FalseSharingDemo {
    public static long ITERATIONS = 50_000_000L;

    // 两个 volatile long 共享同一缓存行（8+8=16字节，小于64字节缓存行）
    public static class Shared {
        public volatile long x;
        public volatile long y;
    }

    // 填充后各自独占缓存行
    public static class Padded {
        public volatile long x;
        public long p1, p2, p3, p4, p5, p6; // 填充

        public volatile long y;
        public long p7, p8, p9, p10, p11, p12; // 填充
    }

    public static void main(String[] args) throws Exception {
        // 测试伪共享
        Shared shared = new Shared();
        long start = System.nanoTime();
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < ITERATIONS; i++) shared.x++;
        });
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < ITERATIONS; i++) shared.y++;
        });
        t1.start(); t2.start();
        t1.join(); t2.join();
        System.out.println("伪共享耗时: " + (System.nanoTime() - start) / 1_000_000 + "ms");

        // 测试填充后
        Padded padded = new Padded();
        start = System.nanoTime();
        Thread t3 = new Thread(() -> {
            for (int i = 0; i < ITERATIONS; i++) padded.x++;
        });
        Thread t4 = new Thread(() -> {
            for (int i = 0; i < ITERATIONS; i++) padded.y++;
        });
        t3.start(); t4.start();
        t3.join(); t4.join();
        System.out.println("填充后耗时: " + (System.nanoTime() - start) / 1_000_000 + "ms");
    }
}
```

## 图解/流程

```
┌──────────────────────────────────────────────────────────┐
│                   伪共享发生过程                            │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  L1 Cache (Core 0)      L1 Cache (Core 1)                │
│  ┌─────────────────┐    ┌─────────────────┐              │
│  │ Cache Line      │    │ Cache Line      │              │
│  │ [x=1][y=0][pad] │    │ [x=1][y=0][pad] │              │
│  └────────┬────────┘    └────────┬────────┘              │
│           │                      │                       │
│  Core 0: x++ (写)               │                       │
│  → 发送 Invalidate 信号 ────────→│ (y 无关但整行失效!)    │
│           │                      │                       │
│  Core 1: y++ (写)               │                       │
│  ←──────── 发送 Invalidate 信号─┤ (x 无关但整行失效!)    │
│           │                      │                       │
│  ┌───────▼────────┐    ┌────────▼─────────┐              │
│  │ 重新从内存加载  │    │ 重新从内存加载   │              │
│  │ [x=2][y=0]     │    │ [x=2][y=1]      │              │
│  └─────────────────┘    └─────────────────┘              │
│                                                          │
│  结果: 大量 Cache Miss → 访问主存 → 性能下降数倍          │
└──────────────────────────────────────────────────────────┘

解决后（缓存行填充）：

  L1 Cache (Core 0)      L1 Cache (Core 1)
  ┌─────────────────┐    ┌─────────────────┐
  │ Line 1          │    │ Line 2          │
  │ [x=2][pad×7]    │    │ [y=1][pad×7]    │  ← 不同缓存行!
  └─────────────────┘    └─────────────────┘
  各自修改，互不影响 ✅
```
