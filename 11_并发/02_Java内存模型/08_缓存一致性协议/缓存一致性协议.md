# 缓存一致性协议（MESI/MESI-Ownership/MOESI）

## 先说结论

缓存一致性协议是 CPU 多核架构中保证各核心缓存数据一致的底层协议。最主流的是 **MESI 协议**，定义了四种缓存行状态（Modified/Exclusive/Shared/Invalid），通过总线消息在核心间协调读写。变体包括 **MESI-Ownership**（引入 Owner 状态减少写回）和 **MOESI**（增加 Owned 状态允许脏数据共享）。理解这些协议是掌握 volatile 内存语义、伪共享等高层概念的基础。

## 深度解析

### 为什么需要缓存一致性协议

现代多核 CPU 中每个核心都有独立的 L1/L2 缓存，而 L3 缓存（或主存）是共享的：

```
Core 0                Core 1                Core N
  │                     │                     │
 L1 Cache             L1 Cache             L1 Cache
 L2 Cache             L2 Cache             L2 Cache
  │                     │                     │
  └─────────────────────┼─────────────────────┘
                        │
                    L3 Cache (共享)
                        │
                     Main Memory
```

当多个核心缓存同一内存地址时，必须保证某个核心修改数据后其他核心能看到最新值。这就是缓存一致性协议解决的问题。

### MESI 协议（最基础、最经典）

#### 四种状态

| 状态 | 含义 | 该缓存行是否有效 | 是否与主存一致 |
|------|------|:---:|:---:|
| **M** (Modified) | 已修改，仅本缓存有 | ✅ | ❌ |
| **E** (Exclusive) | 独占，仅本缓存有，未修改 | ✅ | ✅ |
| **S** (Shared) | 共享，多缓存都有，只读 | ✅ | ✅ |
| **I** (Invalid) | 无效，不能读取 | ❌ | — |

#### 状态转换

```
                    ┌─────────────────────────────────────┐
                    │         CPU 0 读内存地址 X           │
                    │  (其他 Core 无 X 的缓存副本)         │
                    └──────────────┬──────────────────────┘
                                   ▼
                         ┌──────────────────┐
           ┌────────────│     E (Exclusive) │◀────┐
           │  写本地     └────────┬─────────┘     │  其他 Core
           │             Cache   │  本地写        │  请求读 X
           │             Hit     ▼               │
           │        ┌──────────────────┐         │
           │        │  M (Modified)    │─────────┤───▶ Flush to Memory
           │        └────────┬─────────┘         │     + 通知其他 Core
           │                 │ 其他 Core 写 X    │     Invalidate
           │                 ▼                   │
           │         ┌──────────────────┐         │
           │  其他Core│  I (Invalid)     │         │
           │  读 X ──▶└────────┬─────────┘         │
           │           从内存加载│其他 Core         │
           │              最新值  │ 也有副本         │
           │                 ▼                   │
           │         ┌──────────────────┐         │
           └─────────│  S (Shared)      │─────────┘
                     └──────────────────┘
```

#### 关键总线消息

| 消息 | 含义 |
|------|------|
| **Read** | 通知其他 Core 和内存：我要读这个地址 |
| **Read Response** | 返回缓存行数据（优先从其他 Core 的 M/E 状态获取） |
| **Invalidate** | 通知其他 Core：将指定地址标记为 Invalid |
| **Invalidate Ack** | 其他 Core 确认已失效 |
| **Read Invalidate** | = Read + Invalidate，获取数据的同时使其他副本失效 |
| **Writeback** | 将 M 状态缓存行写回内存 |

#### 典型操作流程

**写操作（Store）**：

```
1. CPU 检查本地缓存：Cache Hit → 检查状态
   - M 状态：直接写 ✅（最快路径）
   - E 状态：升级为 M，直接写 ✅
   - S 状态：发送 Invalidate，等待所有 Ack → 升级为 M → 写
   - I 状态：Cache Miss → 发送 Read Invalidate → 获取数据 → 升级为 M → 写
```

### MESI-Ownership 协议

MESI 的扩展，引入 **Owner** 状态替代 Modified 状态：

- Owner：本核心拥有最新数据，且**允许其他核心以 Shared 状态持有旧副本**
- 当其他 Core 请求读取时，Owner 直接提供数据（**转发**），无需先写回内存
- 减少了一次写回内存的总线事务，提升了共享读场景的性能

```
传统 MESI：Core A (M) → Core B 读 → A 写回内存 → B 从内存读（两次总线操作）
MESI-Owner：Core A (O) → Core B 读 → A 直接转发给 B（一次总线操作）
```

### MOESI 协议（AMD 常用）

在 MESI 基础上增加 **Owned** 状态：

| 状态 | 含义 |
|------|------|
| **O** (Owned) | 本缓存有最新数据（脏数据），允许其他核心有 Shared 副本 |

与 MESI-Ownership 类似，但 O 状态可以与 S 状态共存。典型流程：

```
1. Core A 读 X → S 状态
2. Core B 读 X → A 和 B 都是 S 状态
3. Core A 写 X → A 变 O，B 变 I（不需要立即写回内存）
4. Core B 读 X → A(O) 直接转发给 B，B 变 S，A 仍为 O
```

### 与 Java 内存模型的关系

```
Java volatile 写              MESI 协议
──────────────                ─────────
1. 修改工作内存          →    写本地缓存
2. 刷新到主内存          →    发送 Invalidate + 等待 Ack
3. 其他线程可见          →    其他 Core Cache Miss → 从总线获取最新值

Java volatile 读
──────────────
1. 从主内存重新加载      →    本地 Cache Miss → 总线 Read → 获取最新缓存行
```

volatile 的 happens-before 保证底层就依赖 MESI 协议的失效/刷新机制。

### 总线风暴：volatile 滥用导致的性能陷阱

当大量线程频繁读写同一个 `volatile` 变量时，会引发**总线风暴（Bus Storm）**问题：

```java
// ❌ 问题代码：volatile 滥用
public class VolatileStorm {
    private volatile long counter = 0;  // 多个线程频繁写同一地址

    public void increment() {
        for (int i = 0; i < 1_000_000; i++) {
            counter++;  // 每次 increment 都触发 MESI Invalidate + 总线广播
        }
    }
}

// 问题分析：
// 线程1 写 counter → 广播 Invalidate → 其他所有核心的缓存行失效
// 线程2 写 counter → 再次广播 Invalidate
// → 每个 volatile 写都产生一次全局总线事务
// → 大量线程同时写 → 总线带宽被占满 → 性能急剧下降
```

**总线风暴的形成机制**：

```
正常情况：      核心A ──写──▶ 缓存行失效广播 ──▶ 总线 ──▶ 核心B(C)(D) 收到失效
                         ↑
                    每个 volatile 写都触发此广播

风暴时：
线程1 ──▶ 核心A 写 ──▶ 广播 ──┐
线程2 ──▶ 核心B 写 ──▶ 广播 ──┼──▶ 总线带宽占满 ──▶ 性能骤降
线程3 ──▶ 核心C 写 ──▶ 广播 ──┤
线程4 ──▶ 核心D 写 ──▶ 广播 ──┘
```

**解决方案**：

| 方案 | 做法 | 适用场景 |
|------|------|----------|
| `LongAdder` 替代 `AtomicLong` | 写操作分散到不同缓存行 | 高并发计数器 |
| 减少 volatile 写入频率 | 批量更新 + 定期刷新 volatile | 非实时性场景 |
| 使用局部变量缓冲 | 本地计算 → 批量写入 volatile | 计算密集型 + 同步需求 |
| 锁分段 | 多个锁保护不同分区 | 复杂数据结构 |

```java
// ✅ 推荐：用 LongAdder 替代
LongAdder counter = new LongAdder();  // 内部 Cell 数组，每个 Cell 独立缓存行
counter.increment();  // 写分散到不同缓存行，不触发全局失效
```

> **面试警示**：面试中提到 volatile 时，务必补充"滥用会导致总线风暴"这一性能陷阱，体现对 volatile 底层机制的深层理解。

## 易错点/踩坑

- ❌ 认为 volatile 直接操作主存所以慢 → 实际上 volatile 写只是触发总线消息，数据仍在缓存中
- ❌ 混淆缓存行失效（Invalidate）和缓存刷新（Flush） → Invalidate 只是标记无效，不涉及数据传输
- ✅ MESI 的性能瓶颈主要在 **Store Forwarding 等待**（写时等待所有 Invalidate Ack），这也是 Store Buffer 存在的原因
- ✅ Store Buffer（写缓冲）虽然解决了等待问题，但引入了**内存重排序**，这就是为什么需要 Memory Barrier

## 代码示例

### 通过 JMH 观察 volatile 的开销（体现 MESI 协议开销）

```java
import org.openjdk.jmh.annotations.*;
import java.util.concurrent.TimeUnit;

@State(Scope.Group)
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.MICROSECONDS)
public class MESIOverheadBench {

    // 普通变量 — 无缓存一致性开销
    private long plainValue;

    // volatile 变量 — 每次写触发 MESI Invalidate
    private volatile long volatileValue;

    @Benchmark
    @Group("plain")
    public long readPlain() {
        return plainValue;
    }

    @Benchmark
    @Group("plain")
    public void writePlain() {
        plainValue++;
    }

    @Benchmark
    @Group("volatile")
    public long readVolatile() {
        return volatileValue;
    }

    @Benchmark
    @Group("volatile")
    public void writeVolatile() {
        volatileValue++;
    }
    // volatile 组的吞吐量显著低于 plain 组
    // 差距在多线程环境下更为明显
}
```
