# JMM 抽象模型（主内存与工作内存）

## Q1：什么是Java内存模型（JMM）？为什么要定义JMM？

**A**：Java内存模型（Java Memory Model，JMM）是Java语言规范中定义的一套规则，用于规范多线程环境下共享变量的访问方式。
定义JMM的原因：
1. **屏蔽硬件差异**：不同CPU架构（x86、ARM等）的内存模型不同，JMM提供统一抽象
2. **保证可移植性**：Java程序"一次编写，到处运行"，需要统一的内存语义
3. **明确并发语义**：定义线程何时能看到其他线程的写入，以及何时允许重排序
4. **平衡性能与安全**：在保证正确性的前提下，允许编译器和处理器进行合理优化

## Q2：JMM中主内存和工作内存分别是什么？它们之间的关系是什么？

**A**：
**主内存（Main Memory）**：
- 存储所有共享变量（实例字段、静态字段、数组元素）
- 所有线程共享，是线程间通信的媒介
- 对应物理内存
**工作内存（Working Memory）**：
- 每个线程私有的内存区域
- 存储该线程使用的共享变量的副本
- 对应CPU寄存器、高速缓存等硬件
**关系**：
- 线程不能直接操作主内存变量，必须通过工作内存
- 线程间通信必须经过主内存：线程A把工作内存变量刷新到主内存，线程B从主内存读取到自己的工作内存
- 工作内存的变量副本可能与主内存不一致，导致可见性问题

## Q3：JMM中的工作内存和JVM内存结构中的栈、堆有什么关系？

**A**：两者是完全不同的概念：

| 维度 | JMM工作内存 | JVM内存结构 |
|------|------------|-------------|
| 层级 | 抽象概念 | 具体实现 |
| 作用 | 定义并发访问规则 | 管理运行时数据 |
| 内容 | 共享变量的副本 | 对象、方法区等 |
| 对应硬件 | CPU寄存器、缓存 | 物理内存区域 |
**关键点**：
- JMM工作内存不是JVM堆内存的一部分
- 即使是堆中的对象，线程访问时也是先加载到工作内存（缓存/寄存器）
- JVM栈中存储的是基本类型和引用，但多线程共享变量的访问仍遵循JMM规则

## Q4：为什么线程修改了共享变量，其他线程却不一定能立即看到？

**A**：这是因为JMM的分层结构导致的可见性问题：
1. **修改发生在工作内存**：线程修改的是工作内存中的变量副本
2. **刷新到主内存有延迟**：修改不会立即同步到主内存（除非使用`volatile`或同步机制）
3. **其他线程读取的是旧值**：其他线程可能仍使用自己工作内存中的旧副本
**示例**：
```java
// 线程A执行
sharedVar = 1;  // 只修改了工作内存中的副本

// 线程B执行
System.out.println(sharedVar);  // 可能读到0（旧值）
```
**解决方案**：
- 使用`volatile`关键字：保证修改立即刷新到主内存
- 使用`synchronized`：解锁时刷新工作内存到主内存
- 使用`Lock`：释放锁时刷新

## Q5：JMM与硬件内存模型（如TSO、PSO）有什么关系？

**A**：
**硬件内存模型**：
- **TSO（Total Store Order）**：x86架构，较强一致性，Store-Load可重排序
- **PSO（Partial Store Order）**：SPARC架构，Store-Store可重排序
- **WO（Weak Ordering）**：ARM、PowerPC，更多重排序可能
**JMM与硬件的关系**：
1. JMM是**软件层面**的抽象，定义Java程序应有的内存语义
2. JVM实现需要将JMM映射到底层硬件内存模型
3. 在弱一致性硬件（如ARM）上，JVM需要插入更多内存屏障来保证JMM语义
4. 在强一致性硬件（如x86）上，某些内存屏障可以省略

**意义**：JMM让Java程序员无需关心底层硬件差异，编写一次代码即可在所有平台正确运行。
