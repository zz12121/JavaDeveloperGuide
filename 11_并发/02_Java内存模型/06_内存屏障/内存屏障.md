# 内存屏障（LoadLoad/StoreStore/LoadStore/StoreLoad，JMM通过内存屏障实现可见性和有序性）

## 先说结论
内存屏障（Memory Barrier）是 CPU 和编译器用于控制指令重排序和内存可见性的底层机制。JMM 通过在特定操作前后插入四种类型的内存屏障（LoadLoad、StoreStore、LoadStore、StoreLoad），实现 volatile 的可见性和有序性保证，以及 synchronized 的同步语义。

## 深度解析

### 为什么需要内存屏障？

现代 CPU 的性能优化导致两个问题：

**1. 写缓冲（Store Buffer）延迟**
```
CPU1: 写 x=1  → 写入 Store Buffer  → 延迟写回主内存
CPU2: 读 x    → 可能读到旧值 0
```

**2. 失效队列（Invalidate Queue）延迟**
```
收到 Invalidate 消息 → 进入失效队列 → 延迟处理
→ CPU 继续使用缓存中的旧数据
```

**内存屏障的作用**：强制 CPU 按程序顺序访问内存，解决上述问题。

### 四种内存屏障类型

#### 1. LoadLoad 屏障
**作用**：禁止 **Load1** 和 **Load2** 之间重排序
**场景**：volatile 读、锁的获取

```
指令序列：
Load1 (普通读)
LoadLoad (屏障)
Load2 (volatile 读 / 锁获取)

效果：Load1 必须完成后，才能执行 Load2
```

#### 2. StoreStore 屏障
**作用**：禁止 **Store1** 和 **Store2** 之间重排序
**场景**：volatile 写、锁的释放

```
指令序列：
Store1 (普通写)
StoreStore (屏障)
Store2 (volatile 写 / 锁释放)

效果：Store1 必须完成后，才能执行 Store2
```

#### 3. LoadStore 屏障
**作用**：禁止 **Load** 和 **Store** 之间重排序
**场景**：volatile 读写之间

```
指令序列：
Load (普通读)
LoadStore (屏障)
Store (普通写)

效果：Load 必须完成后，才能执行 Store
```

#### 4. StoreLoad 屏障
**作用**：禁止 **Store** 和 **Load** 之间重排序
**场景**：volatile 写后读

```
指令序列：
Store (volatile 写)
StoreLoad (屏障)
Load (volatile 读)

效果：Store 写入主内存后，才能执行 Load
```

### 屏障在 x86 架构上的实现

| 屏障类型 | x86 实现 | 说明 |
|---------|---------|------|
| LoadLoad | `lfence` | 读屏障 |
| StoreStore | `sfence` | 写屏障 |
| StoreLoad | `mfence` 或 `lock` 前缀 | 全屏障（最重） |
| LoadStore | `lfence` | 读屏障 |

**注意**：x86 是**强内存模型**，Store-Load 重排序已被硬件保证，所以 StoreLoad 屏障在 x86 上主要是为了刷新 Store Buffer。

### JMM 对 volatile 的内存屏障实现

#### volatile 写的屏障插入
```java
public void writer() {
    a = 1;              // Store1: 普通写
    StoreStore 屏障;    // 禁止 Store1 重排序到 volatile 写之后
    flag = true;        // Store2: volatile 写
    StoreLoad 屏障;     // 禁止 volatile 写之后的 Load 重排序
}
```

#### volatile 读的屏障插入
```java
public void reader() {
    LoadLoad 屏障;      // 禁止上面的 Load 重排序到 volatile 读之后
    LoadStore 屏障;     // 禁止上面的 Load 重排序到 volatile 写之前
    if (flag) {         // Load: volatile 读
        int i = a;      // Load: 普通读
    }
}
```

### synchronized 的内存屏障实现

**monitorenter（获取锁）：**
```
Load (读取对象头)
LoadLoad 屏障
LoadStore 屏障
Store (标记锁为持有)
```

**monitorexit（释放锁）：**
```
Store (标记锁为空闲)
StoreStore 屏障
StoreLoad 屏障
Load (读取对象头)
```

### 内存屏障与 Happens-Before 的关系

**Happens-Before 是规则，内存屏障是实现：**

```
┌────────────────────────────────────────────┐
│            Happens-Before 规则              │
│                                            │
│  程序次序规则  →  编译器优化边界             │
│  volatile 规则  →  内存屏障禁止重排序        │
│  管程锁规则    →  锁获取/释放的内存屏障      │
│  线程启动/终止 →  特殊的同步点               │
└────────────────────────────────────────────┘
                    ↓ 实现
┌────────────────────────────────────────────┐
│              内存屏障                       │
│                                            │
│  LoadLoad / StoreStore / LoadStore / StoreLoad│
│  在正确的位置插入正确的屏障                   │
│  实现可见性和有序性                         │
└────────────────────────────────────────────┘
```

### Java 层面的显式内存屏障

除了 JVM 自动插入屏障外，Java 9+ 提供了 `VarHandle` 和 `Unsafe` 允许在代码中显式插入内存屏障，这是实现高性能并发库的基础：

```java
import java.lang.invoke.MethodHandles;
import java.lang.invoke.VarHandle;

public class ExplicitBarrierDemo {
    private int value = 0;
    
    // VarHandle：Java 9+ 提供的反射式原子访问API
    private static final VarHandle VH;
    
    static {
        try {
            VH = MethodHandles.privateLookupIn(
                ExplicitBarrierDemo.class,
                MethodHandles.lookup()
            ).findVarHandle(ExplicitBarrierDemo.class, "value", int.class);
        } catch (Exception e) {
            throw new AssertionError(e);
        }
    }
    
    public void writeWithFence() {
        VH.set(this, "value", 42);  // 普通写入
        // 插入 StoreStore 屏障：确保此写入不会重排序到后续操作
        VH.storeStoreFence();
        System.out.println("写入完成");
    }
    
    public int readWithFence() {
        // 插入 LoadLoad 屏障：确保后续读取不会重排序到之前
        VH.loadLoadFence();
        return (int) VH.get(this, "value");  // 读取
    }
    
    //Unsafe 层面的显式屏障（Java 8+）
    public void unsafeFences() {
        // 读屏障
        sun.misc.Unsafe.getUnsafe().loadFence();
        // 写屏障
        sun.misc.Unsafe.getUnsafe().storeFence();
        // 全屏障（读写都阻塞）
        sun.misc.Unsafe.getUnsafe().fullFence();
    }
}
```

**四种显式屏障（VarHandle 提供）**：

| 方法 | 对应屏障 | 作用 |
|------|---------|------|
| `loadLoadFence()` | LoadLoad | 禁止前面的 Load 与后面的 Load 重排序 |
| `storeStoreFence()` | StoreStore | 禁止前面的 Store 与后面的 Store 重排序 |
| `loadStoreFence()` | LoadStore | 禁止前面的 Load 与后面的 Store 重排序 |
| `storeLoadFence()` | StoreLoad | 禁止前面的 Store 与后面的 Load 重排序 |
| `fullFence()` | 全屏障 | 综合上述四种屏障效果 |

> **应用场景**：Disruptor、High-scale 高性能库、ConcurrentLinkedQueue 等底层实现中大量使用显式屏障，在不需要 synchronized 的情况下精细控制内存顺序。

## 易错点/踩坑

- ❌ **混淆**：认为内存屏障只用于禁止重排序
  - 内存屏障还有**刷新缓冲区**、**刷新 CPU 缓存**的作用

- ❌ **误解**：所有屏障都是全屏障
  - LoadLoad、StoreStore 等是**部分屏障**
  - StoreLoad 是全屏障（最昂贵）

- ❌ **忽略**：屏障有性能开销
  - StoreLoad 是最昂贵的屏障
  - x86 上 volatile 的 StoreLoad 成本较高

- ✅ **正确理解**：内存屏障是 CPU 和编译器的协作
  - 编译器在字节码中插入屏障指令
  - CPU 执行屏障指令，控制内存访问顺序

## 代码示例

### volatile 内存屏障的完整示例
```java
public class MemoryBarrierDemo {
    private int a = 0;
    private volatile boolean flag = false;

    // 线程 A 执行
    public void writer() {
        // 屏障1：StoreStore
        a = 1;           // Store1: a=1 必须在 flag=true 之前完成
        // StoreStore Barrier
        flag = true;    // Store2: volatile 写
        // StoreLoad Barrier: 刷新 Store Buffer
    }

    // 线程 B 执行
    public void reader() {
        // LoadLoad Barrier
        // LoadStore Barrier
        if (flag) {      // Load1: volatile 读
            // LoadLoad Barrier: flag 读取完成后才能读 a
            int i = a;   // Load2: 普通读 - 保证读到 a=1
        }
    }
}
```

### happens-before 保证的图示
```
线程 A (writer)                    线程 B (reader)

a = 1;                           
StoreStore屏障                    
flag = true;                     
StoreLoad屏障                     
              ───────── happens-before ─────────>
              if (flag) {
              LoadLoad屏障
              LoadStore屏障
                  int i = a;    ← 一定能读到 a=1
              }
```

## 图解/流程

### 四种内存屏障示意

```
LoadLoad 屏障：
┌────────┬─────────────┬────────┐
│ Load1  │ LoadLoad    │ Load2  │
└────────┴─────────────┴────────┘
  必须完成      屏障        才能开始

StoreStore 屏障：
┌────────┬─────────────┬────────┐
│ Store1 │ StoreStore  │ Store2 │
└────────┴─────────────┴────────┘
  必须完成      屏障        才能开始

LoadStore 屏障：
┌────────┬─────────────┬────────┐
│ Load   │ LoadStore   │ Store  │
└────────┴─────────────┴────────┘
  必须完成      屏障        才能开始

StoreLoad 屏障：
┌────────┬─────────────┬────────┐
│ Store  │ StoreLoad   │ Load   │
└────────┴─────────────┴────────┘
  必须完成      屏障        才能开始
  (最重要、最昂贵的屏障)
```

### Store Buffer 刷新过程

```
┌─────────────────────────────────────────────────────────────┐
│                    Store Buffer 问题                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   CPU1                    Store Buffer           Memory    │
│  ┌──────┐                ┌───────────┐          ┌──────┐  │
│  │Write │───────────────→│ x=1 (pending)│───────→│      │  │
│  │ x=1  │                │            │          │      │  │
│  └──────┘                └───────────┘          └──────┘  │
│                                                             │
│   CPU2                                                       │
│  ┌──────┐                                                   │
│  │Read  │───────────────→ 可能读到 x=0（旧值）               │
│  │ x    │                                                   │
│  └──────┘                                                   │
│                                                             │
│   问题：CPU1 的写入对 CPU2 不可见                              │
│   解决：StoreLoad 屏障刷新 Store Buffer                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘

刷新后：
   CPU1                    Store Buffer           Memory
  ┌──────┐                ┌───────────┐          ┌──────┐
  │Write │───────────────→│           │──────────→│ x=1  │
  │ x=1  │      StoreLoad  │ (空)      │           │      │
  └──────┘          ↓      └───────────┘          └──────┘
                    │
                    ▼ CPU1 等待刷新完成
   CPU2 读 x ──────────────────────────────→ 读到 x=1
```
