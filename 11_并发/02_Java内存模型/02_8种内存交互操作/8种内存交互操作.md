# 8种内存交互操作（lock/unlock/read/load/use/assign/store/write）

## 先说结论

JMM定义了8种原子操作来完成主内存与工作内存之间的数据交互：**lock/unlock**（锁定/解锁）、**read/load**（读取/载入）、**use/assign**（使用/赋值）、**store/write**（存储/写入）。其中read/load、use/assign、store/write必须成对出现，但不保证连续执行。这8种操作是JVM实现内存可见性和原子性的基础机制。

## 深度解析

### 1. 8种操作详解

| 操作 | 作用域 | 功能描述 |
|------|--------|----------|
| **lock** | 主内存 | 将变量标记为线程独占状态 |
| **unlock** | 主内存 | 释放变量的锁定状态 |
| **read** | 主内存 → 工作内存 | 从主内存读取变量值，为load做准备 |
| **load** | 工作内存 | 将read读取的值放入工作内存变量副本 |
| **use** | 工作内存 | 将变量值传递给执行引擎 |
| **assign** | 工作内存 | 将执行引擎的值赋给工作内存变量 |
| **store** | 工作内存 → 主内存 | 将工作内存变量值传出，为write做准备 |
| **write** | 主内存 | 将store传来的值写入主内存变量 |

### 2. 操作配对规则

JMM规定以下操作必须**成对出现**：

```
read ──→ load      (读取后必须载入)
store ──→ write    (存储后必须写入)
```

但**不要求连续执行**，中间可以插入其他操作。

### 3. 操作执行规则

**不允许单独出现**：
- 不允许read后没有load
- 不允许store后没有write
- 不允许load前没有read
- 不允许write前没有store

**顺序约束**：
- 同一个变量，assign后可以紧跟store（但不强制）
- 同一个变量，没有assign则不能store
- 同一个变量，unlock前必须有store+write（变量必须先写回主内存）
- 同一个变量，lock后会清空工作内存中该变量的值，需要重新read+load

### 4. 操作的原子性保证

JMM保证以下操作是**原子性**的：
- read、load、use、assign、store、write：对基本数据类型（除long/double外）的操作
- lock、unlock：监视器锁的获取和释放

**注意**：对于64位的long和double，JVM允许将read/load、store/write拆分为两个32位操作（非原子），但商用JVM基本都实现了原子性。

### JSR-133 之后：8 种操作模型的演进

**重要背景**：JMM 的 8 种内存交互操作模型是 JDK 5（JSR-133）之前的设计。JSR-133 重新定义了 JMM 之后，现代 JMM 的核心已经从"8 种操作"转向了 **Happens-Before 规则**体系。

**为什么 8 种操作不再是面试重点**：

| 时代 | JMM 描述方式 | 核心概念 |
|------|-------------|----------|
| JDK 4（JSR-133 之前） | 8 种内存交互操作 | lock/unlock/read/load/use/assign/store/write |
| JDK 5+（JSR-133 之后） | Happens-Before 规则 + 内存屏障 | 8 种规则 + 4 种屏障 |

**JSR-133 的核心改进**：
1. 重新定义了 `volatile` 的语义（之前的 volatile 不能保证 happens-before）
2. 引入了 **final 字段的安全发布规则**
3. 用 Happens-Before 规则取代了 8 种操作作为 JMM 的主要描述语言

> **面试提示**：如果面试官问到"8 种内存交互操作"，除了描述它们，还需要补充说明"JSR-133 之后，JMM 更侧重于 Happens-Before 规则体系"，这能体现出你对 JMM 历史演进的了解。

## 易错点/踩坑

- ❌ **错误理解**：read和load是同一个操作
  - ✅ **正确**：read是主内存操作，load是工作内存操作，两者可以分离

- ❌ **错误理解**：assign后变量立即对其他线程可见
  - ✅ **正确**：assign只修改工作内存，必须经过store+write到主内存才可见

- ❌ **错误理解**：8种操作都可以由程序员直接调用
  - ✅ **正确**：8种操作是JVM内部实现，程序员通过`volatile`、`synchronized`等关键字间接使用

- ❌ **错误理解**：lock和unlock对应`synchronized`的进入和退出
  - ✅ **正确**：基本对应，但`synchronized`还包含其他语义（如内存屏障）

- ❌ **错误理解**：8种操作都是立即执行的
  - ✅ **正确**：操作之间可以插入其他指令，JMM只保证相对顺序

## 代码示例

```java
public class MemoryOperationDemo {
    private static volatile int counter = 0;
    
    public static void main(String[] args) {
        // 模拟8种内存操作的执行流程
        
        // 线程A：counter++ 的内存操作分解
        new Thread(() -> {
            // counter++ 实际执行：
            // 1. read: 从主内存读取counter值
            // 2. load: 载入到工作内存
            // 3. use: 传递给执行引擎
            // 4. 执行引擎计算 counter + 1
            // 5. assign: 将结果赋给工作内存变量
            // 6. store: 准备写回主内存
            // 7. write: 写入主内存（volatile保证立即执行）
            
            counter++;  // 非原子操作，多线程下需要同步
            
        }, "Thread-A").start();
        
        // synchronized块的内存操作
        synchronized (MemoryOperationDemo.class) {
            // 1. lock: 锁定主内存中的对象监视器
            // 2. read+load: 读取共享变量到工作内存
            // ... 执行业务逻辑 ...
            counter = 10;
            // 3. assign+store+write: 修改并写回
            // 4. unlock: 解锁，刷新内存
        }
    }
}
```

### 操作执行顺序示例

```
线程A执行 x = 1 的内存操作序列：

时间线 ───────────────────────────────────────────────►

主内存:  [read(x)]        [write(x,1)]
           ↓                    ↑
工作内存:  [load(x)] [use(x)] [assign(x,1)] [store(x,1)]
                      ↓           ↑
                   执行引擎: 计算 x=1

说明：
- read/load 成对出现，但中间可能有间隔
- use/assign 在工作内存和执行引擎之间
- store/write 成对出现，assign后才能store
```
