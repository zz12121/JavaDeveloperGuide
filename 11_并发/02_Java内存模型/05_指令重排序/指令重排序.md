# 指令重排序（编译器/CPU/HBA重排序三层次，volatile禁止重排序）

## 先说结论
指令重排序是编译器和处理器为了优化程序性能而对指令执行顺序进行调整的机制。它分为**编译器重排序、CPU 指令重排序和内存系统重排序（HBA）**三层。volatile 通过插入内存屏障，禁止特定类型的重排序，从而保证可见性和有序性。

## 深度解析

### 为什么要重排序？
程序执行效率受两个因素制约：
1. **CPU 计算速度** >> **内存访问速度**（差距可达数百倍）
2. **指令之间存在依赖关系**（数据依赖、控制依赖）

重排序的目标：在**不改变单线程执行结果**的前提下，**尽可能提高并行度**。

### 重排序的三个层次

#### 1. 编译器重排序（Compiler Reordering）
- **执行主体**：JIT 编译器
- **优化方式**：调整无依赖关系的指令顺序
- **示例**：
```java
// 源代码
int a = 1;    // 指令1
int b = 2;    // 指令2
int c = a + b; // 指令3

// 可能重排序为（因为1和2可以交换）
int b = 2;    // 指令2（先执行）
int a = 1;    // 指令1
int c = a + b; // 指令3
```

#### 2. CPU 指令重排序（Instruction Reordering）
- **执行主体**：CPU 的执行单元
- **优化方式**：流水线（Pipeline）技术，乱序执行
- **示例**：Load 和 Store 操作可以重新排序

#### 3. 内存系统重排序 / 内存屏障（HBA - Hardware Buffer Aliasing）
- **执行主体**：Store Buffer、Invalidate Queue 等硬件缓存
- **优化方式**：延迟写回、提前读入
- **问题**：导致 CPU 看到的内存顺序与程序顺序不一致

### 数据依赖性

**编译器不会重排序存在数据依赖的操作：**

| 类型 | 示例 | 说明 |
|------|------|------|
| 写后读 | `a=1; b=a;` | 读依赖于写 |
| 读后写 | `a=b; a=1;` | 写依赖于读 |
| 写后写 | `a=1; a=2;` | 写依赖于写 |

```java
// 下面的代码不会重排序（写后读依赖）
int a = 1;
int b = a + 1;  // 必须先读取 a
```

### 控制依赖性

**编译器可能重排序带有控制依赖的操作：**
```java
if (flag) {    // 控制依赖
    int a = 1;  // 可能被重排序到 if 之外
}
```
- 编译器可以通过**分支预测**、**猜测执行**等技术优化
- 但会保证最终结果与原程序一致

### as-if-serial 语义

**核心原则**：在不改变**单线程程序执行结果**的前提下，可以进行任意重排序。

```java
// 单线程执行结果必须始终为 3
double a = 1.0 / 0.0;  // Infinity
double b = a - a;      // 0.0
// 即使重排序，也不能改变 b=0.0 的结果
```

### 指令重排序的后果

**多线程环境下的可见性问题：**
```java
public class ReorderDemo {
    private int a = 0;
    private int b = 0;

    public void writer() {
        a = 1;     // 操作1
        b = 2;     // 操作2
    }

    public int reader() {
        int x = b; // 操作3
        int y = a; // 操作4
        return x == 2 && y == 1;
    }
}
```
- 在单线程中：始终返回 true
- 多线程中：由于重排序，可能返回 false
  - 如果 1 和 2 重排序：b=2 但 a=0
  - 如果 3 和 4 重排序：先读 a=0，后读 b=2

### `synchronized` 如何禁止重排序

`synchronized` 同样通过内存屏障禁止重排序，但机制与 `volatile` 有所不同：

```java
synchronized (obj) {
    a = 1;      // 操作1
    b = 2;      // 操作2
}               // 释放锁：StoreStore + StoreLoad 屏障

// 以上代码保证了：
// ✅ 锁获取前的操作不会重排序到临界区内
// ✅ 临界区内的操作不会重排序到锁释放后
// ✅ volatile 只禁止与 volatile 变量相关的重排序
// ✅ synchronized 禁止整个同步块内外的重排序（范围更广）
```

**`synchronized` 的屏障序列**：
- **获取锁（monitorenter）**：LoadLoad + LoadStore 屏障 → 禁止临界区前的操作重排序到临界区内
- **释放锁（monitorexit）**：StoreStore + StoreLoad 屏障 → 禁止临界区内的操作重排序到临界区外

| 对比项 | `volatile` | `synchronized` |
|--------|-----------|----------------|
| 屏障范围 | 只针对 volatile 变量的读写 | 整个 synchronized 块 |
| 原子性 | 不保证复合操作原子性 | 保证（互斥锁） |
| 可见性 | 立即刷新到主存 | 释放锁时刷新所有共享变量 |
| 有序性 | 禁止特定重排序 | 禁止临界区内外重排序 |

### 对象安全发布（Safe Publication）

对象发布是指将对象引用暴露给其他线程使用（如 return this、放入集合等）。安全发布确保其他线程看到对象的正确状态：

```java
// ❌ 不安全发布（可能导致对象被部分构造）
class UnsafePublish {
    final int x;
    UnsafePublish() {
        x = 1;
        published = this;  // 构造函数中 this 逃逸
    }
    static UnsafePublish published;
}

// ✅ 方案1：synchronized 块中发布
class SafeBySync {
    final int x;
    SafeBySync() { x = 1; }
    static SafeBySync instance;
    synchronized static void publish() {
        instance = new SafeBySync();  // 同步保证可见性
    }
}

// ✅ 方案2：volatile 引用发布
class SafeByVolatile {
    final int x;
    SafeByVolatile() { x = 1; }
    static volatile SafeByVolatile instance;
    static void publish() {
        instance = new SafeByVolatile();  // volatile 保证可见性
    }
}

// ✅ 方案3：static 初始化（类加载天然线程安全）
class SafeByStatic {
    final int x;
    SafeByStatic() { x = 1; }
    static final SafeByStatic INSTANCE = new SafeByStatic();  // JVM 保证线程安全
}
```

**安全发布的常用方式**（必须满足至少一个条件）：

| 发布方式 | 可见性保证 |
|---------|-----------|
| `static` 初始化 | ✅ JVM 类加载机制保证 |
| `volatile` 引用 | ✅ volatile 写 Happens-Before 读 |
| `synchronized` 块 | ✅ unlock Happens-Before lock |
| `final` 字段（正确构造） | ✅ 对象正确发布后 final 字段可见 |
| `AtomicReference` / JUC 工具 | ✅ CAS 协议保证 |

## 易错点/踩坑

- ❌ **误解**：重排序是随意的
  - 重排序必须遵守 as-if-serial 语义，不能改变程序执行结果

- ❌ **误解**：volatile 完全禁止重排序
  - volatile 只禁止特定类型的重排序（volatile 读写之间）

- ❌ **忽略**：数据依赖不影响重排序
  - 编译器不会重排序有数据依赖关系的指令

- ✅ **正确理解**：重排序是为了性能优化
  - 单线程语义不变，多线程需要额外同步

## 代码示例

### volatile 禁止重排序的示例
```java
public class VolatileBarrier {
    private int a = 0;
    private volatile boolean flag = false;

    public void writer() {
        a = 1;           // 操作1
        flag = true;     // 操作2（volatile 写）
        // 插入 StoreStore 屏障
    }

    public void reader() {
        // 插入 LoadLoad + LoadStore 屏障
        if (flag) {      // 操作3（volatile 读）
            int i = a;   // 操作4 - 一定能读到 a=1
        }
    }
}
```

### 单例模式中的重排序问题
```java
public class Singleton {
    private static Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    // 问题：构造函数执行可能重排序
                    // 先写入引用，再执行构造函数体
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

// 解决：volatile 禁止构造函数的指令重排序
private volatile static Singleton instance;
```

## 图解/流程

### 重排序三层架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    指令重排序三层架构                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1. 编译器重排序（JIT Compiler）                           │   │
│  │    - 无数据/控制依赖的语句重排序                           │   │
│  │    - 最激进的优化                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           ↓                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 2. CPU 指令重排序（Pipeline / OoO Execution）            │   │
│  │    - 流水线乱序执行                                       │   │
│  │    - 保留性（WAR / WAW / RAW）依赖才阻塞                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           ↓                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 3. 内存系统重排序（HBA - Store Buffer / Invalidate Queue） │   │
│  │    - 写缓冲延迟写回                                        │   │
│  │    - 失效队列延迟处理                                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

as-if-serial 语义：
┌────────────────────────────────────────────────────────────────┐
│ "编译器和处理器可以重排序任意代码"                                │
│                                                                 │
│ 前提条件：                                                      │
│   ① 单线程执行结果不能改变                                       │
│   ② 必须遵守数据依赖性（RAW/WAR/WAW）                            │
│   ③ 必须遵守控制依赖性（需要额外同步）                            │
└────────────────────────────────────────────────────────────────┘
```

### volatile 屏障作用图

```
writer() 方法执行序列：

时间 →
┌────────┬────────┬───────────────────────────┐
│ 写 a=1 │ 写flag │ [StoreStore] [StoreLoad]  │
└────────┴────────┴───────────────────────────┘
                ↑
        volatile 写屏障

────────────────────────────────────────────────────

reader() 方法执行序列：

时间 →
┌────────────────────────┬────────┬────────┐
│ [LoadLoad][LoadStore]  │ 读flag │ 读 a   │
└────────────────────────┴────────┴────────┘
         ↑
    volatile 读屏障

屏障效果：
- StoreStore：防止 volatile 写之前的写被重排序到 volatile 写之后
- LoadLoad：防止 volatile 读之后的读被重排序到 volatile 读之前
- LoadStore：防止 volatile 读之后的写被重排序到 volatile 读之前
- StoreLoad：防止 volatile 写之后的读写被重排序到 volatile 读之前
```
