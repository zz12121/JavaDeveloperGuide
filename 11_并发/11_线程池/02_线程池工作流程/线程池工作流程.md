
# 线程池工作流程

## 核心结论

任务提交后按以下顺序处理：**核心线程 → 队列 → 非核心线程 → 拒绝策略**。

## 深度解析

### execute 流程

```
execute(Runnable command)
  ├─ 线程数 < corePoolSize → 创建核心线程执行
  ├─ 核心线程满 → offer 入队
  │   └─ 二次检查：线程池可能已关闭
  ├─ 队列满 → 线程数 < maximumPoolSize → 创建非核心线程
  └─ 线程数也满 → reject 拒绝策略
```

### 源码（简化）

```java
public void execute(Runnable command) {
    int c = ctl.get();
    
    // 1. 核心线程未满 → 创建核心线程
    if (workerCountOf(c) < corePoolSize) {
        if (addWorker(command, true)) return;
        c = ctl.get();
    }
    
    // 2. 入队
    if (isRunning(c) && workQueue.offer(command)) {
        int recheck = ctl.get();
        if (!isRunning(recheck) && remove(command))
            reject(command);
        else if (workerCountOf(recheck) == 0)
            addWorker(null, false);
    }
    
    // 3. 创建非核心线程  4. 拒绝
    else if (!addWorker(command, false))
        reject(command);
}
```

### Worker 生命周期

```
addWorker() → 创建 Worker → 启动线程
  → runWorker() → firstTask != null ? 执行 : getTask()
    → getTask() 循环：
      非核心线程: poll(keepAliveTime) → 超时返回 null → 退出
      核心线程:   take() → 阻塞等待
    → getTask() 返回 null → processWorkerExit() → 线程退出
```

### 关键细节

1. **二次检查**：入队后检查线程池状态，可能已关闭
2. **非核心超时**：用 `poll(keepAliveTime)` 实现
3. **核心不超时**：用 `take()` 无限等待

## 易错点与踩坑

### 1. 队列满的判断时机
- 任务提交时，先检查核心线程 → 再入队 → 再检查非核心线程 → 最后拒绝
- 不是"队列满了才创建非核心线程"，而是 **offer 失败才创建**
- `LinkedBlockingQueue.offer()` 无界时永远返回 true → 非核心线程永远不创建

### 2. addWorker 失败的多种原因
```java
// addWorker 可能失败的情况：
// 1. 线程池已 shutdown，不再接受新任务
// 2. 线程数超过 maximumPoolSize（ctl CAS 失败）
// 3. ThreadFactory 创建线程失败（OOM）
// 以上情况都会触发 reject
```

### 3. 核心线程满后的"二次检查"
- 入队后立即检查线程池状态，因为可能已被其他线程 shutdown
- 如果 shutdown 了且任务还在队列中 → remove + reject
- 如果线程数为 0 → 额外创建一个非核心线程处理队列任务

### 4. getTask 返回 null 的两种情况
- 非核心线程：`poll(keepAliveTime)` 超时返回 null
- 任意线程（allowCoreThreadTimeOut=true）：超时返回 null
- 核心线程（默认）：`take()` 永久阻塞，不返回 null
