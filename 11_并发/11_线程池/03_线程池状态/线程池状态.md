
# 线程池状态

## 核心结论

ThreadPoolExecutor 用一个 AtomicInteger（ctl）的高 3 位存储运行状态，低 29 位存储工作线程数。

## 深度解析

### 五种状态

| 状态 | 值 | 说明 |
|------|----|------|
| RUNNING | 111（-1） | 接受新任务，处理队列任务 |
| SHUTDOWN | 000（0） | 不接受新任务，处理队列任务 |
| STOP | 001（1） | 不接受新任务，不处理队列，中断进行中任务 |
| TIDYING | 010（2） | 所有任务已完成，线程数为 0 |
| TERMINATED | 011（3） | terminated() 执行完毕 |

### 状态转换

```
RUNNING
  ├─ shutdown() → SHUTDOWN
  ├─ shutdownNow() → STOP
SHUTDOWN
  ├─ 队列空 + 线程数为0 → TIDYING
  └─ shutdownNow() → STOP
STOP
  └─ 线程数为0 → TIDYING
TIDYING
  └─ terminated() 钩子方法完成 → TERMINATED
```

### ctl 设计

```java
private final AtomicInteger ctl = new AtomicInteger(ctlOf(RUNNING, 0));

// 高3位：运行状态
// 低29位：工作线程数

private static int runStateOf(int c)     { return c & ~CAPACITY; } // 取高3位
private static int workerCountOf(int c)  { return c & CAPACITY; }  // 取低29位
private static boolean isRunning(int c)  { return c < SHUTDOWN; }  // RUNNING < 0
```

### shutdown vs shutdownNow

| 方法 | 新任务 | 队列任务 | 正在执行任务 | 返回值 |
|------|--------|---------|------------|--------|
| shutdown() | 拒绝 | 继续执行 | 继续执行 | void |
| shutdownNow() | 拒绝 | 不执行（移除） | 尝试中断 | List<Runnable> 未执行任务 |

## 易错点与踩坑

### 1. ctl 的高低位不是简单的位运算
- 高 3 位是状态（RUNNING = -1，即 111...111），不是 0、1、2、3
- `isRunning(c)` 判断的是 `c < SHUTDOWN`，利用 RUNNING=-1 的特性
- **面试常问**：为什么 RUNNING 要设计成负数？→ 为了用 `c < SHUTDOWN` 一行判断

### 2. TIDYING 到 TERMINATED 不是自动的
- TIDYING 状态下会执行 `terminated()` 钩子方法
- 只有 `terminated()` 执行完毕后才是 TERMINATED
- 如果 `terminated()` 方法阻塞或死锁 → 线程池永远无法终止

### 3. shutdown 和 shutdownNow 的中断差异
```java
// shutdown: 不中断正在执行的任务，只中断等待 getTask() 的空闲线程
// shutdownNow: 对所有工作线程调用 Thread.interrupt()
// 注意：interrupt() 只是设置中断标志，不保证立即停止
// 如果任务不响应中断（不检查 isInterrupted()）→ shutdownNow 也停不下来
```

### 4. 状态不可逆
- 状态只能从左到右转换（RUNNING → SHUTDOWN → STOP → TIDYING → TERMINATED）
- **没有任何方法可以将线程池恢复到 RUNNING 状态**
