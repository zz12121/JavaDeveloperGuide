
# 线程池拒绝策略

## 核心结论

当线程池**队列满且线程数达到 maximumPoolSize** 时，触发拒绝策略。JDK 提供 4 种内置策略，可自定义。

## 深度解析

### 四种内置策略

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| **AbortPolicy**（默认） | 抛 RejectedExecutionException | 需要感知失败 |
| **CallerRunsPolicy** | 由提交任务的线程执行 | 降级，不丢弃 |
| **DiscardPolicy** | 静默丢弃任务 | 可容忍丢失 |
| **DiscardOldestPolicy** | 丢弃队列最旧任务，重新提交 | 保证最新任务执行 |

### 各策略详解

```java
// 1. AbortPolicy（默认）
// 抛异常，可以 catch 后记录日志/告警
public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
    throw new RejectedExecutionException("Task " + r.toString() +
                                         " rejected from " + e.toString());
}

// 2. CallerRunsPolicy（推荐生产使用）
// 提交线程自己执行，既不丢弃也不异常，但会阻塞提交线程
public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
    if (!e.isShutdown()) {
        r.run(); // 调用者线程执行
    }
}

// 3. DiscardPolicy
// 静默丢弃，无日志无异常
public void rejectedExecution(Runnable r, ThreadPoolExecutor e) { }

// 4. DiscardOldestPolicy
// 丢弃队列头部的最旧任务，重新提交当前任务
public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
    if (!e.isShutdown()) {
        e.getQueue().poll(); // 移除队头
        e.execute(r);        // 重新提交
    }
}
```

### 自定义拒绝策略

```java
public class LogRejectPolicy implements RejectedExecutionHandler {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        log.warn("任务被拒绝: pool={}, queue={}, active={}, rejected={}",
                 e.getPoolSize(), e.getQueue().size(),
                 e.getActiveCount(), e.getTaskCount());
        // 可以：记录到数据库、发送告警、降级处理
    }
}
```

### 触发条件

```
拒绝策略只在以下情况触发：
1. 线程池状态不是 RUNNING
2. 队列已满（workQueue.offer 返回 false）
3. 线程数达到 maximumPoolSize（addWorker 失败）

注意：使用无界队列时，永远不会触发拒绝策略
```

## 易错点与踩坑

### 1. AbortPolicy 异常被吞
```java
// ❌ 拒绝策略异常如果在 execute 内部被吞（不会，会抛到调用者）
// 但在 submit 场景下会被封装到 Future 中
Future<?> f = executor.submit(task);
// 如果任务提交时就被拒绝 → RejectedExecutionException 封装在 Future 中
// 不调用 f.get() → 完全不知道被拒绝了
```

### 2. CallerRunsPolicy 不是万能的
```java
// CallerRunsPolicy 让提交线程自己执行任务
// 风险：如果提交线程是 Tomcat 请求线程 → 
// Tomcat 线程被阻塞 → 无法处理新请求 → 服务"卡死"
// 在 Web 应用中使用 CallerRunsPolicy 要特别小心
```

### 3. DiscardPolicy 静默丢弃最危险
```java
// 任务被丢弃，无日志无异常
// 生产中几乎不应该使用 DiscardPolicy
// 如果必须丢弃，至少要自定义策略记录日志
```

### 4. DiscardOldestPolicy 的竞态条件
```java
// poll() + execute() 不是原子操作
// 高并发下可能：
// 线程A: poll() 移除旧任务
// 线程B: poll() 也移除了旧任务
// 结果：两个旧任务被移除，只有一个新任务被提交
// 但实际影响很小，因为队列通常不会在恰好同时满
```


# 拒绝策略使用场景

## 核心结论

根据业务容忍度选择拒绝策略：不能丢用 CallerRunsPolicy，可丢用 DiscardPolicy，需要感知用 AbortPolicy。

## 深度解析

### 选型决策树

```
任务丢失是否可接受？
├─ 不可接受 → CallerRunsPolicy（降级处理）
│              或自定义（记录+告警+降级）
├─ 可接受 → 只保留最新任务？
│           ├─ 是 → DiscardOldestPolicy
│           └─ 否 → DiscardPolicy
└─ 需要感知失败 → AbortPolicy（catch 异常后处理）
```

### 典型场景

|场景|推荐策略|原因|
|---|---|---|
|订单处理|CallerRunsPolicy|不能丢订单|
|日志写入|DiscardPolicy|丢几条日志无所谓|
|实时数据|DiscardOldestPolicy|保留最新数据|
|关键业务|自定义（告警+降级）|不能丢，需要监控|
|批量计算|AbortPolicy|感知失败后重试|

### CallerRunsPolicy 的"限流"效果

```
正常：提交线程 → 线程池执行
拒绝时：提交线程自己执行 → 阻塞提交线程 → 自然限流

效果：线程池过载时，提交速度自动降低，给线程池"喘息"时间
```

## 易错点与踩坑

### 1. Web 应用中 CallerRunsPolicy 的陷阱

```java
// Tomcat/Undertow 请求线程被线程池阻塞
// 如果线程池过载 → Tomcat 线程全部阻塞 → 整个服务无响应
// 更好的做法：AbortPolicy + 降级处理
new ThreadPoolExecutor.AbortPolicy() {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        log.warn("服务降级：任务被拒绝");
        // 走降级逻辑（返回默认值、缓存、MQ 等）
    }
};
```

### 2. 日志场景用 DiscardPolicy 也要小心

```java
// "丢几条日志无所谓" → 但如果是审计日志/错误日志呢？
// 建议：用有界队列 + 自定义策略，按日志级别区分
// ERROR/WARN 优先保留，DEBUG/INFO 可丢弃
```

### 3. 自定义策略中的死锁风险

```java
// 自定义策略中如果调用了线程池的 execute() → 可能递归拒绝
public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
    e.execute(r); // 如果还是满的 → 再次触发拒绝策略 → StackOverflow
}
```

### 4. 拒绝策略与事务

```java
// 如果任务涉及数据库事务，被拒绝后需要回滚或补偿
// DiscardPolicy 直接丢弃 → 事务状态不确定
// CallerRunsPolicy 执行 → 可能在非事务上下文中执行
```

# 自定义拒绝策略

## 先说结论

当线程池队列已满且线程数达到 maximumPoolSize 时，新提交的任务会触发拒绝策略。JDK 内置四种策略各有局限，生产环境通常需要**自定义 RejectedExecutionHandler**，实现日志记录、任务持久化、降级处理等业务逻辑。

## 深度解析

### JDK 内置四种拒绝策略

|策略|行为|适用场景|
|---|---|---|
|`AbortPolicy`（默认）|抛出 RejectedExecutionException|需要感知任务被拒绝|
|`DiscardPolicy`|静默丢弃任务|可容忍数据丢失|
|`DiscardOldestPolicy`|丢弃队列最旧任务，重新提交|关注最新数据|
|`CallerRunsPolicy`|由提交任务的线程（调用者）执行|简单降级，自动限流|

### 自定义拒绝策略

```java
public class LogAndPersistRejectHandler implements RejectedExecutionHandler {
    private static final Logger log = LoggerFactory.getLogger(LogAndPersistRejectHandler.class);
    private final Queue<Runnable> fallbackQueue = new ConcurrentLinkedQueue<>();

    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        // 1. 记录日志
        log.warn("线程池[{}]任务被拒绝! 活跃线程={}, 队列大小={}, 池大小={}",
            executor.toString(),
            executor.getActiveCount(),
            executor.getQueue().size(),
            executor.getPoolSize());

        // 2. 降级：存入本地队列稍后重试
        fallbackQueue.offer(r);
        
        // 3. 也可以发送到 MQ 等其他中间件
        // sendToMQ(r);
        
        // 4. 触发告警
        // alertService.notify("线程池任务被拒绝");
    }

    public int retryRejectedTasks(ThreadPoolExecutor executor) {
        int count = 0;
        Runnable r;
        while ((r = fallbackQueue.poll()) != null) {
            try {
                executor.execute(r);
                count++;
            } catch (RejectedExecutionException e) {
                fallbackQueue.offer(r);
                break;
            }
        }
        return count;
    }
}
```

### 常见自定义场景

|场景|实现思路|
|---|---|
|**日志记录**|记录被拒绝的任务信息和线程池状态|
|**持久化到MQ**|将任务发送到消息队列，异步消费|
|**降级到本地队列**|存入内存队列，定时重试|
|**数据库持久化**|将任务信息写入DB，后续补偿|
|**熔断告警**|触发告警通知运维人员|
|**计费/限流**|按用户维度计数拒绝次数，触发限流|

## 易错点/踩坑

- ❌ 使用 DiscardPolicy 静默丢弃，任务丢失无感知
- ✅ 至少要记录日志，方便事后排查
- ❌ CallerRunsPolicy 中执行耗时任务会阻塞调用线程
- ✅ CallerRunsPolicy 适合 IO 密集型的短任务
- ❌ 自定义拒绝策略中的处理逻辑太重
- ✅ 拒绝策略应尽量轻量，把任务转出去即可

## 代码示例

```java
// 生产环境组合方案：日志 + 降级到MQ
public class ProductionRejectHandler implements RejectedExecutionHandler {
    private static final Logger log = LoggerFactory.getLogger(ProductionRejectHandler.class);

    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        log.warn("任务被拒绝, active={}, queue={}, pool={}",
            executor.getActiveCount(),
            executor.getQueue().size(),
            executor.getPoolSize());
        // 降级到消息队列
        try {
            mqTemplate.convertAndSend("fallback.task.queue", r);
        } catch (Exception e) {
            log.error("降级发送MQ失败", e);
        }
    }
}

// 使用
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4, 8, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000),
    new ProductionThreadFactory("order-service"),
    new ProductionRejectHandler()  // 自定义拒绝策略
);
```
