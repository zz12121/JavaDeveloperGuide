
# 线程池异常处理

## 核心结论

execute 提交的任务异常直接抛到调用方线程（但会被 ThreadPoolExecutor catch 后吞掉）。submit 提交的任务异常封装在 Future 中，调用 `future.get()` 时抛出 ExecutionException。

## 深度解析

### execute 的异常处理

```java
// execute：异常会被 runWorker 的 try-catch 捕获
// 但不会影响后续任务执行
executor.execute(() -> {
    throw new RuntimeException("任务异常"); 
    // 异常被 ThreadPoolExecutor.afterExecute() 记录
    // 如果没有自定义 afterExecute → 异常被"吞掉"
});

// 通过自定义 ThreadPoolExecutor 处理
new ThreadPoolExecutor(...) {
    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        super.afterExecute(r, t);
        if (t != null) {
            log.error("任务执行异常", t);
        }
        if (r instanceof Future<?>) {
            try {
                ((Future<?>) r).get();
            } catch (InterruptedException | ExecutionException e) {
                log.error("Future任务异常", e.getCause());
            }
        }
    }
};
```

### submit 的异常处理

```java
// submit：异常封装在 Future 中
Future<?> future = executor.submit(() -> {
    throw new RuntimeException("任务异常");
});

try {
    future.get(); // 抛 ExecutionException
} catch (ExecutionException e) {
    log.error("任务失败: " + e.getCause().getMessage(), e.getCause());
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}
```

### UncaughtExceptionHandler

```java
// 设置线程工厂的异常处理器
ThreadFactory factory = r -> {
    Thread t = new Thread(r);
    t.setUncaughtExceptionHandler((thread, throwable) -> {
        log.error("线程 {} 未捕获异常", thread.getName(), throwable);
    });
    return t;
};
```

### 三种方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| submit + future.get() | 精确捕获每个任务异常 | 必须调用 get()，否则异常被吞 |
| 自定义 afterExecute | 全局统一处理 | 需要继承 ThreadPoolExecutor |
| UncaughtExceptionHandler | 线程级别处理 | execute 模式下不触发（被 runWorker catch 了） |

### 最佳实践

```java
// 1. 统一用 submit，不要用 execute
Future<?> f = executor.submit(task);

// 2. 统一 get 处理异常
try {
    f.get(timeout, unit);
} catch (ExecutionException e) {
    log.error("任务异常", e.getCause());
    // 降级处理
} catch (TimeoutException e) {
    f.cancel(true);
    log.warn("任务超时");
}

// 3. 或用 CompletableFuture（JDK8+）
CompletableFuture.runAsync(task, executor)
    .exceptionally(ex -> {
        log.error("任务异常", ex);
        return null;
    });
```

## 易错点与踩坑

### 1. submit 不调用 get() → 异常丢失（最高频问题）
```java
// ❌ 最常见的问题
List<Future<?>> futures = new ArrayList<>();
for (Task task : tasks) {
    futures.add(executor.submit(task)); // 异常被封装在 Future 中
}
// 没有调用 future.get() → 所有异常丢失

// ✅ 遍历获取结果
for (Future<?> f : futures) {
    try {
        f.get();
    } catch (ExecutionException e) {
        log.error("任务异常", e.getCause());
    }
}
```

### 2. UncaughtExceptionHandler 对 execute 无效
```java
// execute 中的异常被 runWorker 的 try-catch 捕获
// UncaughtExceptionHandler 根本不会被触发
// 因为异常没有逃出 Thread.run()
// 结论：UncaughtExceptionHandler 只对非线程池的线程有效
```

### 3. afterExecute 对 submit 异常的处理
```java
// submit 的任务被包装为 FutureTask
// FutureTask.run() 内部 catch 了异常
// afterExecute 收到的 Throwable 是 null
// 异常需要通过 Future.get() 获取
// 所以 afterExecute 中需要判断 r instanceof Future<?>
```

### 4. CompletableFuture 是更好的异常处理方案
```java
// CompletableFuture 提供了更优雅的异常处理
CompletableFuture.runAsync(task, executor)
    .thenAccept(result -> log.info("成功"))
    .exceptionally(ex -> {
        log.error("失败", ex);
        return null; // 降级返回值
    });
// 优势：链式调用、自动传播、不丢失异常
```
