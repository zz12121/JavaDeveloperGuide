
# 线程池调优

## 核心结论

线程池调优需要综合考虑**CPU 核数、任务类型、队列容量、超时时间、拒绝策略**。核心目标：CPU 利用率高、任务响应快、资源可控。

## 深度解析

### 调优步骤

```
1. 确定任务类型（CPU密集/IO密集/混合）
2. 设置核心线程数和最大线程数
3. 选择合适的队列和容量
4. 设置超时时间
5. 选择拒绝策略
6. 监控并调整
```

### 不同场景的推荐配置

```java
int cores = Runtime.getRuntime().availableProcessors();

// CPU 密集型
new ThreadPoolExecutor(
    cores, cores,
    0L, MILLISECONDS,
    new ArrayBlockingQueue<>(cores * 10), // 短队列
    new ThreadPoolExecutor.CallerRunsPolicy()
);

// IO 密集型
new ThreadPoolExecutor(
    cores * 2, cores * 4,
    60L, SECONDS,
    new LinkedBlockingQueue<>(1000),
    new ThreadPoolExecutor.CallerRunsPolicy()
);

// 混合型（拆分两个池）
ThreadPoolExecutor cpuPool = new ThreadPoolExecutor(cores, cores, ...);
ThreadPoolExecutor ioPool = new ThreadPoolExecutor(cores*2, cores*4, ...);

// 低延迟场景
new ThreadPoolExecutor(
    cores, cores * 2,
    30L, SECONDS,
    new SynchronousQueue<>(),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
// SynchronousQueue 减少排队延迟
```

### 常见问题及调优

| 问题 | 原因 | 调优 |
|------|------|------|
| 任务延迟高 | 队列太长 | 缩小队列 + 增大 maxPoolSize |
| CPU 利用率低 | 线程数不够 | 增大 corePoolSize |
| OOM | 无界队列 | 改为有界队列 |
| 频繁创建销毁线程 | keepAliveTime 太短 | 适当延长 |
| 拒绝任务 | 队列小 + 线程少 | 增大队列或线程数 |
| 上下文切换高 | 线程数过多 | 减少线程数 |

### 调优原则

1. **先监控再调优**：基于真实数据而非猜测
2. **一次只调一个参数**：便于定位效果
3. **压测验证**：调优后用 JMeter/wrk 压测
4. **渐进调整**：小幅调整，观察指标变化

## 易错点与踩坑

### 1. "一次只调一个参数"原则
```java
// ❌ 同时调整多个参数，无法判断效果
pool.setCorePoolSize(20);
pool.setMaximumPoolSize(50);
pool.setKeepAliveTime(120, SECONDS);
// 3个参数同时改 → 性能变化无法归因

// ✅ 一次调一个，观察指标
pool.setCorePoolSize(20);
// 观察 CPU、延迟、队列长度 → 再决定下一步
```

### 2. 监控数据的时间窗口选择
```java
// 不要基于瞬时数据做决策
// ❌ 队列长度 = 5 → "队列太小，加大"
// ✅ 过去 10 分钟队列长度 P99 = 5 → "队列大小合适"
// 使用滑动窗口统计而非瞬时值
```

### 3. SynchronousQueue 不是万能的
```java
// SynchronousQueue 减少了排队延迟
// 但代价是：每个任务需要即时分配线程
// 如果线程数已满 → 直接触发拒绝策略
// 不适合任务量大但允许排队的场景
```

### 4. 生产环境调优 Checklist
```
□ 队列是否有界？
□ 拒绝策略是否合理？
□ 是否有监控告警？
□ 线程命名是否有意义？（排查问题必备）
□ 是否配置了优雅关闭？
□ 是否有压测数据支撑？
□ 核心线程数 vs CPU 核数是否合理？
```
