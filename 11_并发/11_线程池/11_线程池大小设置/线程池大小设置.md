
# 线程池大小设置

## 核心结论

**CPU 密集型**：线程数 ≈ CPU 核数 + 1。**IO 密集型**：线程数 ≈ CPU 核数 × 2（或更多）。核心思想是**让 CPU 尽量不空闲**。

## 深度解析

### CPU 密集型

```
线程数 = CPU 核数 + 1（或 CPU 核数）

原因：
- CPU 密集型任务一直在计算，线程数过多导致频繁上下文切换
- 线程数 = CPU 核数时，每个核心一个线程，CPU 利用率最高
- +1 是为了防止某线程因缺页中断等原因暂停
```

### IO 密集型

```
线程数 = CPU 核数 × 2（推荐范围 2~10 倍）

更精确公式：
线程数 = CPU 核数 × (1 + IO等待时间 / CPU计算时间)

原因：
- IO 密集型任务大部分时间在等待（网络、磁盘），不占用 CPU
- 增加线程数可以在等待期间让其他线程使用 CPU
```

### 混合型

```
拆分为两个线程池：CPU 密集型池 + IO 密集型池
或
线程数 = CPU 核数 × (1 + 平均等待时间 / 平均计算时间)
```

### 实际建议

```java
int cpuCores = Runtime.getRuntime().availableProcessors();

// CPU 密集型
int cpuPoolSize = cpuCores + 1;

// IO 密集型（保守）
int ioPoolSize = cpuCores * 2;

// IO 密集型（激进）
int ioPoolSize = cpuCores * 10;

// 通用公式
// N = CPU × U × (1 + W/C)
// U = 目标 CPU 利用率（0~1）
// W = 等待时间
// C = 计算时间
```

### 队列大小

```
有界队列推荐：100~10000（根据任务量）

- 队列太小 → 频繁触发拒绝策略
- 队列太大 → 内存占用高，响应延迟
```

## 易错点与踩坑

### 1. CPU 核数获取方式
```java
// ✅ 正确：获取逻辑核数
int cores = Runtime.getRuntime().availableProcessors();

// ❌ 错误认知：
// - 不是物理核数（超线程下逻辑核 = 物理核 × 2）
// - 在 Docker 容器中可能获取到宿主机的核数（JDK 10 之前）
// - JDK 10+ 使用 -XX:ActiveProcessorCount 可覆盖
```

### 2. IO 密集型的"乘数"不能无限大
```
线程数 = CPU × (1 + W/C)

W = IO等待时间, C = CPU计算时间

如果 W/C = 100（数据库查询等），理论上线程数 = CPU × 101
但实际受限于：
- 内存（每个线程栈 1MB）
- 数据库连接数
- 文件句柄数
- 上下文切换开销

经验值：IO 密集型线程数一般不超过 200~500
```

### 3. 队列长度与线程数的平衡
```
队列太大 → 任务延迟高，maximumPoolSize 不生效
队列太小 → 频繁触发拒绝策略
线程太多 → 上下文切换开销大
线程太少 → CPU 利用率低

没有银弹，需要压测调优
```

### 4. 不要为每个任务创建一个线程池
```java
// ❌ 每个请求创建线程池 → 线程池本身开销 + 线程创建销毁
for (Request req : requests) {
    ExecutorService pool = Executors.newFixedThreadPool(10);
    pool.submit(() -> process(req));
    pool.shutdown();
}

// ✅ 全局共享线程池
private static final ExecutorService POOL = new ThreadPoolExecutor(...);
```

## 压测调优实战

### 压测目标确定

```
1. 确定业务目标：QPS、RT(响应时间)、成功率
2. 确定资源限制：CPU、内存、数据库连接数
3. 确定线程池参数范围：corePoolSize、maxPoolSize、queueCapacity
```

### 压测方案设计

```java
// 使用 JMH 或自定义压测
public class ThreadPoolBenchmark {
    
    // 测试不同线程数下的表现
    @Param({"4", "8", "16", "32", "64"})
    private int threadCount;
    
    @Benchmark
    public void testThroughput() {
        // 模拟 IO 密集型任务
        threadPool.submit(() -> {
            // 模拟 50ms IO 操作 + 10ms 计算
            Thread.sleep(50);
            doCompute(10);
        });
    }
}
```

### 调优过程示例

**场景**：HTTP 接口调用下游服务，平均耗时 100ms，目标 QPS 1000

```
初始配置：core=10, max=20, queue=100
压测结果：QPS=80, RT=120ms, CPU=15%

分析：线程数太少，CPU 利用率低

调整配置：core=50, max=100, queue=500
压测结果：QPS=450, RT=110ms, CPU=60%

分析：线程数增加，吞吐量提升，但仍有空间

调整配置：core=100, max=200, queue=1000
压测结果：QPS=950, RT=105ms, CPU=85%

分析：接近目标，继续微调

最终配置：core=120, max=200, queue=2000
压测结果：QPS=1050, RT=100ms, CPU=90%
✅ 满足目标
```

### 关键观察指标

| 指标 | 健康范围 | 异常表现 | 调整方向 |
|------|---------|---------|---------|
| CPU 利用率 | 70%~85% | <50% 或 >95% | 增加/减少线程数 |
| 队列积压 | <容量的50% | >容量的80% | 增大线程数或队列 |
| 拒绝率 | 0% | >0.1% | 增大队列或 maxPoolSize |
| RT(响应时间) | 接近理论值 | 明显升高 | 检查队列积压 |
| 吞吐量 | 接近目标 | 远低于目标 | 增加线程数 |

### 动态调优参数

```java
// 运行时调整线程池参数
ThreadPoolExecutor executor = ...;

// 调整核心线程数
executor.setCorePoolSize(100);

// 调整最大线程数
executor.setMaximumPoolSize(200);

// 调整存活时间
executor.setKeepAliveTime(60, TimeUnit.SECONDS);

// 注意：队列容量无法动态调整，需重新创建线程池
```
