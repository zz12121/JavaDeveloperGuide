# ReentrantReadWriteLock

## Q1：ReentrantReadWriteLock 的 state 是怎么设计的？

**A**：用一个 int 的 32 位拆分使用：

- **高 16 位**：读锁持有次数（共享计数）
- **低 16 位**：写锁重入次数（独占计数）

```java
// 读锁计数：右移 16 位
static int sharedCount(int c) { return c >>> SHARED_SHIFT; }

// 写锁计数：低 16 位
static int exclusiveCount(int c) { return c & EXCLUSIVE_MASK; }
```

读锁最大 65535，写锁最大 65535。每次读锁获取，state += 65536（SHARED_UNIT）；每次写锁获取，state += 1。

---

## Q2：读写锁的互斥规则是什么？

**A**：

| | 读锁 | 写锁 |
|---|------|------|
| **读锁** | ✅ 共享 | ❌ 互斥 |
| **写锁** | ❌ 互斥 | ❌ 互斥 |

- **读读共享**：多个线程可同时读
- **读写互斥**：有读锁时不能写，有写锁时不能读（写锁持有者自己除外）
- **写写互斥**：同一时刻只有一个写线程

## Q3：什么是锁降级？为什么不支持锁升级？

**A**：

**锁降级**：持有写锁 → 获取读锁 → 释放写锁。这是安全的，因为释放写锁时数据已更新，持有读锁保证后续读取的是最新数据。

**锁升级**：持有读锁 → 获取写锁。这会导致**死锁**：

- 线程 A 持有读锁，尝试获取写锁（等待其他读锁释放）
- 线程 B 持有读锁，也在尝试获取写锁
- 双方都在等对方释放读锁 → 死锁

所以 ReentrantReadWriteLock **不支持锁升级**。

---

## Q4：ReentrantReadWriteLock 有什么缺点？

**A**：

1. **写锁饥饿**：大量读线程时，写锁可能长时间获取不到（读锁不断到来，写锁永远排不上）
2. **不支持公平策略的完美实现**：非公平模式下读锁可能插队导致写饥饿
3. **不支持锁升级**：持有读锁时获取写锁会死锁

解决方案：
- JDK8 提供 `StampedLock`（乐观读）解决写饥饿问题
- 需要锁升级时用独占锁替代
