# QA_队列容量设置

## 面试题

### Q1: 线程池的队列容量设置多少合适？

**参考答案：**

没有标准答案，需要根据业务场景计算：

```
经验公式：队列容量 = 核心线程数 × 允许等待时间 / 平均任务执行时间

示例：
- 核心线程 10 个
- 任务平均执行 100ms
- 允许等待 5 秒
- 队列容量 = 10 × 5 / 0.1 = 500
```

考虑因素：
- CPU 密集型：队列容量可以小（任务执行快）
- IO 密集型：队列容量需要大（任务等待时间长）
- 突发流量：容量 = 峰值 × 安全系数

---

### Q2: 为什么要避免使用无界队列？

**参考答案：**

无界队列（如 `new LinkedBlockingQueue<>()`）容量为 `Integer.MAX_VALUE`，相当于无限堆积。

风险：
1. **OOM 风险**：生产速度 > 消费速度时，队列无限增长
2. **延迟增加**：任务积压越多，响应延迟越高
3. **资源浪费**：内存用于存储等待任务

解决方案：
- 指定有界容量
- 配合拒绝策略
- 监控队列积压

### Q3: ArrayBlockingQueue 的容量设为多少性能最好？

**参考答案：**

建议设为 **2 的幂次**（如 128、256、512、1024）。

原因：2^n 可以用位运算优化取模操作：

```java
// 非 2^n：需要取模
putIndex = (putIndex + 1) % capacity;

// 2^n：位运算更快
putIndex = (putIndex + 1) & (capacity - 1);
```

实测：1024 容量比 1000 容量吞吐量高约 10%~20%。

### Q4: 队列容量和线程数如何配合？

**参考答案：**

队列容量决定了线程池的"蓄水池"大小：

```
队列容量小 + 线程数多 → 频繁创建非核心线程 → CPU 开销大
队列容量大 + 线程数少 → 任务积压 → 响应延迟高
```

平衡原则：
- `corePoolSize`：正常负载时的线程数
- `queueCapacity`：允许的积压量
- `maxPoolSize`：突发流量时的最大线程数

推荐配置：
```java
new ThreadPoolExecutor(
    10,                                      // 核心线程
    50,                                      // 最大线程
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(200),           // 允许积压 200 个
    new CallerRunsPolicy()                    // 拒绝时降级
);
```
