# 队列容量设置

## 核心结论

队列容量设置需要平衡**内存占用、吞吐量、响应延迟**三个因素。无界队列（如 `new LinkedBlockingQueue()`）等于无限堆积，生产环境必须指定有界容量。

## 深度解析

### 容量设置决策树

```
任务特性？
├── 延迟敏感（需要快速响应）→ SynchronousQueue 或小队列
├── 内存敏感 → 小队列 + 拒绝策略
└── 普通任务 → 根据吞吐比计算容量
```

### 经验公式

```
最佳容量 = 生产速率 × 允许积压时间 - 消费者处理能力

示例：
- 生产速率：1000 qps
- 消费速率：800 qps
- 允许积压：30秒
- 最佳容量 ≈ 1000 × 30 - 800 × 30 = 6000
```

### 线程池场景容量计算

```java
// 线程池参数与队列容量关系
int corePoolSize = 10;      // 核心线程数
int maxPoolSize = 20;       // 最大线程数
int queueCapacity;          // 待计算

// 经验公式
queueCapacity = corePoolSize × 预计任务等待时间 / 平均任务执行时间

示例：
- 核心线程 10 个
- 任务平均执行 100ms
- 允许任务等待时间 5 秒
- 队列容量 = 10 × 5 / 0.1 = 500
```

### 不同场景推荐容量

| 场景 | 推荐容量 | 说明 |
|------|---------|------|
| CPU 密集型 | corePoolSize × 2 | 计算密集，队列缓冲作用小 |
| IO 密集型 | corePoolSize × 10~50 | 等待时间长，需要更大缓冲 |
| 批处理 | batchSize × 批次 | 根据业务批次大小设置 |
| 高优先级任务 | 较小 | 避免低优先级任务长期积压 |
| 低优先级任务 | 较大 | 允许更多积压 |

### 容量与拒绝策略的配合

```java
// 队列容量 vs 线程数 配合关系
// 总处理能力 = corePoolSize + (maxPoolSize - corePoolSize) × 利用率
// 队列容量决定了触发扩容的时机

// 容量太小 → 频繁触发拒绝
new ThreadPoolExecutor(10, 20, 60L, SECONDS,
    new ArrayBlockingQueue<>(10),  // ⚠️ 太小，队列很快满
    new AbortPolicy());

// 容量太大 → 内存浪费 + 延迟增加
new ThreadPoolExecutor(10, 20, 60L, SECONDS,
    new ArrayBlockingQueue<>(100000),  // ⚠️ 太大，内存浪费
    new AbortPolicy());

// ✅ 合理容量 = 预估积压量 × 安全系数
int expectedBacklog = 1000;  // 预估积压
int safetyFactor = 2;        // 安全系数
int queueCapacity = expectedBacklog * safetyFactor;  // 2000
```

## 易错点与踩坑

### 1. 无界队列 = OOM 风险
```java
// ❌ 危险：无界队列，生产速度 > 消费速度时 OOM
new LinkedBlockingQueue<>();          // 默认 Integer.MAX_VALUE
Executors.newFixedThreadPool(10);     // 内部用无界 LBQ

// ✅ 必须指定容量
new LinkedBlockingQueue<>(1000);
new LinkedBlockingQueue<>(expectedBacklog);

// ✅ 监控告警
if (queue.remainingCapacity() < threshold) {
    alert("队列即将满载");
}
```

### 2. 容量设置不考虑突发流量
```java
// ❌ 只考虑正常流量
int normalQps = 1000;
int capacity = normalQps;  // 1000

// ✅ 考虑突发流量（通常 3~10 倍）
int peakQps = 5000;
int capacity = peakQps;  // 5000
// 或用百分位：P99 × 安全系数
```

### 3. ArrayBlockingQueue 容量最好设为 2^n
```java
// ❌ 性能一般
new ArrayBlockingQueue<>(1000);  // 取模开销较大

// ✅ 性能更好
new ArrayBlockingQueue<>(1024);  // 2^10
new ArrayBlockingQueue<>(2048);  // 2^11

// 原因：2^n 可以用位运算优化取模
// index = (index + 1) & (capacity - 1)
```

### 4. 队列容量与内存估算
```java
// 每个队列元素占用内存估算
// 以 String 为例
String s = "hello";  // 对象头 12B + 字符数组 ~10B + 引用 8B ≈ 30B

// 队列内存占用
int capacity = 10000;
long memoryBytes = capacity * 30L;  // 约 300KB

// 估算公式
memory = capacity × 平均元素大小 × 安全系数

// 10000 个 String：~300KB
// 10000 个 byte[]（1KB）：~10MB
// 10000 个对象（复杂）：~2MB
```

### 5. 动态调整容量需要重建队列
```java
// ❌ LinkedBlockingQueue 没有 setCapacity 方法
queue.setCapacity(2000);  // ❌ 不存在

// ✅ 动态调整方案
class DynamicQueue<T> {
    private volatile BlockingQueue<T> queue;
    
    public void resize(int newCapacity) {
        BlockingQueue<T> newQueue = new LinkedBlockingQueue<>(newCapacity);
        queue.drainTo(newQueue);  // 迁移数据
        this.queue = newQueue;
    }
}

// ⚠️ 注意：迁移期间可能丢失数据
```

### 6. SynchronousQueue 不需要容量
```java
// SynchronousQueue 容量为 0，不需要设置
new SynchronousQueue<>();  // 够了

// 但如果需要限制"等待线程数"，可以用带容量的版本（实际上还是 0）
// 不存在带容量的 SynchronousQueue
```

## 生产配置示例

### HTTP 请求处理
```java
// 场景：Web 接口，QPS 1000，平均耗时 50ms
// 目标：允许 1 秒延迟
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    50,                                     // core = CPU × 2
    100,                                    // max = CPU × 4
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(5000),        // 积压 5 秒
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

### 异步任务处理
```java
// 场景：异步任务，平均耗时 1s，可以积压 30s
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,
    20,
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(300),          // 300 个任务
    new ThreadPoolExecutor.AbortPolicy()
);
```

### 日志处理
```java
// 场景：日志写入，允许丢少量日志
LinkedBlockingQueue<String> queue = new LinkedBlockingQueue<>(10000);
// 队列满时新日志会被丢弃（生产端自行处理）
```
