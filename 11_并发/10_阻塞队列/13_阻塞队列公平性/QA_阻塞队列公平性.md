# QA_阻塞队列公平性

## 面试题

### Q1: 公平锁和非公平锁的区别？

**参考答案：**

| 维度 | 公平锁 | 非公平锁 |
|------|--------|----------|
| 获取顺序 | 按等待时间（FIFO） | 可能"插队" |
| 吞吐量 | 较低（有检查开销） | 较高 |
| 饥饿 | 不会 | 可能 |
| 实现 | 需要维护等待队列 | CAS 直接抢 |

公平锁通过 `hasQueuedPredecessors()` 检查是否有更早等待的线程，非公平锁直接 CAS 抢锁。

---

### Q2: 为什么非公平锁吞吐量更高？

**参考答案：**

1. **减少上下文切换**：非公平锁允许线程在锁释放时立即抢锁，减少线程进入等待状态的次数
2. **减少入队开销**：公平锁每次都要检查等待队列，非公平锁直接 CAS
3. **利用持有时间**：如果持有锁的线程很快释放，后到的线程可以直接获取

实测：非公平锁比公平锁吞吐量高 3~5 倍。

---

### Q3: 什么场景下应该用公平锁？

**参考答案：**

需要保证**顺序公平**或**防止饥饿**的场景：

1. **金融交易系统**：防止后到的订单先成交导致套利
2. **任务调度系统**：按提交顺序处理，保证优先级
3. **资源分配**：防止某些线程永远得不到资源

其他场景默认用非公平锁（性能更好）。

---

### Q4: ArrayBlockingQueue 的 fair 参数有什么用？

**参考答案：**

`fair=true` 时，使用公平锁。等待时间最长的线程优先获取锁。

```java
// 公平队列：按提交顺序处理
ArrayBlockingQueue<Task> fairQueue = new ArrayBlockingQueue<>(100, true);

// 非公平队列（默认）：吞吐量更高
ArrayBlockingQueue<Task> unfairQueue = new ArrayBlockingQueue<>(100, false);
```

公平队列适合对顺序有要求的业务，非公平队列适合追求高吞吐的场景。

### Q5: 虚假唤醒是什么？如何避免？

**参考答案：**

虚假唤醒：`Condition.await()` 可能没有调用 `signal()` 也被唤醒（JDK 文档说明）。

避免方法：**始终用 while 循环检查条件**

```java
// ✅ 正确：while 循环
while (count == 0) {
    notEmpty.await();
}

// ❌ 错误：if 判断
if (count == 0) {          // 虚假唤醒时会继续执行
    notEmpty.await();
}
```
