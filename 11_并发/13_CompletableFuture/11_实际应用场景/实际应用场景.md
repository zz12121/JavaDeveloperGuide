
# CompletableFuture实际应用场景

## 核心结论

CompletableFuture 是 Java 异步编程的核心工具，适用于**多服务并行调用、超时控制、异步编排、熔断降级**等场景，能显著提升系统吞吐量。

## 深度解析

### 场景一：多服务并行调用

最经典场景——多个独立服务同时调用，总耗时 = 最慢的那个：

```java
// 串行：500ms + 300ms + 200ms = 1000ms
// 并行：max(500ms, 300ms, 200ms) = 500ms

CompletableFuture<User> userFuture = CompletableFuture.supplyAsync(() -> userService.getUser(id), ioPool);
CompletableFuture<Order> orderFuture = CompletableFuture.supplyAsync(() -> orderService.getOrders(id), ioPool);
CompletableFuture<Account> accountFuture = CompletableFuture.supplyAsync(() -> accountService.getAccount(id), ioPool);

// 等待全部完成
CompletableFuture.allOf(userFuture, orderFuture, accountFuture).join();

User user = userFuture.get();
Order order = orderFuture.get();
Account account = accountFuture.get();
```

### 场景二：异步调用编排

服务间有依赖关系，形成 DAG：

```java
// A -> B -> D
// A -> C -> D
// B 和 C 并行执行

CompletableFuture<String> A = supplyAsync(() -> callA());
CompletableFuture<String> B = A.thenCompose(a -> supplyAsync(() -> callB(a)));
CompletableFuture<String> C = A.thenCompose(a -> supplyAsync(() -> callC(a)));

CompletableFuture<String> D = B.thenCombine(C, (b, c) -> merge(b, c));
```

### 场景三：超时控制

```java
// JDK9+ orTimeout
CompletableFuture<String> future = supplyAsync(() -> remoteCall())
    .orTimeout(3, TimeUnit.SECONDS)
    .exceptionally(ex -> {
        if (ex instanceof TimeoutException) {
            return "fallback"; // 超时降级
        }
        return "error";
    });

// JDK8 方式：使用 Executor + ScheduledExecutorService
CompletableFuture<String> future = supplyAsync(() -> remoteCall())
    .thenApplyAsync(result -> result, delayedExecutor(3, TimeUnit.SECONDS, scheduledPool));
```

### 场景四：熔断降级

```java
public CompletableFuture<Result> queryWithFallback(String id) {
    return CompletableFuture.supplyAsync(() -> primaryService.query(id), ioPool)
        .exceptionally(ex -> {
            log.warn("主服务异常，降级到缓存", ex);
            return cacheService.get(id); // 降级
        })
        .thenCompose(result -> {
            if (result == null) {
                return CompletableFuture.supplyAsync(() -> secondaryService.query(id), ioPool);
            }
            return CompletableFuture.completedFuture(result);
        })
        .thenCombine(
            CompletableFuture.supplyAsync(() -> enrichmentService.enrich(id), ioPool),
            Result::merge
        )
        .orTimeout(5, TimeUnit.SECONDS)
        .exceptionally(ex -> Result.fallback()); // 最终兜底
}
```

### 场景五：批量并行处理

```java
List<String> ids = List.of("id1", "id2", "id3", ..., "id100");

// 并行处理，控制并发度
List<CompletableFuture<User>> futures = ids.stream()
    .map(id -> CompletableFuture.supplyAsync(() -> fetchUser(id), ioPool))
    .collect(Collectors.toList());

// 汇总结果
List<User> users = futures.stream()
    .map(CompletableFuture::join)
    .collect(Collectors.toList());
```

### 场景六：多数据源聚合查询

```java
public CompletableFuture<AggregateData> aggregate(String keyword) {
    CompletableFuture<List<Product>> esFuture = supplyAsync(
        () -> esSearch.search(keyword), esPool
    );
    CompletableFuture<List<Product>> dbFuture = supplyAsync(
        () -> dbSearch.search(keyword), dbPool
    );
    CompletableFuture<List<Product>> cacheFuture = supplyAsync(
        () -> cacheSearch.search(keyword), cachePool
    );

    return esFuture
        .thenCombine(dbFuture, (es, db) -> mergeAndDedup(es, db))
        .thenCombine(cacheFuture, (merged, cached) -> mergeAndDedup(merged, cached))
        .thenApply(list -> {
            // 排序、截断、格式化
            return list.stream()
                .sorted(Comparator.comparing(Product::getScore).reversed())
                .limit(20)
                .collect(Collectors.toList());
        });
}
```

### 生产环境最佳实践

| 要点 | 说明 |
|------|------|
| 指定线程池 | 不要用 commonPool，创建专用 IO/CPU 线程池 |
| 设置超时 | 每个外部调用都设 orTimeout，防止无限等待 |
| 异常处理 | 每个阶段都要 exceptionally 兜底 |
| 日志埋点 | 记录每个阶段的耗时，方便排查 |
| 线程池隔离 | IO 密集和 CPU 密集使用不同线程池 |
| 限制并发 | 用 Semaphore 控制对下游服务的并发请求数 |

## 易错点与踩坑

### 1. 并行调用未控制并发度
```java
// ❌ 10000 个任务全部提交 → 线程池过载
List<CompletableFuture<User>> futures = ids.stream()
    .map(id -> supplyAsync(() -> fetchUser(id), ioPool))
    .collect(Collectors.toList());

// ✅ 用 Semaphore 控制并发度
Semaphore semaphore = new Semaphore(50);
List<CompletableFuture<User>> futures = ids.stream()
    .map(id -> supplyAsync(() -> {
        semaphore.acquire();
        try { return fetchUser(id); }
        finally { semaphore.release(); }
    }, ioPool))
    .collect(Collectors.toList());
```

### 2. 异常导致 allOf 提前返回
```java
// ❌ allOf 在任一 future 异常时就进入异常状态
CompletableFuture.allOf(f1, f2, f3).join(); // f2 异常 → 整个 join() 抛异常
// f3 可能还在执行中，但调用方已经收到异常

// ✅ 对每个 future 做异常兜底
List<CompletableFuture<?>> safeFutures = futures.stream()
    .map(f -> f.exceptionally(ex -> {
        log.error("任务异常", ex);
        return null;
    }))
    .collect(Collectors.toList());
CompletableFuture.allOf(safeFutures.toArray(new CompletableFuture[0])).join();
```

### 3. 线程池生命周期管理
```java
// ❌ 线程池作为局部变量，任务可能没执行完就被 GC
public void process() {
    ExecutorService pool = Executors.newFixedThreadPool(10);
    supplyAsync(task, pool); // 如果方法返回后 pool 被 GC → 任务可能中断
    pool.shutdown(); // 但 shutdown 后新任务被拒绝
}

// ✅ 线程池作为成员变量，统一管理
@Service
public class AsyncService {
    private final ExecutorService ioPool = new ThreadPoolExecutor(...);
    private final ExecutorService cpuPool = new ThreadPoolExecutor(...);
    
    @PreDestroy
    public void shutdown() {
        ioPool.shutdown();
        cpuPool.shutdown();
    }
}
```

### 4. CompletableFuture 链的内存泄漏
```java
// 每个 thenApply 创建新的 CF → 长链会产生大量临时对象
// 正常情况下 GC 可以回收，但如果有引用持有 → 可能泄漏

// ❌ 持有过时的 CF 引用
class Service {
    private CompletableFuture<?> latestFuture; // 一直持有最新的 CF
    // 如果链很长且没有清理 → GC 无法回收中间的 CF
}
```
