# 锁获取流程（tryAcquire → 失败 → addWaiter入队 → acquireQueued自旋）

## 先说结论

AQS 独占模式获取锁的完整流程：`tryAcquire` 尝试获取 → 失败则 `addWaiter` 尾插法入队 → `acquireQueued` 自旋/park → 前驱为 head 时再次 `tryAcquire` → 成功则 `setHead` 成为新 head → 失败则 `park` 阻塞等待 `unpark` 唤醒。

## 深度解析

### 完整流程图

```
lock() / acquire(1)
│
├── 1. tryAcquire(1)                    ← 子类实现
│   ├── state==0
│   │   ├── 非公平: CAS(state,0,1) → 成功 → 返回
│   │   └── 公平: hasQueuedPredecessors()?
│   │       ├── 有等待者 → 失败 ↓
│   │       └── 无等待者 → CAS(state,0,1)
│   └── state!=0
│       ├── 当前线程==持有者 → setState(c+1) → 返回（重入）
│       └── 其他线程 → 失败 ↓
│
├── 2. addWaiter(Node.EXCLUSIVE)        ← 入队
│   ├── CAS tail → 成功 → 入队完成
│   └── CAS 失败 → enq() 自旋入队
│
├── 3. acquireQueued(node, 1)           ← 自旋+park
│   │
│   ├── for (;;) {
│   │   ├── 前驱 == head?
│   │   │   ├── YES → tryAcquire(1)
│   │   │   │   ├── 成功 → setHead(node) → 返回 ✅
│   │   │   │   └── 失败 ↓
│   │   │   └── NO ↓
│   │   │
│   │   ├── shouldParkAfterFailedAcquire(prev, node)
│   │   │   ├── prev.ws == SIGNAL → return true
│   │   │   ├── prev.ws > 0 (CANCELLED) → 跳过prev
│   │   │   └── prev.ws == 0 → CAS prev.ws = SIGNAL → return false
│   │   │
│   │   └── return true?
│   │       ├── YES → parkAndCheckInterrupt()
│   │       │   ├── LockSupport.park(this)  → ⭐ 阻塞
│   │       │   └── 被唤醒 → 检查中断 → 回到循环顶部
│   │       └── NO → 继续自旋
│   │   }
│   └── 异常 → cancelAcquire(node)
│
└── 4. 如果被中断过 → selfInterrupt()
```

### 各阶段详解

**阶段1：tryAcquire**

```java
// 非公平
if (c == 0 && compareAndSetState(0, 1)) → 获取成功
if (current == owner) setState(c + 1)     → 重入成功

// 公平（多一步检查）
if (c == 0 && !hasQueuedPredecessors() && compareAndSetState(0, 1))
```

**阶段2：addWaiter**

```java
// 快速路径：CAS 尾插
Node node = new Node(Thread.currentThread(), EXCLUSIVE);
if (tail != null) {
    node.prev = tail;
    if (compareAndSetTail(tail, node)) {
        tail.next = node;
        return node;
    }
}
// 慢速路径：enq 自旋
enq(node);
```

**阶段3：acquireQueued 自旋**

```
迭代1: 前驱!=head → shouldPark → prev.ws=0 → CAS设SIGNAL → false → 继续
迭代2: 前驱!=head → shouldPark → prev.ws=SIGNAL → true → park!
... 等待 unpark ...
迭代3: 被唤醒 → 前驱==head → tryAcquire → 成功 → setHead → 返回
```

### 时间线示例

```
时刻  线程A              线程B              麟列状态                state
 t0   lock→CAS(0,1)✅                      head哨兵                1
 t1                      lock→CAS(0,1)✗    head←B                1
 t2                      addWaiter(B)       head←B(tail)          1
 t3                      自旋:前驱==head    head←B                1
 t4                      tryAcquire✗        head←B                1
 t5                      shouldPark→设SIGNAL head←B                1
 t6                      park               head←B                1
 t7   unlock→tryRelease                      head←B                0
 t8   unparkSuccessor(B)                     head←B                0
 t9                      unpark唤醒          head←B                0
 t10                     自旋:前驱==head    head←B                0
 t11                     tryAcquire✅        B=head                1
 t12                     setHead(B)          B(head,thread=null)   1
```

## 易错点/踩坑

- ❌ acquireQueued 中的 park 会导致 CPU 空转——park 是真正的阻塞，释放 CPU
- ❌ 认为被唤醒后直接执行——被唤醒后回到自旋，仍需 tryAcquire 成功
- ✅ 整个流程保证锁的获取是公平的（公平模式下）和可中断的

## 代码示例

```java
// 用 ReentrantLock 跟踪完整流程
public class LockTraceDemo {
    private static final ReentrantLock lock = new ReentrantLock(true);

    public static void main(String[] args) throws Exception {
        lock.lock();
        System.out.println("T0: main 获取锁, state=" + lock.getHoldCount());

        Thread t1 = new Thread(() -> {
            System.out.println("T1: 尝试获取锁...");
            lock.lock();
            System.out.println("T1: 获取锁成功");
            lock.unlock();
        });

        t1.start();
        Thread.sleep(100); // 确保 t1 入队
        System.out.println("T0: 队列长度=" + lock.getQueueLength());

        lock.unlock();
        System.out.println("T0: 释放锁");

        t1.join();
    }
}
```
