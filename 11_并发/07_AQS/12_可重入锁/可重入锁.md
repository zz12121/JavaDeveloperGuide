# 可重入锁（state记录重入次数，同线程重复lock不阻塞）

## 先说结论

可重入锁（Reentrant）指同一线程可以多次获取同一把锁而不会死锁。AQS 的 state 记录重入次数，每次 lock() 时 state + 1，unlock() 时 state - 1，state 归零时锁完全释放。`ReentrantLock` 和 `synchronized` 都是可重入锁。

## 深度解析

### 重入原理

```
ThreadA:
  lock()   → state: 0 → 1 (首次获取)
  lock()   → state: 1 → 2 (重入)
  lock()   → state: 2 → 3 (重入)
  unlock() → state: 3 → 2
  unlock() → state: 2 → 1
  unlock() → state: 1 → 0 → ⭐ 完全释放 → unpark后继
```

### 源码中的重入处理

```java
// tryAcquire 中的重入
final boolean nonfairTryAcquire(int acquires) {
    Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        // 首次获取：CAS
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    }
    // ⭐ 重入检查：当前线程 == 持有者
    else if (current == getExclusiveOwnerThread()) {
        int nextc = c + acquires;
        if (nextc < 0) throw new Error("Maximum lock count exceeded");
        setState(nextc);  // 不需要 CAS，当前线程独占
        return true;
    }
    return false;
}

// tryRelease 中的重入释放
protected final boolean tryRelease(int releases) {
    int c = getState() - releases;
    if (Thread.currentThread() != getExclusiveOwnerThread())
        throw new IllegalMonitorStateException();
    boolean free = false;
    if (c == 0) {  // ⭐ state 归零才完全释放
        free = true;
        setExclusiveOwnerThread(null);
    }
    setState(c);
    return free;
}
```

### 重入的必要性

```java
// 递归方法
public void recursive(int depth) {
    lock.lock();     // 同一线程第二次 lock
    try {
        if (depth > 0) {
            recursive(depth - 1);  // 递归调用
        }
    } finally {
        lock.unlock();
    }
}

// 子类调用父类同步方法
class Parent {
    public synchronized void doSomething() {
        // 子类调用时持有锁
    }
}
class Child extends Parent {
    public synchronized void doSomething() {
        super.doSomething(); // 如果不可重入 → 死锁！
    }
}
```

### synchronized 也是可重入的

```java
public class ReentrantDemo {
    public synchronized void methodA() {
        methodB(); // 同一线程，可重入
    }
    public synchronized void methodB() {
        // 不会死锁
    }
}
```

### 最大重入次数

```java
if (nextc < 0) throw new Error("Maximum lock count exceeded");
```

`state` 是 `int` 类型（32位），理论最大重入次数约 2^31 - 1（2147483647），实际上不可能达到。

## 易错点/踩坑

- ❌ lock 次数 > unlock 次数——锁不会释放，其他线程永远等不到
- ❌ unlock 次数 > lock 次数——抛 `IllegalMonitorStateException`
- ✅ 最佳实践：lock/unlock 严格配对，推荐在 finally 中 unlock

## 代码示例

```java
// 可重入的实际应用
public class TaskProcessor {
    private final ReentrantLock lock = new ReentrantLock();

    public void processTask() {
        lock.lock();
        try {
            validateTask();    // 内部也 lock（重入）
            executeTask();     // 内部也 lock（重入）
        } finally {
            lock.unlock();
        }
    }

    private void validateTask() {
        lock.lock();
        try {
            // 验证逻辑（需要锁保护）
        } finally {
            lock.unlock();
        }
    }

    private void executeTask() {
        lock.lock();
        try {
            // 执行逻辑（需要锁保护）
        } finally {
            lock.unlock();
        }
    }
}
```

## 补充：synchronized 底层原理

### monitorenter / monitorexit

`synchronized` 编译后生成 `monitorenter` 和 `monitorexit` 字节码指令，由 JVM 的 `ObjectMonitor` 实现：

```cpp
// JVM 源码（objectMonitor.hpp）简化版
class ObjectMonitor {
    ObjectWaiter* _head;    // 等待队列（CXBQ / _WaitSet）
    ObjectWaiter* _tail;
    Thread* _owner;          // 当前持有者线程
    int _recursions;        // 重入次数
    intptr_t _count;        // 等待者计数
};

// monitorenter 字节码
void ObjectMonitor::enter(TRAPS) {
    Thread* self = THREAD;
    if (_owner == NULL) {           // 无锁 → CAS 抢
        if (Atomic::cmpxchg_ptr(self, &_owner, NULL) == NULL) return;
    }
    if (_owner == self) {           // 重入 → _recursions++
        _recursions++;
        return;
    }
    // 其他线程持有 → 入队等待（park）
    AddWaiter(&_WaitSet);
    // 阻塞等待...
}
```

### 锁升级机制（偏向锁 → 轻量级锁 → 重量级锁）

JVM 的 synchronized 锁有四种状态，逐级升级（**不可降级**）：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
         ↓          ↓          ↓
      线程ID      Mark Word   ObjectMonitor
      (Mark Word)  (CAS栈)    (内核态 park)
```

#### ① 偏向锁（Biased Lock）

- **适用场景**：单线程独占，无竞争
- **原理**：Mark Word 记录线程 ID，Mark Word 存储格式：`[epoch:2 | age:4 | 1|01]` → 偏向线程ID后变成 `[thread:54 | epoch:2 | 1|1|01]`
- **撤销**：其他线程尝试获取时，升级为轻量级锁

```java
// -XX:+UseBiasedLocking（默认 JDK 6+ 开启）
// -XX:BiasedLockingStartupDelay=0（关闭延迟）
// JDK 15+ 偏向锁已废弃（--disable-experiments）
```

#### ② 轻量级锁（Thin Lock）

- **适用场景**：多线程交替执行，无实际竞争
- **原理**：Mark Word 被复制到线程栈帧的 Lock Record，新 Mark Word 用 CAS 指向 Lock Record：`[ptr:62 | 00]`
- **获取**：线程栈的 Lock Record 记录了对象头，CAS 替换 Mark Word 成功即获取锁
- **自旋**：不 park 线程，在用户态自旋（默认 10 次）

```java
// -XX:PreBlockSpin=10（自旋次数）
// -XX:+UseSpinning（JDK 11+ 默认开启自适应自旋）
```

#### ③ 重量级锁（Fat Lock）

- **适用场景**：多线程激烈竞争
- **原理**：Mark Word 指向堆中的 `ObjectMonitor`，等待线程被 park，进入内核态（系统调用）
- **代价**：线程从用户态切换到内核态（~μs 级），远慢于轻量级锁的用户态 CAS（~ns 级）

```
用户态（CAS 自旋）        内核态（park）
    ↑                         ↑
  轻量级锁                  重量级锁
(快速，无切换)            (慢，有线程切换)
```

### 对比总结

| 维度 | 偏向锁 | 轻量级锁 | 重量级锁 |
|------|--------|----------|----------|
| 线程状态 | 用户态 | 用户态 | 内核态（park） |
| 标记位 [偏向:1 \| 00] | `101`（偏向） | `00`（轻量） | `10`（重量） |
| 性能开销 | 最低（几乎无） | 低（CAS） | 高（系统调用） |
| 适用场景 | 单线程 | 多线程交替 | 多线程竞争 |
| 锁升级 | 竞争时升级 | 竞争激烈升级 | 不可降级 |
| Mark Word | 存线程ID | 指向线程栈帧 | 指向ObjectMonitor |

> ⚠️ **synchronized 的非公平**：synchronized 只有重量级锁模式（偏向/轻量都是优化，非真正的锁），不支持公平模式。ReentrantLock 的公平锁通过 AQS 队列保证 FIFO。
