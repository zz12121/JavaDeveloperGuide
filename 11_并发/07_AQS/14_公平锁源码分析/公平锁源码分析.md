# 公平锁源码分析（hasQueuedPredecessors检查CLH队列前驱，有则让步保证FIFO）

## 先说结论

公平锁的核心在于 `hasQueuedPredecessors()` 方法——它检查 CLH 同步队列中是否存在当前线程的前驱等待者。如果队列非空（`tail != head`）且队首等待节点不属于当前线程（`first == null || first.thread != currentThread`），则公平锁放弃本次 CAS 抢锁，让步排队保证 FIFO 顺序。这就是公平锁与非公平锁在 `tryAcquire` 中的**唯一源码级差异**。

## 深度解析

### 1. 公平锁 vs 非公平锁的 tryAcquire 源码对比

```java
// ========== NonfairSync.tryAcquire（非公平）==========
protected final boolean tryAcquire(int acquires) {
    return nonfairTryAcquire(acquires);
}

// Sync.nonfairTryAcquire
final boolean nonfairTryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        if (compareAndSetState(0, acquires)) {  // ← 直接CAS抢锁，不看队列
            setExclusiveOwnerThread(current);
            return true;
        }
    }
    // ... 重入逻辑省略
}

// ========== FairSync.tryAcquire（公平）==========
protected final boolean tryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        if (!hasQueuedPredecessors() &&          // ← 先检查队列有没有前驱！
            compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    }
    // ... 重入逻辑省略
}
```

> **关键差异**：公平锁的 `if` 条件中多了 `!hasQueuedPredecessors()` 短路与判断。只有返回 `false`（无前驱）时才执行 CAS。

### 2. hasQueuedPredecessors() 逐行源码分析

```java
// AbstractQueuedSynchronizer.hasQueuedPredecessors()
public final boolean hasQueuedPredecessors() {
    Node t = tail;                     // 尾指针（读取一次，快照）
    Node h = head;                     // 头指针（读取一次，快照）
    Node s;
    return h != t &&                   // ① 队列不为空（有等待者）
        ((s = h.next) == null ||       // ② head.next == null：有线程正在入队中
         s.thread != Thread.currentThread()); // ③ 队首等待者不是当前线程
}
```

**三个条件拆解**：

| 条件 | 含义 | 场景 |
|------|------|------|
| `h != t` | 队列非空 | `head == tail` 说明队列空或只有哨兵，直接返回 `false` |
| `(s = h.next) == null` | head 还没有后继 | 正在执行 `addWaiter` 的 `CAS tail` 还没完成 `t.next = node` |
| `s.thread != currentThread()` | 首等待者不是自己 | 用于**重入判断**——持有锁的线程再次 lock 时可以跳过队列 |

**执行流程图**：

```
hasQueuedPredecessors()
│
├── h == t（队列为空）→ return false（无前驱，可以去抢锁）
│
├── h != t（队列非空）
│   ├── h.next == null（有线程正在入队）→ return true（有前驱，排队）
│   ├── h.next.thread == currentThread → return false（重入场景，可以抢锁）
│   └── h.next.thread != currentThread → return true（有前驱，排队）
│
└── return true → hasQueuedPredecessors() 返回 true
    → !hasQueuedPredecessors() 为 false
    → 公平锁放弃本次 CAS，进入 acquire() 入队排队
```

### 3. CLH 队列中等待节点的链表结构

```
初始状态（无等待者）:
    head ──→ null
    tail ──→ null
    h == t → hasQueuedPredecessors() 返回 false

T1 持锁，T2 入队后:
    head(哨兵) ←─prev── Node(T2, SIGNAL) ←─prev── Node(T3, 0)
       │                     │                        │
       └─────next────────────┘                        │
                             └────────next────────────┘
    tail 指向 Node(T3)
    head != tail → hasQueuedPredecessors() 检查 head.next
    head.next = Node(T2), thread = T2 ≠ T1 → 返回 true（T1重入例外）

T1 释放锁，T2 被唤醒:
    head = Node(T2)  ← setHead，清空 thread 和 prev
       │
       └─────next────→ Node(T3, 0)
    tail 仍指向 Node(T3)
```

> **head 节点是哨兵节点**，其 `thread` 字段为 `null`，不代表任何等待线程。真正的第一个等待者是 `head.next`。

### 4. 公平锁性能为什么不如非公平锁

```
                    非公平锁                           公平锁
T1: lock → 获取      lock → 获取                      lock → 获取
T2: lock → CAS抢     lock → CAS抢✗ → 入队 → park       lock → hasQueued→排队 → park
T3: lock → CAS抢     lock → CAS抢✗ → 入队 → park       lock → hasQueued→排队 → park
T1: unlock           unlock                            unlock
     → CAS抢锁被T2或T3拿到   → unpark(B)                    → unpark(B)
                        无需唤醒延迟                      唤醒有 park→unpark 延迟

额外开销对比:
┌──────────────┬──────────────────────────────┬──────────────────────────────┐
│   维度       │ 非公平锁                      │ 公平锁                       │
├──────────────┼──────────────────────────────┼──────────────────────────────┤
│ CAS 尝试     │ 新线程先 CAS 抢（失败才入队）  │ 每次先 hasQueued 检查         │
│ 线程切换     │ 抢锁成功则 0 次切换            │ 每个等待者至少 1 次 park/unpark│
│ 唤醒延迟     │ 新线程可能直接 CAS 成功        │ 必须等 unpark 唤醒再 tryAcquire│
│ 吞吐量       │ 更高（减少不必要的上下文切换）  │ 更低（每次严格排队）           │
│ 饥饿风险     │ 有（新线程可能连续插队）        │ 无（严格 FIFO）               │
└──────────────┴──────────────────────────────┴──────────────────────────────┘
```

**核心原因**：非公平锁允许"插队成功"跳过 park/unpark 的上下文切换开销。在高并发场景下，这个差异会被放大——公平锁每个线程都必须经历一次完整的 阻塞→唤醒→抢占 循环。

## 易错点/踩坑

- ❌ 认为 `hasQueuedPredecessors()` 返回 `true` 代表有前驱→当前线程不能获取锁——**重入场景例外**：持有锁的线程再次调用 `lock()` 时，虽然队列非空，但 `head.next.thread == currentThread`，返回 `false` 允许重入
- ❌ 认为 `h.next == null` 说明队列为空——实际是**有线程正在入队**（`addWaiter` 中 CAS tail 成功但还未执行 `t.next = node` 的中间状态）
- ❌ 认为公平锁能保证绝对公平——`hasQueuedPredecessors()` 和 `compareAndSetState` 之间存在 TOCTOU 竞态窗口，极端情况下仍可能插队
- ✅ `synchronized` 关键字底层只支持非公平模式，无法配置为公平锁

```java
/**
 * 公平锁源码级分析：对比 hasQueuedPredecessors 的行为
 */
public class FairLockSourceDemo {
    private static final ReentrantLock fairLock = new ReentrantLock(true);
    private static final ReentrantLock unfairLock = new ReentrantLock(false);

    public static void main(String[] args) throws Exception {
        // ========== 场景1：验证公平锁的 FIFO 顺序 ==========
        System.out.println("=== 公平锁 FIFO 验证 ===");
        for (int i = 0; i < 5; i++) {
            final int id = i;
            new Thread(() -> {
                fairLock.lock();
                try {
                    System.out.println("公平锁: 线程" + id + " 获取锁");
                } finally {
                    fairLock.unlock();
                }
            }, "Fair-" + id).start();
            Thread.sleep(10); // 确保按顺序入队
        }
        Thread.sleep(1000);

        // ========== 场景2：源码级 hasQueuedPredecessors 模拟 ==========
        System.out.println("\n=== hasQueuedPredecessors 逻辑模拟 ===");
        // 模拟 AQS 内部 CLH 队列结构
        //
        // 情况1：队列为空 → h == t → 返回 false（允许抢锁）
        //   head == null, tail == null
        //
        // 情况2：队列有等待者 → h != t 且 h.next.thread != current → 返回 true
        //   head(哨兵) → Node(T2) → Node(T3) → tail
        //   当前线程 T1 调用 hasQueued → true（有前驱，排队）
        //
        // 情况3：重入场景 → h != t 但 h.next.thread == current → 返回 false
        //   head(哨兵) → Node(T1) → tail
        //   当前线程 T1 再次调用 hasQueued → false（允许重入）
    }
}

/**
 * 关键源码对照（JDK 17 AbstractQueuedSynchronizer）：
 *
 * // 公平锁的 tryAcquire 中唯一多出的判断
 * if (!hasQueuedPredecessors() &&
 *     compareAndSetState(0, acquires)) {
 *     setExclusiveOwnerThread(current);
 *     return true;
 * }
 *
 * // 非公平锁直接 CAS，无前置检查
 * if (compareAndSetState(0, acquires)) {
 *     setExclusiveOwnerThread(current);
 *     return true;
 * }
 */
```
