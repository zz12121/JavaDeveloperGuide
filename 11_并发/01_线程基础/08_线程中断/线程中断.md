# 线程中断（Thread Interruption）

## 先说结论

Java 线程中断机制不是强制终止线程，而是一种**协作式**的通信方式：通过设置**中断标志**，告诉目标线程"有人希望你停止执行"。线程可以选择响应中断（抛出 InterruptedException 或主动检查）来优雅地结束。

## 深度解析

### 三个核心方法

```java
public class Thread implements Runnable {
    // 中断目标线程，设置中断标志为 true
    public void interrupt() {
        if (this != Thread.currentThread())
            checkAccess();
        synchronized (this) {
            interrupted = false;  // 先清除
            interrupted = true;  // 再设置（保证可见性）
        }
    }

    // 判断当前线程的中断标志（静态方法，同时清除中断标志）
    public static boolean interrupted() {
        return currentThread().isInterrupted(true);
    }

    // 判断目标线程的中断标志（实例方法，不清除标志）
    public boolean isInterrupted() {
        return isInterrupted(false);
    }
}
```

### 中断的语义

1. **`interrupt()`**：
   - 设置目标线程的中断标志为 `true`
   - 如果目标线程正处于阻塞状态（sleep/join/wait 等），会**立即抛出 `InterruptedException`**
   - 如果目标线程正在运行，会设置标志但不强制停止

2. **`isInterrupted()`**：
   - 只检查中断标志，不改变标志状态
   - 用于线程内部判断是否收到中断请求

3. **`interrupted()`**：
   - **静态方法**，检查**当前线程**的中断标志
   - **会清除**中断标志（连续调用两次第二次返回 false）

### 中断响应场景

```java
public class InterruptResponse {
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                try {
                    // 模拟耗时操作
                    Thread.sleep(1000);
                    // 业务逻辑
                } catch (InterruptedException e) {
                    // 收到中断后，InterruptedException 会清除中断标志
                    // 重新设置中断标志
                    Thread.currentThread().interrupt();
                    break;  // 退出循环
                }
            }
            System.out.println("线程安全退出");
        });

        t.start();
        Thread.sleep(3000);
        t.interrupt();  // 发送中断请求
    }
}
```

## 易错点/踩坑

- ❌ 误以为 `interrupt()` 会直接终止线程 —— 不会，只有在阻塞时才会抛异常
- ❌ 在 catch 中没有重新设置中断标志 —— `InterruptedException` 会清除标志，需要重新设置
- ❌ 使用 `interrupted()` 判断其他线程的中断状态 —— `interrupted()` 是静态方法，操作的是当前线程
- ❌ 在 finally 块中继续执行阻塞操作 —— 可能导致线程无法响应中断
- ✅ 使用 `volatile boolean` + `isInterrupted()` 实现可中断的循环
- ✅ 捕获 `InterruptedException` 后，应该重新中断或退出

## 代码示例

### 场景1：响应阻塞状态的中断

```java
public class InterruptBlocking {
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            try {
                System.out.println("线程开始睡眠...");
                Thread.sleep(10000);  // 睡眠10秒
                System.out.println("睡眠完成");
            } catch (InterruptedException e) {
                System.out.println("线程被中断，退出睡眠");
                // 注意：此时中断标志已被清除
            }
        });

        t.start();
        Thread.sleep(1000);
        t.interrupt();  // 中断睡眠中的线程
    }
}
```

### 场景2：可中断的任务执行

```java
public class InterruptTask implements Runnable {
    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            try {
                doWork();
            } catch (InterruptedException e) {
                // 收到中断，优雅退出
                Thread.currentThread().interrupt();
                break;
            }
        }
        cleanup();  // 清理资源
    }

    private void doWork() throws InterruptedException {
        // 检查是否有待处理的任务
        // 模拟阻塞操作
        synchronized (this) {
            wait();  // 等待任务
        }
    }

    private void cleanup() {
        System.out.println("清理资源，线程退出");
    }
}
```

### 场景3：使用 ExecutorService 的中断

```java
public class InterruptExecutor {
    public static void main(String[] args) throws InterruptedException {
        ExecutorService executor = Executors.newSingleThreadExecutor();

        Future<?> future = executor.submit(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                // 执行任务
            }
        });

        Thread.sleep(1000);
        future.cancel(true);  // 中断正在执行的任务
        executor.shutdownNow();  // 中断所有任务

        // 等待最多10秒让任务结束
        if (!executor.awaitTermination(10, TimeUnit.SECONDS)) {
            System.out.println("任务未能在规定时间内结束");
        }
    }
}
```

### 为什么不用 stop()？

`Thread.stop()` / `Thread.destroy()` 等方法已被 `@Deprecated`，**强烈不推荐使用**：

| 方法 | 废弃原因 | 风险 |
|------|---------|------|
| `stop()` | 强制终止线程，不给机会清理资源 | **破坏锁的一致性**：持有的锁会立即释放，可能暴露不一致的数据状态 |
| `suspend()` | 挂起线程但不释放锁 | **容易死锁**：其他等待该锁的线程永远无法继续 |
| `destroy()` | 强制销毁（从未实现）| 无法保证资源释放 |

**`stop()` 破坏锁一致性的示例**：
```java
class BankAccount {
    private int balance = 1000;

    public synchronized void transfer(int amount) {
        balance -= amount;  // 如果 stop() 在此处触发
        // ↑ 锁会被立即释放，balance 已扣减但目标账户还未增加
        // 数据不一致！且 synchronized 保护的原子性被破坏
        targetAccount.balance += amount;
    }
}
```

**正确替代方案**：使用 `interrupt()` + 协作式退出，让线程在安全点自行结束。

### LockSupport.park() 与中断机制

`LockSupport.park()` 是 JUC 框架的底层阻塞原语，与 `interrupt()` 有以下交互：

```java
// park() 可以被 interrupt() 唤醒（不会抛 InterruptedException！）
Thread t = new Thread(() -> {
    System.out.println("线程进入 park 前，中断标志: " + Thread.currentThread().isInterrupted());
    
    LockSupport.park();  // 阻塞在此
    
    // 被 unpark 或 interrupt 唤醒后，中断标志仍为 true（不清除！）
    System.out.println("线程被唤醒，中断标志: " + Thread.currentThread().isInterrupted());
});
t.start();
Thread.sleep(500);
t.interrupt();  // 唤醒 park，但不抛异常，中断标志保留
```

**`park()` vs `sleep()` 中断行为对比**：

| 方法 | 被 interrupt() 后 | 中断标志 | 是否抛异常 |
|------|-----------------|----------|-----------|
| `sleep()` | 立即唤醒 | **清除** | ✅ 抛 `InterruptedException` |
| `LockSupport.park()` | 立即唤醒 | **保留** | ❌ 不抛异常 |
| `wait()` | 立即唤醒 | **清除** | ✅ 抛 `InterruptedException` |

> **实践建议**：使用 `park()` 的代码应在唤醒后主动检查 `isInterrupted()`，判断是被 `unpark()` 唤醒还是被中断唤醒。
