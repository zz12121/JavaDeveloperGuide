# JDK 18+锁策略变化

## Q1：为什么 JDK 要移除偏向锁？

**参考答案：**

偏向锁虽然在单线程环境下能减少锁获取开销，但在现代 Java 应用中暴露出三个核心问题：

**1. 维护成本过高**
偏向锁的实现需要维护大量复杂状态，包括：
- 对象头中的偏向线程 ID、Epoch 字段
- 批量重偏向（Bulk Rebiasing）机制：当大量线程交替访问同一批对象时触发
- 批量撤销（Bulk Revoking）机制：当安全点检测到偏向失效时触发
这些机制增加了 JVM 代码复杂度，也带来了额外的运行时开销。

**2. 适用场景减少**
- 云计算和容器化：容器内线程生命周期短，偏向锁未热化就可能失效
- 微服务架构：锁竞争模式从"长期单线程持有"变为"高频短期竞争"
- G1/ZGC 等低停顿 GC：GC 时触发偏向锁批量撤销的开销更明显

**3. 替代技术成熟**
- 逃逸分析（Escape Analysis）能更准确地判断对象作用域
- Lock Elision 可以在运行时消除不必要的锁
- 轻量级锁的 CAS 操作在现代 CPU 上开销已很低

**面试加分项**：可以提及 JEP 423 的具体内容，以及 JDK 15 废弃 → JDK 17 默认关闭 → JDK 18 移除的三阶段演进过程。

## Q2：JDK 18 之后 synchronized 的锁升级路径是什么？

**参考答案：**

JDK 18 之后，synchronized 的锁升级路径简化为三阶段：

```
无锁 → 轻量级锁 → 重量级锁
```

**详细过程：**

**阶段1：无锁状态（Unlocked）**
- 新创建对象的 Mark Word 默认是无锁状态
- 锁标志位（lock bits）为 `01`，偏向标志位（biased_lock）为 `0`

**阶段2：轻量级锁（Lightweight Lock）**
- 当线程尝试获取锁时，在当前线程栈帧中创建锁记录（Lock Record）
- 通过 CAS 将对象头的 Mark Word 更新为指向锁记录的指针
- 如果 CAS 成功，获得轻量级锁
- 如果 CAS 失败，说明存在竞争，升级为重量级锁

**阶段3：重量级锁（Heavyweight Lock）**
- 向操作系统申请互斥锁（mutex）
- 对象头 Mark Word 更新为指向Monitor对象的指针
- 未获取锁的线程进入阻塞（BLOCKED）状态

**对比 JDK 8（有偏向锁）：**
```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
        ↑
   同一线程    锁竞争或偏向撤销
   再次进入
```

## Q3：移除偏向锁对现有代码有什么影响？

**参考答案：**

**对业务代码的影响：几乎为零**
- synchronized 语法完全不变
- 锁的语义保持一致
- 绝大多数业务代码无需任何修改

**需要关注的变更：**

1. **JVM 参数变化**
   - `-XX:+UseBiasedLocking` / `-XX:-UseBiasedLocking` 在 JDK 18+ 无效
   - `-XX:BiasedLockingStartupDelay` 参数被移除
   - 如果启动脚本中配置了这些参数，JVM 会输出警告

2. **监控和诊断工具**
   - JMX 中的偏向锁相关指标不再存在
   - `jstack`、`jstat` 输出中不再有偏向锁信息
   - 如果有监控脚本依赖这些指标，需要更新

3. **性能表现可能改善**
   - 消除了偏向锁撤销的额外开销
   - 减少了对象头的存储空间
   - 在高竞争场景下，避免了偏向→轻量的二次升级

4. **JOL（Java Object Layout）输出变化**
   - 对象头布局不再包含 "biased" 标记
   - 输出格式更为简洁

**面试加分项**：如果被问到"如何判断应用是否受此影响"，可以回答：
- 检查启动日志中是否有偏向锁相关参数警告
- 使用 JOL 查看对象头布局
- 对比 JDK 8 和 JDK 18+ 的性能基准测试结果
