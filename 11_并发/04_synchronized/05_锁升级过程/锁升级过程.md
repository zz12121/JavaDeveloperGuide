# 锁升级过程（无锁→偏向锁→轻量级锁→重量级锁）

## 先说结论

synchronized 的锁有四个级别：**无锁 → 偏向锁 → 轻量级锁 → 重量级锁**。锁升级过程是**单向不可逆**的，JVM 根据竞争情况自动升级，目的是在低竞争场景下尽量减少同步开销。JDK 15 之后偏向锁被废弃（JEP 374），默认只有无锁 → 轻量级锁 → 重量级锁。

## 深度解析

### 四种锁状态

| 锁状态     | 标志位 | 适用场景         | 实现机制                 |
| ---------- | ------ | ---------------- | ------------------------ |
| 无锁       | 01     | 无竞争           | 直接访问                 |
| 偏向锁     | 01     | 只有单线程访问   | Mark Word 记录线程 ID    |
| 轻量级锁   | 00     | 短时间竞争       | CAS + 自旋 + Lock Record |
| 重量级锁   | 10     | 激烈竞争         | ObjectMonitor + OS 互斥  |

### 升级过程详解

1. **无锁 → 偏向锁**：第一次获取锁时，CAS 将线程 ID 写入 Mark Word，后续同一线程重入只需比对线程 ID，无需 CAS
2. **偏向锁 → 轻量级锁**：其他线程尝试竞争时，撤销偏向锁，在当前线程栈帧创建 Lock Record，CAS 将 Mark Word 替换为指向 Lock Record 的指针
3. **轻量级锁 → 重量级锁**：自旋超过阈值或竞争线程超过 2 个，膨胀为重量级锁，创建 ObjectMonitor

### 锁升级的关键数据结构

- **偏向锁**：Mark Word 中存储 thread ID + epoch
- **轻量级锁**：Lock Record（栈帧中）存储 Displaced Mark Word
- **重量级锁**：Mark Word 指向 ObjectMonitor（堆中）

## 易错点/踩坑

- ❌ 以为锁可以降级
- ✅ 锁升级是单向不可逆的
- ❌ 以为偏向锁在 JDK 15+ 仍然默认启用
- ✅ JDK 15 默认禁用偏向锁，JDK 18 正式移除
- ❌ 以为轻量级锁比偏向锁更"高级"所以更好
- ✅ 不同锁适用于不同竞争程度

## 代码示例

```java
// 锁升级演示
public class LockEscalationDemo {
    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        // 偏向锁：单线程访问
        synchronized (lock) {
            System.out.println("偏向锁阶段：单线程重入");
        }

        // 轻量级锁：另一个线程短暂竞争
        Thread t1 = new Thread(() -> {
            synchronized (lock) {
                System.out.println("轻量级锁阶段：短暂竞争");
            }
        });
        t1.start();
        t1.join();

        // 重量级锁：多个线程激烈竞争
        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                synchronized (lock) {
                    try { Thread.sleep(100); } catch (Exception e) {}
                }
            }).start();
        }
        System.out.println("重量级锁阶段：激烈竞争");
    }
}
```

## 图解/流程

```
锁升级过程（单向不可逆）：

  无锁(01)           偏向锁(01)         轻量级锁(00)        重量级锁(10)
  ┌──────┐          ┌──────┐          ┌──────┐          ┌──────┐
  │ Mark │──首次──>│ Mark │──竞争──>│ Mark │──自旋──>│ Mark │

                         │                   │                   │
                    单线程重入              自旋+CAS          OS互斥
                    无代价                 短暂竞争           BLOCKED
```


# 偏向锁

## 先说结论

偏向锁是 JDK 6 引入的锁优化，核心思想是"**偏向第一个获取锁的线程**"。第一次加锁时将线程 ID 记录在 Mark Word 中，后续同一线程重入只需比对线程 ID 即可，无需任何 CAS 操作，实现近乎零开销的锁。

## 深度解析

### 工作原理

1. **首次加锁**：CAS 将当前线程 ID 写入 Mark Word，同时将偏向锁标志位置为 1（标志位 01 + biased=1）
2. **重入**：检查 Mark Word 中的线程 ID 是否等于当前线程，是则直接进入，无任何同步操作
3. **竞争**：其他线程尝试获取时，触发偏向锁撤销，升级为轻量级锁

### Mark Word 结构（偏向锁状态）

```
|---------------------------------------------|------|----|------|
|            Thread ID (54 bit)               |epoch|age|01/1  |
|---------------------------------------------|------|----|------|
```

### 批量重偏向与批量撤销

当偏向锁撤销达到阈值（默认 20 次）时，JVM 会进行**批量重偏向**——将对象重新偏向于当前正在竞争的线程。如果撤销继续增加到更高阈值（默认 40 次），则进行**批量撤销**——禁用该类的偏向锁，所有对象升级为轻量级锁。

## 易错点/踩坑

- ❌ 以为偏向锁适用于所有场景
- ✅ 偏向锁只适合几乎没有竞争的场景（单线程重入）
- ❌ 以为对象创建时就有偏向锁
- ✅ 默认有 4 秒延迟才会启用偏向锁（-XX:BiasedLockingStartupDelay=4000）
- ❌ 以为 hashCode 和偏向锁可以共存
- ✅ 调用 hashCode() 后偏向锁立即撤销（Mark Word 需要存 hashCode）

## 代码示例

```java
// 偏向锁延迟启动
public class BiasedLockDemo {
    // -XX:+PrintSafepointStatistics 可以观察偏向锁相关 safepoint
    public static void main(String[] args) throws InterruptedException {
        Object lock = new Object();

        // 同一线程多次获取 —— 偏向锁生效
        for (int i = 0; i < 100; i++) {
            synchronized (lock) {
                // 几乎零开销，只是比对线程 ID
            }
        }

        // 其他线程竞争 —— 偏向锁撤销
        Thread t = new Thread(() -> {
            synchronized (lock) {
                System.out.println("偏向锁撤销，升级为轻量级锁");
            }
        });
        t.start();
        t.join();
    }
}
```

# 偏向锁撤销

## 先说结论

偏向锁撤销是一个**昂贵的操作**，需要在 safepoint 暂停所有线程。JVM 通过**批量重偏向**（阈值 20）和**批量撤销**（阈值 40）来分摊撤销成本：当某类的偏向锁撤销次数过多时，先尝试批量重偏向给当前线程，若仍然频繁撤销则直接禁用该类的偏向锁。

## 深度解析

### 偏向锁撤销的条件

以下情况会触发偏向锁撤销：

1. 其他线程尝试竞争偏向锁对象
2. 调用了对象的 `hashCode()` 方法
3. 调用了对象的 `wait()/notify()` 方法

### 撤销过程（必须在 safepoint 进行）

1. 暂停持有偏向锁的线程
2. 遍历线程栈，查找该对象的 Lock Record
3. 如果存在 Lock Record，说明锁被重入，将 Mark Word 恢复为无锁状态
4. 将锁升级为轻量级锁
5. 唤醒暂停的线程

### 批量重偏向（阈值 20）

当同一个类的对象被撤销偏向锁达到 20 次时，JVM 会将该类的 epoch 加 1。所有对象下次检查时发现 epoch 不匹配，会尝试重新偏向于当前线程，而不是直接升级为轻量级锁。

### 批量撤销（阈值 40）

如果撤销继续增加到 40 次，JVM 会直接**禁用该类的偏向锁**，所有该类的对象都升级为轻量级锁或无锁状态。

## 易错点/踩坑

- ❌ 以为偏向锁撤销是低开销的
- ✅ 偏向锁撤销需要 safepoint 暂停所有线程，成本很高
- ❌ 以为批量重偏向和批量撤销是同一个概念
- ✅ 批量重偏向是重新偏向给新线程，批量撤销是彻底禁用偏向锁
- ❌ 忽视偏向锁撤销对 GC 停顿的影响
- ✅ 高并发应用中偏向锁撤销是 GC safepoint 停顿的重要原因

## 代码示例

```java
// 批量重偏向演示
public class BulkRebiasDemo {
    // -XX:BiasedLockingStartupDelay=0 立即启用偏向锁
    // -XX:+PrintBiasedLockingStatistics 打印偏向锁统计
    public static void main(String[] args) throws InterruptedException {
        List<Object> list = new ArrayList<>();

        // 线程1：创建 30 个偏向锁对象
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 30; i++) {
                Object obj = new Object();
                synchronized (obj) { /* 偏向 t1 */ }
                list.add(obj);
            }
        });
        t1.start(); t1.join();

        // 线程2：竞争这些对象
        // 前 20 个触发批量重偏向（epoch+1，重新偏向 t2）
        // 后 10 个正常升级为轻量级锁
        Thread t2 = new Thread(() -> {
            for (Object obj : list) {
                synchronized (obj) { /* 竞争 */ }
            }
        });
        t2.start(); t2.join();
    }
}
```

## 图解/流程

```
偏向锁撤销阈值机制：
  撤销次数    0 ────── 20 ────── 40 ────>
             │         │         │
             │ 正常撤销 │ 批量重偏向│ 批量撤销
             │         │ epoch+1 │ 禁用偏向锁
             │         │ 重新偏向 │ 该类所有对象

```

# 轻量级锁

## 先说结论

轻量级锁是 JDK 6 引入的锁优化，适用于**短时间的轻度竞争**场景。当前线程在栈帧中创建 Lock Record，将对象 Mark Word 复制到 Lock Record 中（Displaced Mark Word），然后 CAS 将 Mark Word 替换为指向 Lock Record 的指针。竞争线程通过**自适应自旋**等待，避免进入内核态的阻塞。

## 深度解析

### 加锁过程

1. 在当前线程栈帧中创建 Lock Record，包含 Displaced Mark Word（保存原始 Mark Word）和对象引用
2. CAS 尝试将对象 Mark Word 替换为指向 Lock Record 的指针（锁标志改为 00）
3. CAS 成功：获取锁
4. CAS 失败：说明有竞争，进入自旋

### 解锁过程

1. CAS 将 Displaced Mark Word 替换回对象头
2. CAS 成功：释放锁完成
3. CAS 失败：说明有其他线程竞争，锁已膨胀为重量级锁，唤醒被阻塞的线程

### 自适应自旋

JDK 6 之后引入自适应自旋：

- 如果上次自旋成功获取了锁，JVM 认为下次也大概率成功，增加自旋次数
- 如果上次自旋失败了，JVM 减少自旋次数甚至不自旋
- 自旋次数由 `-XX:PreBlockSpin` 控制（默认 10），自适应模式下由 JVM 动态调整

## 易错点/踩坑

- ❌ 以为轻量级锁适用于所有场景
- ✅ 轻量级锁只适合短时间竞争，长时间持有锁应直接用重量级锁
- ❌ 以为自旋是无限循环
- ✅ 自旋有次数限制（默认 10 次），自适应模式下动态调整
- ❌ 以为轻量级锁不消耗 CPU
- ✅ 自旋会消耗 CPU 周期，空闲等待 ≠ 无开销

## 代码示例

```java
public class LightweightLockDemo {
    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        // 轻量级锁：两个线程交替竞争，自旋等待
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 1000; i++) {
                synchronized (lock) { /* 短临界区，适合轻量级锁 */ }
            }
        });
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 1000; i++) {
                synchronized (lock) { /* 短临界区，适合轻量级锁 */ }
            }
        });
        t1.start(); t2.start();
        t1.join(); t2.join();
    }
}
```

# 轻量级锁膨胀

## 先说结论

轻量级锁在**自旋超过阈值**或**竞争线程过多**时会膨胀为重量级锁。膨胀过程包括：创建 ObjectMonitor、将 Mark Word 替换为指向 ObjectMonitor 的指针、竞争线程从自旋状态转入 BLOCKED 状态。膨胀不可逆。

## 深度解析

### 膨胀触发条件

1. **自旋失败**：当前线程自旋超过自适应自旋阈值仍然没有获取到锁
2. **多个竞争者**：竞争线程数 ≥ 2 时，自旋效率低下
3. **持有锁线程阻塞**：持有锁的线程调用了 wait()，需要 ObjectMonitor 支持

### 膨胀过程

1. 为对象分配（或从空闲列表获取）ObjectMonitor
2. 将当前线程设置为 ObjectMonitor 的 _owner
3. 将 Mark Word 替换为指向 ObjectMonitor 的指针（标志位改为 10）
4. 将自旋等待的线程移入 ObjectMonitor 的 _EntryList
5. 其他竞争线程进入 BLOCKED 状态，等待 _owner 释放锁

### 为什么要膨胀

自旋锁在竞争激烈时的问题：

- 多个线程空转浪费 CPU 资源
- 自旋期间持有锁的线程可能需要更长时间才能释放锁
- 不如直接让出 CPU，让其他线程工作

## 易错点/踩坑

- ❌ 以为自旋次数越多越好
- ✅ 激烈竞争时自旋反而浪费 CPU，膨胀为重量级锁更高效
- ❌ 以为轻量级锁膨胀后可以恢复
- ✅ 锁升级单向不可逆
- ❌ 以为膨胀过程不需要 safepoint
- ✅ 膨胀涉及堆内存分配（ObjectMonitor），可能在 safepoint 进行

## 代码示例

```java
// 演示轻量级锁膨胀
public class InflationDemo {
    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        // 线程1 长时间持有锁
        Thread t1 = new Thread(() -> {
            synchronized (lock) {
                try {
                    Thread.sleep(5000); // 长时间持有
                } catch (InterruptedException e) {}
            }
        });
        t1.start();
        Thread.sleep(100); // 确保 t1 先获取锁

        // 多个线程竞争 → 自旋超限 → 膨胀为重量级锁
        for (int i = 0; i < 5; i++) {
            new Thread(() -> {
                synchronized (lock) {
                    System.out.println(Thread.currentThread().getName() + " 获取锁");
                }
            }).start();
        }
    }
}
```

# 重量级锁

## 先说结论

重量级锁是 synchronized 锁的最高级别，依赖 **ObjectMonitor**（C++ 实现）和**操作系统互斥量（Mutex）**实现。竞争线程进入 BLOCKED 状态，涉及用户态到内核态的切换。ObjectMonitor 维护了 _owner、_EntryList、_WaitSet 三个关键字段来管理锁的持有和等待。

## 深度解析

### ObjectMonitor 结构

```c++
ObjectMonitor::_owner      // 持有锁的线程
ObjectMonitor::_EntryList  // 竞争锁的线程队列（BLOCKED）
ObjectMonitor::_WaitSet    // 调用 wait() 的线程队列（WAITING）
ObjectMonitor::_recursions // 重入次数
ObjectMonitor::_count      // 等待线程总数
```

### 线程状态流转

```
获取锁失败 → _EntryList (BLOCKED)
调用 wait() → _WaitSet (WAITING)
notify()   → _WaitSet → _EntryList (BLOCKED)
锁释放     → 唤醒 _EntryList 中的线程竞争
```

### 获取锁流程

1. 通过 `monitorenter` 或 `ACC_SYNCHRONIZED` 进入 ObjectMonitor::enter
2. 如果 _owner == null 或 _owner == 当前线程（可重入），获取成功
3. 否则通过 OS 的 mutex 将线程挂起，加入 _EntryList，线程状态变为 BLOCKED
4. 锁释放后，OS 唤醒 _EntryList 中的线程重新竞争

### 释放锁流程

1. _recursions--，如果 > 0 则只是减少重入次数
2. 如果 _recursions == 0，释放锁，_owner = null
3. 唤醒 _EntryList 中的一个线程（非公平）

## 易错点/踩坑

- ❌ 以为重量级锁一定比轻量级锁慢
- ✅ 无竞争时重量级锁的开销和轻量级锁接近，竞争激烈时重量级锁反而更合理
- ❌ 以为 notify() 唤醒的线程直接获取锁
- ✅ notify() 将线程从 _WaitSet 移到 _EntryList，仍需竞争锁
- ❌ 以为 _EntryList 是公平的 FIFO
- ✅ _EntryList 唤醒是非公平的

## 代码示例

```java
public class HeavyLockDemo {
    private static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        // wait/notify 演示重量级锁的 WaitSet 和 EntryList
        Thread waiter = new Thread(() -> {
            synchronized (lock) {
                try {
                    System.out.println("waiter: 等待中...");
                    lock.wait(); // 进入 _WaitSet
                    System.out.println("waiter: 被唤醒");
                } catch (InterruptedException e) {}
            }
        });

        Thread notifier = new Thread(() -> {
            synchronized (lock) {
                System.out.println("notifier: 发出通知");
                lock.notify(); // waiter 从 _WaitSet 移到 _EntryList
            }
        });

        waiter.start();
        Thread.sleep(100);
        notifier.start();
    }
}
```

## 关联知识点
