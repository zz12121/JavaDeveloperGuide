# synchronized 锁升级全过程

## 一句话总结

JVM 根据竞争激烈程度，将锁从轻到重自动升级：**无锁 → 偏向锁 → 轻量级锁 → 重量级锁**，整个过程**单向不可逆**。JDK 15 废弃偏向锁后，主流路径简化为 **无锁 → 轻量级锁 → 重量级锁**。

---


## 1. 无锁 → 偏向锁（JDK < 15）

**设计动机**：大部分锁不仅没有竞争，而且总是同一线程重入。把线程ID记在对象头里，重入时比对ID即可，零开销。

**实现原理**：
- Mark Word 结构：`| thread ID (54bit) | epoch (2bit) | age (4bit) | 01/1 |`
- 首次加锁：CAS 将当前线程ID写入 Mark Word
- 重入：比对 Mark Word 中的线程ID，匹配则直接进入

**撤销条件**（一旦触发就升级为轻量级锁）：
1. 其他线程尝试竞争
2. 调用了 `hashCode()`（Mark Word 空间存不下 threadID 和 hashCode）
3. 调用了 `wait()/notify()`（需要 ObjectMonitor）

**批量重偏向/撤销**：
```
撤销次数:  0 ────── 20 ────── 40 ────>
           │         │         │
         正常撤销   批量重偏向  批量撤销
                   epoch+1    禁用该类偏向锁
                   重新偏向    → 轻量级锁
```

> **为什么 JDK 15 废弃偏向锁？** 撤销需要在 safepoint 暂停所有线程（STW），批量撤销开销大；且偏向锁相关 bug 占了 JVM 同步代码的很大比例；现代轻量级锁已经够快，偏向锁的边际收益太小。

---

## 2. 偏向锁 → 轻量级锁（或直接 无锁 → 轻量级锁）

**设计动机**：大多数锁的竞争是短暂的，不需要让线程进入内核态阻塞。在用户态用 CAS + 自旋解决。

**加锁过程**：
```
1. 在当前栈帧创建 Lock Record
   ┌─────────────────┐
   │ Displaced Mark   │ ← 复制对象原始 Mark Word 到这里
   │ obj reference    │ ← 指向锁对象
   └─────────────────┘

2. CAS 将对象 Mark Word 替换为指向 Lock Record 的指针（标志位 → 00）
   成功 → 获取锁
   失败 → 有竞争，进入自旋

3. 解锁时 CAS 把 Displaced Mark Word 写回
   成功 → 释放完成
   失败 → 锁已膨胀为重量级锁，唤醒 EntryList 中的线程
```

**自适应自旋**（JDK 6+）：
- 上次自旋成功 → 这次增加自旋次数（认为大概率成功）
- 上次自旋失败 → 这次减少甚至不自旋（认为再自旋也是浪费）
- 比 `-XX:PreBlockSpin=10` 固定值更智能

---

## 3. 轻量级锁 → 重量级锁

**设计动机**：自旋在竞争激烈时反而浪费 CPU（多个线程空转，持有锁的线程得不到 CPU 时间执行）。不如让出 CPU，让 OS 调度。

**膨胀触发条件**：
1. 自旋超过自适应阈值仍然失败
2. 竞争线程 ≥ 2（自旋效率极低）
3. 持有锁的线程调用了 `wait()`（需要 ObjectMonitor 的 _WaitSet）

**膨胀过程**：
```
1. 分配 ObjectMonitor
2. 持有锁线程 → _owner
3. Mark Word → 指向 ObjectMonitor（标志位 → 10）
4. 自旋中的线程 → 移入 _EntryList → BLOCKED 状态
```

**ObjectMonitor 结构**：
```c++
_owner       // 持有锁的线程
_EntryList   // 竞争锁的线程队列（BLOCKED）
_WaitSet     // 调用 wait() 的线程（WAITING）
_recursions  // 重入次数
```

**线程状态流转**：
```
竞争失败 → _EntryList(BLOCKED) → 获取锁成功 → _owner → 释放锁 → 唤醒EntryList
                  ↑                                                      │
持有锁 + wait() → _WaitSet(WAITING) → notify() → _EntryList(BLOCKED) ───┘
```

> **关键**：notify() 不会让线程直接获取锁！它是从 _WaitSet 移到 _EntryList，线程需要重新竞争锁。

## 4. 四种锁状态对比

| 锁状态 | 标志位 | 适用场景 | 开销来源 | JDK 15+ |
|--------|--------|---------|---------|---------|
| 无锁 | 01 | 无竞争 | 无 | ✅ |
| 偏向锁 | 01/1 | 仅单线程重入 | 零（比对线程ID） | ❌ 废弃 |
| 轻量级锁 | 00 | 短时间轻度竞争 | CAS + 自旋（用户态） | ✅ |
| 重量级锁 | 10 | 激烈竞争 | OS 上下文切换（内核态） | ✅ |

## 5. 与其他概念的关系

```
                        ┌──────────────┐
                        │   CAS 原子操作  │ ← 轻量级锁的底层实现
                        └──────┬───────┘
                               │
┌──────────────┐        ┌──────┴───────┐        ┌──────────────┐
│  Mark Word   │◄──────│   锁升级过程   │──────►│ ObjectMonitor│
│ (对象头存储)  │        │  (状态机)      │        │ (C++实现)     │
└──────────────┘        └──────┬───────┘        └──────────────┘
                               │
                    ┌──────────┼──────────┐
                    │          │          │
               ┌────┴───┐ ┌───┴───┐ ┌───┴────┐
               │锁消除   │ │锁粗化  │ │自适应自旋│
               │(JIT优化)│ │(JIT优化)│ │(JDK6+) │
               └────────┘ └───────┘ └────────┘
```

- **CAS** → 轻量级锁加锁/解锁的底层操作
- **Mark Word** → 对象头中存储锁状态的数据结构
- **ObjectMonitor** → 重量级锁的 C++ 实现，管理 _EntryList 和 _WaitSet
- **锁消除/锁粗化** → JIT 编译器对锁的优化（和锁升级是正交的优化维度）
- **ReentrantLock** → 基于 AQS 实现，不依赖对象头的锁升级机制
