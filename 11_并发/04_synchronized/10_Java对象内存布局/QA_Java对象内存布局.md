# Java对象内存布局（对象头/实例数据/对齐填充）

## Q1：Java 对象在内存中的布局是怎样的？

**答：** Java 对象在 JVM 堆内存中的布局由三部分组成：**对象头（Object Header）**、**实例数据（Instance Data）**、**对齐填充（Padding）**。

### （1）对象头（Object Header）

对象头包含两部分（数组对象还有第三部分）：

**① Mark Word（8 字节）**
- 存储对象运行时数据，包括 hashcode、GC 分代年龄、锁状态标志、偏向线程 ID 等
- 采用**复用设计**，同一块 64 bit 空间在不同锁状态下存储不同内容
- 是 synchronized 锁机制的底层实现基础

**② Klass Pointer（类型指针，4~8 字节）**
- 指向对象所属类在方法区的 `Class` 对象元数据
- 开启指针压缩（`-XX:+UseCompressedOops`，默认开启）后为 4 字节，否则 8 字节

**③ 数组长度（4 字节，仅数组对象）**
- 只有数组类型才有的额外字段，记录数组长度

### （2）实例数据（Instance Data）

存储对象的有效信息，包括：
- 从父类继承的所有字段
- 子类自身定义的字段
- 字段按宽度递减排列（`-XX:FieldsAllocationStyle=0`）：long/double → int/float → short/char → byte/boolean → 引用
- 目的：减少内存碎片，提升 CPU Cache Line 命中率

### （3）对齐填充（Padding）

- 对齐单位为 **8 字节**，对象大小必须是 8 的整数倍
- 不是必须的，仅在对象大小不足 8 字节倍数时填充 0x00
- 目的：配合 CPU 的内存对齐机制，提升内存访问效率

### 对象大小计算示例

```
空对象：  12（对象头）+ 0（实例数据）+ 4（填充）= 16 字节
含int字段：12（对象头）+ 4（int）+ 0（填充）= 16 字节
含long字段：12（对象头）+ 8（long）+ 4（填充）= 24 字节
int数组[10]：16（头+长度）+ 40（10×4B）+ 0（填充）= 56 字节
```

## Q2：Mark Word 中存储了哪些信息？不同锁状态下有什么区别？

**答：** Mark Word 是 64 bit 的复用数据结构，其存储内容随锁状态动态变化。通过最后 2 bit 的**锁标志位（lock）** 区分不同状态。

### 各状态的 Mark Word 内容对比

| 锁状态 | lock | biased | 存储内容（62 bit） | 典型场景 |
|-------|------|--------|------------------|---------|
| **无锁** | 01 | 0 | unused(25) + **hashcode**(31) + unused(1) + age(4) | 对象刚创建，未被任何线程锁定 |
| **偏向锁** | 01 | 1 | **thread ID**(54) + epoch(2) + unused(1) + age(4) | 仅有一个线程反复进入同步块 |
| **轻量级锁** | 00 | — | **Lock Record 指针**(62) | 有轻微竞争，CAS 抢锁成功 |
| **重量级锁** | 10 | — | **Monitor 指针**(62) | 竞争激烈，锁膨胀为重量级锁 |
| **GC 标记** | 11 | — | 空 | GC 阶段临时标记 |

### 关键变化说明

1. **无锁 → 偏向锁**：hashcode 被覆盖，替换为偏向线程 ID 和 epoch
2. **偏向锁 → 轻量级锁**：整个 62 bit 替换为指向当前线程栈中 Lock Record 的指针
3. **轻量级锁 → 重量级锁**：Lock Record 指针替换为指向 ObjectMonitor 的指针
4. **GC 标记**：GC 过程中使用的临时状态，lock 标志位为 11

### 高频面试追问

**Q：为什么 hashcode 被清除后就无法进入偏向锁？**

因为偏向锁需要将 Mark Word 的 54 bit 存储线程 ID，而 hashcode 占据了其中 31 bit。两者不能同时存在，所以一旦计算了 hashcode，对象的偏向锁标记位会被置为 0，后续只能走轻量级锁或重量级锁。

## Q3：什么是对象对齐填充？为什么要 8 字节对齐？

**答：**

### 什么是对齐填充？

对齐填充（Padding）是在对象头和实例数据之后，为了使对象总大小达到 8 字节的整数倍而补齐的空字节（0x00）。

### 为什么需要 8 字节对齐？

**（1）CPU 内存访问效率**
- CPU 不是逐字节访问内存的，而是按**字长（Word）**为单位访问（64 位 CPU 一次读写 8 字节）
- 8 字节对齐保证对象地址的低 3 位为 0，CPU 可以一次读取完整的对象数据
- 如果不对齐，CPU 可能需要多次内存访问才能读取一个对象

**（2）GC 效率**
- 垃圾收集器在扫描堆时需要移动对象（Copying GC）
- 8 字节对齐使得对象移动时的地址计算更简单高效
- 卡片表（Card Table）以 512 字节为粒度，8 字节对齐的地址计算能被整除

**（3）HotSpot 设计选择**
- 64 位 JVM 的 Oop（Ordinary Object Pointer）操作以 8 字节为基本单位
- Mark Word 本身就是 8 字节，以 8 为对齐单位是自然选择

### 什么时候不需要填充？

当对象头 + 实例数据的大小恰好是 8 的倍数时，不需要对齐填充：

```java
// 空对象：12 字节（头）→ 需要填充 4 字节 → 总计 16 字节
class Empty {}                     // size = 16B

// 一个 int：12 + 4 = 16 → 已对齐，不需要填充
class OneInt { int a; }            // size = 16B

// 一个 long：12 + 8 = 20 → 需要填充 4 字节 → 总计 24 字节
class OneLong { long a; }          // size = 24B

// 两个 int：12 + 8 = 20 → 需要填充 4 字节 → 总计 24 字节
class TwoInt { int a; int b; }     // size = 24B

// 三个 int：12 + 12 = 24 → 已对齐，不需要填充
class ThreeInt { int a; int b; int c; }  // size = 24B
```

### 8 字节 vs 其他对齐值？

理论上 4 字节对齐也可以工作（32 位 JVM 就是用 4 字节对齐），但 64 位 JVM 选择 8 字节是为了：
- 与 64 位 CPU 的字长匹配
- 与 Mark Word 的 8 字节大小一致
- 简化 GC 的内存管理逻辑
