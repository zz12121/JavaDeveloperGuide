# JDK 15偏向锁废弃

## Q1：为什么 JDK 15 要废弃偏向锁？

**A**：JEP 374 给出了四个原因：

1. **撤销成本高**：偏向锁撤销需要 safepoint 暂停所有线程，是 GC 停顿的重要来源
2. **收益有限**：现代高并发应用中，"几乎没有竞争"的场景越来越少
3. **代码复杂**：偏向锁是 JVM 中 bug 密度最高的子系统之一
4. **维护不值得**：随着并发编程模式的发展，偏向锁的边际价值越来越小

---

## Q2：偏向锁废弃后 synchronized 的锁升级路径？

**A**：废弃后 synchronized 的锁升级路径变为：

**无锁 → 轻量级锁 → 重量级锁**

少了偏向锁阶段。首次加锁直接进入轻量级锁（CAS + Lock Record），竞争时自旋，自旋失败膨胀为重量级锁。

对绝大多数应用来说影响极小，因为轻量级锁在短竞争场景下性能已经足够好。

---

## Q3：如果面试官问到偏向锁，还需要了解吗？

**A**：需要了解，原因：

1. JDK 8~14 仍然是大量生产环境的主流版本，这些版本中偏向锁仍然是默认启用的
2. 理解偏向锁原理是理解锁升级整体机制的必要环节
3. 面试中考察偏向锁主要是考对 synchronized 底层原理的理解深度，不一定要用到最新版本

回答策略：先解释偏向锁原理，再补充说明 JDK 15 之后已废弃及其原因，展示知识的完整性和时效性。


> **代码示例：JDK 版本间的锁升级路径对比**

```java
// JDK 8~14: 无锁 → 偏向锁 → 轻量级锁 → 重量级锁
// JDK 15~17: 无锁 → 轻量级锁 → 重量级锁（偏向锁默认禁用）
// JDK 18+:   无锁 → 轻量级锁 → 重量级锁（偏向锁彻底移除）

Object obj = new Object();
synchronized (obj) {
    // JDK 8:  偏向锁（Mark Word 末2位=01，记录线程ID）
    // JDK 18: 轻量级锁（Mark Word 末2位=00，指向 Lock Record）
}

// 如果需要强制关闭偏向锁（JDK 8~14）:
// JVM 参数: -XX:-UseBiasedLocking
```
