# 偏向锁 vs 轻量级锁（无竞争用偏向，短竞争用轻量）

## Q1：偏向锁和轻量级锁有什么区别？

**A**：

| 维度     | 偏向锁               | 轻量级锁             |
| -------- | -------------------- | -------------------- |
| 场景     | 无竞争（单线程重入）  | 短时间轻度竞争       |
| 机制     | 记录线程 ID          | Lock Record + CAS    |
| 重入开销 | 零                   | 极小                 |
| 竞争处理 | 撤销偏向锁           | 自旋等待             |
| 状态     | JDK 15 已废弃        | 始终可用             |

一句话：**偏向锁假设没有竞争，轻量级锁假设竞争很短暂**。

---

## Q2：什么场景下偏向锁比轻量级锁好？

**A**：偏向锁在**单线程反复获取同一把锁**的场景下最优。例如：

- GUI 事件处理中的同步（通常是同一个事件线程处理）
- 单例模式的实例方法（虽然单例理论上没有并发问题，但使用者可能加 synchronized）

此时偏向锁的重入开销为**零**（只需比对线程 ID），而轻量级锁每次重入仍需要检查 Lock Record。

---

## Q3：为什么 JDK 15 废弃了偏向锁？

**A**：三个原因：

1. **维护成本高**：偏向锁撤销需要在 safepoint 暂停所有线程，成为 GC 停顿的主要来源之一
2. **收益递减**：现代应用并发度高，纯粹的"无竞争"场景越来越少
3. **代码复杂**：偏向锁相关代码占 JVM 同步子系统的很大比例，维护负担重

JDK 18 正式移除后，synchronized 的默认优化路径是无锁 → 轻量级锁 → 重量级锁。


> **代码示例：JDK 8 下观察偏向锁 vs 轻量级锁（需 -XX:+UseBiasedLocking）**

```java
// JDK 8 默认启用偏向锁，延迟 4 秒开启
Object obj = new Object();

// 同一线程重入 → 偏向锁（零开销重入）
synchronized (obj) { /* 偏向锁：只需比对线程 ID */ }
synchronized (obj) { /* 偏向锁：仍只需比对线程 ID */ }

// 不同线程竞争 → 撤销偏向锁 → 升级为轻量级锁
new Thread(() -> synchronized (obj) { /* 轻量级锁：CAS + Lock Record */ }).start();

// JDK 15+: 去掉偏向锁阶段，首次加锁直接进入轻量级锁
```
