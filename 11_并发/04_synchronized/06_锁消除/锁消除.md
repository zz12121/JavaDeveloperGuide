# 锁消除（JIT逃逸分析，无逃逸对象的锁被消除）

## 先说结论

锁消除（Lock Elimination）是 JIT 编译器的优化手段，基于**逃逸分析**（Escape Analysis）：如果 JVM 判断一个对象不会逃逸出当前线程（即不会被其他线程访问），则可以安全地消除该对象上的 synchronized 同步操作，避免不必要的锁开销。

## 深度解析

### 逃逸分析的三种情况

1. **不逃逸**：对象仅在当前方法内使用 → 可做锁消除、标量替换
2. **方法逃逸**：对象被作为参数传给其他方法 → 不可消除锁
3. **线程逃逸**：对象被其他线程访问 → 不可消除锁

### 锁消除的条件

- 对象不逃逸出当前线程
- synchronized 作用在该对象上
- JIT 编译时（C2 编译器）开启了逃逸分析

### 经典案例：StringBuffer

```java
public String concat(String a, String b) {
    // sb 不会逃逸出 concat 方法，JIT 可以消除锁
    StringBuffer sb = new StringBuffer();
    sb.append(a);
    sb.append(b);
    return sb.toString();
}
// StringBuffer 的 append() 方法有 synchronized
// 但 JIT 逃逸分析后发现 sb 不会逃逸，消除锁
```

## 易错点/踩坑

- ❌ 以为 JIT 一定能做逃逸分析
- ✅ 逃逸分析需要 C2 编译器参与，且消耗资源，不一定每个对象都分析
- ❌ 以为 StringBuilder 和 StringBuffer 性能差很多
- ✅ 锁消除后 StringBuffer 和 StringBuilder 性能几乎相同
- ❌ 以为所有 synchronized 都会被优化掉
- ✅ 只有证明对象不逃逸时才会消除

## 代码示例

```java
public class LockEliminationDemo {
    // -XX:+EliminateLocks 启用锁消除（默认开启）
    // -XX:-EliminateLocks 禁用锁消除

    public static void main(String[] args) {
        long start = System.currentTimeMillis();
        for (int i = 0; i < 10000000; i++) {
            // sb 不逃逸，JIT 消除 StringBuffer 的 synchronized
            StringBuffer sb = new StringBuffer();
            sb.append("hello");
            sb.append("world");
        }
        long end = System.currentTimeMillis();
        System.out.println("耗时: " + (end - start) + "ms");
    }
}
```
