# 锁粗化（JIT将多个相邻加锁合并成一个）

## 先说结论

锁粗化（Lock Coarsening）是 JIT 编译器的优化手段：如果 JIT 检测到一系列连续的对**同一对象**的加锁/解锁操作，且中间没有共享变量的读写，会将多次加锁合并为一次，减少锁的获取和释放次数。

## 深度解析

### 锁粗化的条件

1. 连续多次对**同一个对象**加锁/解锁
2. 锁之间没有其他线程可见的操作
3. JIT 编译器在运行时分析热点代码后执行优化

### 典型场景

```java
// 优化前：每次循环都加锁/解锁
for (int i = 0; i < 1000; i++) {
    synchronized (lock) {
        list.add(i);
    }
}

// JIT 锁粗化后：等效于
synchronized (lock) {
    for (int i = 0; i < 1000; i++) {
        list.add(i);
    }
}
```

### 锁粗化 vs 锁细化

- **锁粗化**：减少加锁次数，适合连续短临界区
- **锁细化**：减小锁粒度，适合长临界区中只有一小部分需要同步

两者看似矛盾，实际各有适用场景。JIT 会根据代码特征选择合适的策略。

## 易错点/踩坑

- ❌ 以为锁粗化一定提升性能
- ✅ 锁粒度过细导致频繁加锁解锁才有必要粗化
- ❌ 以为手动写循环外的 synchronized 和 JIT 锁粗化一样
- ✅ JIT 锁粗化更智能，会考虑中间是否有线程可见操作
- ❌ 忽视锁粗化可能导致锁持有时间过长
- ✅ 锁粗化应该恰到好处，不是越多越好

## 代码示例

```java
public class LockCoarseningDemo {
    private final Object lock = new Object();
    private int count = 0;

    // JIT 会将三次加锁合并为一次
    public void method() {
        synchronized (lock) { count++; }
        // 中间没有共享操作
        synchronized (lock) { count++; }
        // 中间没有共享操作
        synchronized (lock) { count++; }
    }

    // 手动锁粗化（推荐）
    public void methodOptimized() {
        synchronized (lock) {
            count++;
            count++;
            count++;
        }
    }
}
```
