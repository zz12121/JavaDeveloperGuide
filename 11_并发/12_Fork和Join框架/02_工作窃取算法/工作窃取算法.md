
# 工作窃取算法

## 核心结论

工作窃取（Work Stealing）是 ForkJoinPool 的核心调度算法。每个线程维护独立的双端队列（Deque），从自己的队列**头部**（LIFO）取任务执行，空闲时从其他线程队列的**尾部（FIFO）**"偷"任务。这种设计最大化 CPU 利用率，同时最小化竞争。

## 深度解析

### 核心机制

```
线程A的队列          线程B的队列          线程C的队列
[head] ← 取任务     [head] ← 取任务     [head] ← 空了！
  T3                   T6                   ↑
  T2                   T5              窃取方向
  T1                   T4              从尾部偷 ↓
                     [tail] ← T4 被 C 偷走
```

### 为什么用双端队列？

| 操作 | 端 | 策略 | 原因 |
|------|-----|------|------|
| 本线程取任务 | 头部 | LIFO（栈） | 子任务刚 fork，缓存热，LIFO 命中率高 |
| 其他线程窃取 | 尾部 | FIFO（队列） | 尾部是较早提交的大任务，适合窃取后拆分 |

**头尾分离天然减少竞争**：本线程操作头部，窃取线程操作尾部，CAS 竞争概率极低。

### 窃取过程详解

1. 线程 C 的队列为空
2. 线程 C 随机选择一个目标线程（如 B）
3. 通过 CAS 从线程 B 的队列尾部取走一个任务
4. 如果线程 B 也为空，继续尝试其他线程
5. 如果所有队列都空，线程让步（yield）或阻塞等待

### 性能优势

- **负载均衡**：自动将任务从繁忙线程分配到空闲线程
- **缓存友好**：LIFO 取任务，子任务数据大概率在 CPU 缓存中
- **低竞争**：头尾操作分离，CAS 冲突少
- **可扩展性好**：线程数增加时窃取效率不受影响

### 注意事项

- 窃取本身有开销（CAS 操作 + 线程切换），**任务粒度不能太小**
- 如果任务拆分过细，窃取开销反而超过并行收益
- 一般设置合理的阈值（如数组长度 < 10000 时不再拆分）
