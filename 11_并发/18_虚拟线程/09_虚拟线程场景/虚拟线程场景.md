
# 虚拟线程适合场景

## 核心结论

虚拟线程最适合**IO 密集型、高并发、阻塞多**的场景。核心判断标准：**任务的瓶颈是否是等待（IO/网络/数据库），而不是 CPU 计算**。虚拟线程通过"阻塞即挂起"机制，让载体线程在等待期间服务其他任务，极大提升吞吐量。

## 深度解析

### 适合场景总览

```
                    虚拟线程适用性判断
                         │
              ┌──────────┼──────────┐
              │                     │
         IO 密集型              CPU 密集型
         大量阻塞等待           大量计算
              │                     │
        ✅ 非常适合             ❌ 不适合
```

### 场景一：HTTP 服务端（最典型）

```java
// 传统方式：200 平台线程 → 最多 200 并发请求
// 虚拟线程：百万虚拟线程 → 轻松处理百万并发请求

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    var server = HttpServer.create(8080, 0);
    server.setExecutor(executor);  // 每个请求一个虚拟线程
    server.createContext("/api", exchange -> {
        // 处理请求：查数据库 → 调外部API → 返回响应
        var user = userService.findById(id);    // 阻塞IO → 挂起
        var data = remoteService.fetch(user);   // 网络IO → 挂起
        sendResponse(exchange, data);           // 无需等待，恢复执行
    });
    server.start();
}
```

**优势**：每个请求独立线程，代码写起来像同步，运行效果像异步。

### 场景二：数据库连接与 CRUD

```java
// 传统方式：数据库连接池 + 平台线程
// 问题：连接池大小 ≤ 平台线程数，高并发时大量请求排队等连接

// 虚拟线程方式：大量虚拟线程 + 连接池
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<Result>> futures = new ArrayList<>();
    for (int i = 0; i < 10_000; i++) {
        futures.add(executor.submit(() -> {
            try (var conn = dataSource.getConnection()) { // 阻塞等连接 → 挂起
                return conn.prepareStatement(sql).executeQuery();
            }
        }));
    }
}
```

> ⚠️ 注意：虚拟线程不减少数据库连接需求，连接池仍然是瓶颈。

### 场景三：调用外部 API

```java
// 批量调用多个外部接口，每个都是独立的网络IO
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<CompletableFuture<String>> futures = ids.stream()
        .map(id -> CompletableFuture.supplyAsync(
            () -> httpClient.send(request(id), BodyHandlers.ofString()).body(),
            executor
        ))
        .toList();

    CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
}
```

**优势**：10 万个外部调用同时发出，载体线程只有 CPU 核数个，但虚拟线程在等待响应时自动让出。

### 场景四：文件处理

```java
// 批量读取/处理大量文件
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Files.walk(dir).parallel()
        .forEach(file -> executor.submit(() -> {
            String content = Files.readString(file);  // 阻塞IO → 挂起
            processContent(content);
        }));
}
```

### 场景五：任务编排与 Pipeline

```java
// 串行编排多个IO步骤，每个步骤都会阻塞
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> {
        var order = getOrderFromDB();          // DB IO
        var payment = verifyPayment(order);    // HTTP IO
        var inventory = checkInventory(order); // HTTP IO
        var shipping = arrangeShipping(order); // HTTP IO
        return new Result(order, payment, inventory, shipping);
    });
}
```

**代码看起来像同步，但每个 IO 操作都会自动挂起，不占用载体线程。**

### 场景六：消息队列消费（新增重点场景）

消息队列消费是虚拟线程的经典高并发场景：每个消息处理逻辑通常是 IO 密集型（查库、调接口），非常适合虚拟线程的"阻塞即挂起"机制。

```java
// Kafka 消费者示例：每个消息独立虚拟线程
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    var consumer = new KafkaConsumer<String, String>(props);
    consumer.subscribe(List.of("order-topic"));

    while (running) {
        var records = consumer.poll(Duration.ofMillis(100));
        for (var record : records) {
            executor.submit(() -> {
                Order order = parseOrder(record);              // CPU 计算
                processOrder(order);                            // DB 读写 → 挂起
                sendNotification(order);                       // HTTP 调用 → 挂起
                consumer.commitSync();                         // 提交位移 → 挂起
            });
        }
    }
}
```

```java
// RabbitMQ 监听器示例
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    var channel = connection.createChannel();
    channel.basicQos(10);  // 预取10条，控制背压

    DeliverCallback deliver = (tag, msg) -> {
        executor.submit(() -> {
            try {
                Order order = deserialize(msg.getBody());
                processOrder(order);    // IO 密集 → 挂起
                channel.basicAck(msg.getEnvelope().getDeliveryTag(), false);
            } catch (Exception e) {
                channel.basicNack(msg.getEnvelope().getDeliveryTag(), false, true);
            }
        });
    };

    channel.basicConsume("order-queue", false, deliver, tag -> {});
}
```

**为什么消息队列场景特别适合虚拟线程**：

| 特点 | 虚拟线程优势 |
|------|-------------|
| 消息数量 >> 消费者线程数 | 百万消息轻松并发处理，载体线程数始终 = CPU 核数 |
| 每条消息处理涉及 DB/远程调用 | IO 阻塞时自动挂起，不浪费载体线程 |
| 消息处理逻辑简单直白 | 用同步代码风格，无需复杂的异步框架 |
| 背压控制 | 配合有界队列或预取数（prefetch）实现 |

> ⚠️ 注意：消息队列的连接池（Kafka Producer、RabbitMQ Connection）仍然是资源瓶颈，虚拟线程不会减少连接数需求。每个消息处理中的 DB 连接也由连接池管理，不因虚拟线程数量多而减少。

### 适合场景的关键特征

| 特征 | 说明 |
|------|------|
| 任务数量远大于 CPU 核数 | 虚拟线程优势明显 |
| 每个任务有大量等待时间 | 等待期间释放载体线程 |
| IO 操作是主要瓶颈 | 网络、数据库、文件等 |
| 代码逻辑简单直接 | 不需要复杂的异步回调 |
| 需要高吞吐量 | 并发量越大，优势越大 |

# 虚拟线程不适合场景

## 核心结论

虚拟线程不适合**CPU 密集型任务**和**长时间持有锁的场景**。虚拟线程的优势在于 IO 等待时自动挂起、释放载体线程；如果任务一直在做 CPU 计算，虚拟线程不仅没有优势，反而因为频繁的上下文切换带来额外开销。载体线程数量 = CPU 核数，CPU 密集任务正好与之匹配，虚拟线程的"挂起-恢复"毫无用武之地。

## 深度解析

### 不适合场景总览

```
              虚拟线程适用性判断
                    │
         ┌──────────┼──────────┐
         │                     │
    适合（IO 密集）         不适合（CPU 密集）
         │                     │
    ✅ 阻塞多、等待多        ❌ 计算多、等待少
    ✅ 任务数 >> CPU核数     ❌ 任务数 ≈ CPU核数
```

### 场景一：CPU 密集型计算

```java
// ❌ 不适合用虚拟线程
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            // 纯 CPU 计算，没有阻塞操作
            // 虚拟线程永远不会挂起，永远占着载体线程
            BigInteger result = BigInteger.ONE;
            for (int j = 0; j < 100000; j++) {
                result = result.multiply(BigInteger.valueOf(j));
            }
            return result;
        });
    }
}
```

**问题**：
- 100 万个虚拟线程，但载体线程只有 8 个（CPU 核数）
- CPU 密集任务从不挂起 → 载体线程永远被占用
- 剩余 999,992 个虚拟线程排队等待，和只用 8 个平台线程效果一样
- 虚拟线程的调度额外增加开销

**正确做法**：使用 `ForkJoinPool` 或固定大小的 `ThreadPoolExecutor`。

### 场景二：长时间持有锁

```java
// ❌ 不适合用虚拟线程
ReentrantLock lock = new ReentrantLock();

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            lock.lock();
            try {
                // 长时间持有锁，期间做大量计算
                heavyComputation();
            } finally {
                lock.unlock();
            }
        });
    }
}
```

**问题**：锁竞争导致大量虚拟线程等待，虽然虚拟线程在等待时能挂起，但锁本身成为瓶颈，并发度被锁限制，虚拟线程的并发优势无法发挥。

### 场景三：缓存密集型操作

```java
// ❌ 命中缓存后全是内存计算
executor.submit(() -> {
    var cached = cache.get(key);  // 缓存命中，无 IO
    var processed = transform(cached);  // 纯 CPU
    var filtered = filter(processed);   // 纯 CPU
    return filtered;
});
```

大部分时间在内存中操作，没有阻塞等待，虚拟线程没有优势。

### 场景四：Native 方法中的长时间阻塞

```java
// ⚠️ 虚拟线程中调用 Native 阻塞方法可能导致 Pinning
executor.submit(() -> {
    // JNI 调用中阻塞 → 可能钉住载体线程
    nativeBlockingOperation();
});
```

Native 方法中的阻塞不会被 JVM 拦截，虚拟线程无法卸载，载体线程被钉住。

### 场景五：对延迟敏感的实时任务

```java
// ❌ 虚拟线程调度有额外开销
// 适合追求吞吐量（throughput），不适合追求最低延迟（latency）
executor.submit(() -> {
    // 需要 <1ms 响应的实时计算
    realTimeProcess();
});
```

虚拟线程的 Mount/Unmount 有额外开销（微秒级），对极端低延迟场景不友好。

### 对比总结

| 维度 | 虚拟线程 | 平台线程（线程池） |
|------|---------|------------------|
| IO 密集型 | ✅ 极佳 | ⚠️ 受线程数限制 |
| CPU 密集型 | ❌ 无优势甚至更差 | ✅ 合适 |
| 混合型（IO为主） | ✅ 适合 | ⚠️ 需要合理配置线程池 |
| 混合型（CPU为主） | ⚠️ 需要拆分 | ✅ 合适 |
| 实时任务 | ⚠️ 有调度延迟 | ✅ 确定性更好 |
| Native 阻塞 | ⚠️ 可能 Pinning | ✅ 无此问题 |

### 混合场景的处理策略

当系统同时有 IO 密集和 CPU 密集任务时，**分开处理**：

```java
// IO 密集任务 → 虚拟线程
var ioExecutor = Executors.newVirtualThreadPerTaskExecutor();

// CPU 密集任务 → 固定大小线程池
var cpuExecutor = Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors());

// 根据任务类型选择执行器
if (task.isIoBound()) {
    ioExecutor.submit(task);
} else {
    cpuExecutor.submit(task);
}
```

### 不适合场景：需要严格顺序处理（新增）

当业务要求消息或请求**必须按顺序处理**时，虚拟线程的并行处理优势反而成为障碍：

```java
// ❌ 场景：订单状态机流转，必须严格按时间顺序
executor.submit(() -> processStep1(order));  // 虚拟线程 A
executor.submit(() -> processStep2(order));  // 虚拟线程 B（可能比 A 先完成）
// 结果：Step2 先于 Step1 执行，业务逻辑错误
```

**解决思路**：
1. **单线程串行处理**：使用单个平台线程或单线程队列
2. **分片隔离**：不同分片的顺序独立，互不影响（如按用户 ID hash 分片）
3. **版本号乐观锁**：通过版本号确保操作顺序有效性
