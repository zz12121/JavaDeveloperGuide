
# 虚拟线程原理

## 核心结论

虚拟线程基于 **M:N 调度模型**，由 JVM 的 `ForkJoinPool` 作为载体线程池，使用 **Continuation（续体）** 实现挂起/恢复。阻塞操作时，虚拟线程的状态保存到 Continuation 栈帧中，释放载体线程；恢复时重新挂载到载体线程继续执行。

## 深度解析

### M:N 调度模型

```
虚拟线程 (M个)                    载体线程 (N个, ForkJoinPool)
─────────────                    ─────────────────────
VT-1 ─┐
VT-2 ─┤
VT-3 ─┼─→  Carrier-1 (Platform Thread)
VT-4 ─┤
VT-5 ─┘
VT-6 ─┐
VT-7 ─┼─→  Carrier-2 (Platform Thread)
VT-8 ─┘
...
M >> N（M=百万, N=CPU核数）
```

### Continuation（续体）

```java
// Continuation 是虚拟线程的核心
// 本质：可暂停和恢复的栈帧

Continuation cont = new Continuation(SCOPE, () -> {
    step1();        // 执行
    Continuation.yield();  // ← 挂起点，保存当前状态
    step2();        // 恢复后从这里继续
    step3();
});

cont.run();   // 执行到 yield 挂起
cont.run();   // 从 yield 处恢复
```

### 挂起与恢复流程

```
1. 虚拟线程执行到阻塞操作（如 Socket.read）
2. JVM 拦截阻塞操作
3. 将虚拟线程的栈状态保存到 Continuation 中
4. 虚拟线程与载体线程解绑
5. 载体线程可以执行其他虚拟线程
6. IO 操作完成后，虚拟线程被调度
7. 恢复 Continuation，挂载到可用载体线程继续执行
```

### 载体线程池

```java
// 载体线程池是 ForkJoinPool，默认大小 = CPU 核数
// 可通过系统属性调整
-Djdk.virtualThreadScheduler.parallelism=16
-Djdk.virtualThreadScheduler.maxPoolSize=256
```

### Pinning 问题

```java
// JDK 21 之前：synchronized 块内阻塞会"钉住"载体线程
synchronized (lock) {
    socket.read(); // ❌ 载体线程被阻塞，无法服务其他虚拟线程
}

// 解决：用 ReentrantLock 替代 synchronized
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    socket.read(); // ✅ 虚拟线程挂起，载体线程释放
} finally {
    lock.unlock();
}

// JDK 21+：大部分 synchronized 场景已优化，但仍可能存在 pinning
```

### 底层数据结构

```
VirtualThread
├── continuation      // Continuation 对象（保存执行状态）
├── carrier           // 当前挂载的载体线程
├── runnable          // 要执行的任务
└── state             // NEW / RUNNING / WAITING / TERMINATED
```
