
# 虚拟线程优势

## 核心结论

虚拟线程的三大核心优势：**百万级并发**、**极低内存占用**、**透明挂起机制**。开发者可以用简单的同步代码风格获得接近异步框架的高并发性能。

## 深度解析

### 优势一：百万级并发

```java
// 百万虚拟线程同时启动
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 1_000_000).forEach(i -> {
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1));
            return i;
        });
    });
}
// 平台线程：直接 OOM 或系统崩溃
// 虚拟线程：轻松处理，只占用少量载体线程
```

### 优势二：极低内存占用

| 线程类型 | 1万线程 | 100万线程 |
|---------|--------|----------|
| 平台线程（1MB/线程） | ~10 GB | 不可能 |
| 虚拟线程（~1KB/线程） | ~10 MB | ~1 GB |

虚拟线程的栈在堆上分配，按需增长，不活跃时可以 GC 回收。

### 优势三：透明挂起

```java
// 开发者写同步代码，JVM 自动处理异步
Thread.startVirtualThread(() -> {
    var response1 = httpClient.send(request1);  // IO 阻塞 → 自动挂起
    var response2 = httpClient.send(request2);  // IO 阻塞 → 自动挂起
    var result = db.query(sql);                 // DB 阻塞 → 自动挂起
    // 整个过程不占用载体线程的等待时间
});
```

vs 异步编程（回调地狱）：

```java
// 传统异步：回调嵌套，代码可读性差
httpClient.send(request1, response1 -> {
    httpClient.send(request2, response2 -> {
        db.query(sql, result -> {
            // 三层嵌套...
        });
    });
});
```

### 优势四：无需线程池管理

```java
// 传统：需要精心配置线程池参数
new ThreadPoolExecutor(
    corePoolSize, maxPoolSize, keepAliveTime,
    workQueue, threadFactory, handler
);

// 虚拟线程：每个任务一个虚拟线程，无需池化
Executors.newVirtualThreadPerTaskExecutor();
```

### 优势五：ReentrantLock 与 `withLock()` 自动挂起

JDK 21 为虚拟线程增强了 `ReentrantLock`——当虚拟线程等待锁时**自动挂起**，不占用载体线程：

```java
// JDK 21+ 的 withLock() 扩展：自动 park/unpark
ReentrantLock lock = new ReentrantLock();

lock.lock();  // 获取不到时：虚拟线程自动挂起（park），载体线程释放
try {
    // 临界区代码
} finally {
    lock.unlock();  // 释放时自动 unpark 一个等待的虚拟线程
}

// withLock() 更简洁：try-with-resources 风格
lock.withLock(() -> {
    // 虚拟线程等待锁时自动挂起，锁可用时自动恢复
    process();
});
```

**为什么 withLock 对虚拟线程特别重要**：

| 对比 | `synchronized` | `ReentrantLock.withLock()` |
|------|---------------|--------------------------|
| 平台线程 | 阻塞 OS 线程 | 阻塞 OS 线程 |
| 虚拟线程 | ⚠️ 可能 pinning（JDK 21+ 已优化大部分） | ✅ 完美挂起（推荐） |
| API 风格 | 简单 | try-with-resources，更安全 |

> 💡 **推荐**：在虚拟线程中使用 `ReentrantLock` + `withLock()` 替代 `synchronized`，获得最佳挂起效果。

### 与传统方案性能对比

```
10万并发 HTTP 请求（IO 密集）：
平台线程池（200线程）：吞吐量 ~2000 req/s，大量请求排队
虚拟线程（无限制）：  吞吐量 ~50000 req/s，几乎无排队
响应式框架（WebFlux）：吞吐量 ~40000 req/s，代码复杂度高
```
