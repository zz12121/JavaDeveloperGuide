
# 虚拟线程限制

## 核心结论

虚拟线程虽然强大，但存在三个主要限制：**Pinning（钉住）**、**ThreadLocal 内存膨胀**、**载体线程池无法自定义**。其中 Pinning 是最需要关注的问题，可能导致载体线程被阻塞，无法服务其他虚拟线程。

## 深度解析

### 限制一：Pinning（钉住）

虚拟线程在 `synchronized` 块或方法内执行阻塞操作时，**无法卸载**（unmount），载体线程被"钉住"：

```
虚拟线程执行流程（Pinning 场景）：

正常情况：
  VT → mount → Carrier-1 → 阻塞IO → unmount → Carrier-1 释放 ✅

Pinning 情况：
  VT → mount → Carrier-1 → 进入 synchronized → 阻塞IO → ❌ 无法 unmount
                                                    → Carrier-1 被钉住，其他 VT 等待
```

**Pinning 的原因**：
- JVM 需要确保 `synchronized` 的原子性和可见性
- `synchronized` 使用的 monitor 机制与载体线程绑定
- 虚拟线程卸载后无法保持 monitor 的所有权

**解决方案**：

| 方案 | 操作 | 说明 |
|------|------|------|
| 替换锁 | `synchronized` → `ReentrantLock` | **推荐方案** |
| 升级 JDK | JDK 21+ 已部分优化 | 大部分 `synchronized` 场景已解决 |
| 检测工具 | `-Djdk.tracePinnedThreads=short` | 运行时检测 pinning |

```java
// ❌ 可能导致 pinning
synchronized (lock) {
    socket.read(buffer);  // 载体线程被钉住
}

// ✅ 使用 ReentrantLock 解决
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    socket.read(buffer);  // 虚拟线程正常挂起
} finally {
    lock.unlock();
}
```

> **JDK 21+ 的优化**：JEP 491 已经优化了大部分 `synchronized` 的 pinning 问题，只有在 `synchronized` 块内持有 monitor 且调用本地方法（Native Method）阻塞时才会发生。

### 限制二：ThreadLocal 内存膨胀

虚拟线程数量可达百万级，每个虚拟线程都有自己的 `ThreadLocal` 副本：

```
平台线程场景（200 线程）：
  ThreadLocal 大小 × 200 = 200 份副本 ✅ 可接受

虚拟线程场景（1,000,000 线程）：
  ThreadLocal 大小 × 1,000,000 = 100万份副本 ❌ OOM 风险
```

**解决方案**：使用 `Scoped Values`（JDK 21 Preview，JDK 22+ 稳定）替代 ThreadLocal。

### 限制三：载体线程池不可自定义

虚拟线程默认运行在 `ForkJoinPool` 上（即载体线程池），该池的大小等于可用处理器数：

- **无法设置**载体线程池的线程数
- **无法替换**为自定义的 `ExecutorService`
- 载体线程始终是平台线程（Daemon 线程）

### 限制四：不支持 stop/suspend

```java
Thread vt = Thread.startVirtualThread(task);
vt.stop();    // ❌ 抛出 UnsupportedOperationException
vt.suspend(); // ❌ 抛出 UnsupportedOperationException
```

虚拟线程协作式调度，不支持传统的线程控制方法。

### 限制四：不支持 stop/suspend

```java
Thread vt = Thread.startVirtualThread(task);
vt.stop();    // ❌ 抛出 UnsupportedOperationException
vt.suspend(); // ❌ 抛出 UnsupportedOperationException
```

虚拟线程是协作式调度，不支持传统的线程控制方法。

### 限制五：interrupt() 的限制（重要）

`Thread.interrupt()` 在虚拟线程中的行为**发生了变化**：

**平台线程中**：`interrupt()` 会立即唤醒 `sleep()`/`wait()`/`park()` 中的线程，抛出 `InterruptedException`。

**虚拟线程中**：`interrupt()` **只设置中断标志位**，不直接唤醒被 park 的虚拟线程（JVM 层面处理）：

```java
// 平台线程：interrupt() 立即唤醒 park()
Thread t = new Thread(() -> {
    LockSupport.park();  // 被 interrupt() 唤醒
    System.out.println("awaken");
});
t.start();
t.interrupt();  // 立即唤醒

// 虚拟线程：interrupt() 设置标志位，但不一定立即唤醒
Thread vt = Thread.startVirtualThread(() -> {
    LockSupport.park();
    // 不会被立即唤醒，需要 JVM 内部逻辑处理
});
vt.interrupt();  // 设置中断标志，虚拟线程可能还在 parking 状态
```

> 💡 **最佳实践**：不要在虚拟线程中使用 `interrupt()` 来取消任务，应该使用 **StructuredTaskScope** 的作用域取消机制。

### 限制六：Thread.sleep() vs LockSupport.park() 对比

在虚拟线程中，两者的行为有细微差别：

| 对比 | `Thread.sleep()` | `LockSupport.park()` |
|------|-------------------|----------------------|
| 挂起机制 | JVM 原生支持，100% 挂起 | JVM 原生支持，100% 挂起 |
| 中断响应 | 抛出 `InterruptedException` | 设置中断标志，不抛异常 |
| 虚拟线程中的行为 | 完全一致 | 完全一致 |
| 精确度 | 受 JVM 调度影响（微秒级误差） | 更高精度（需要手动循环检查） |
| 使用场景 | 延迟等待（如 `Thread.sleep(1000)`) | 锁机制底层（如 `ReentrantLock`） |

**为什么虚拟线程中推荐使用 `Thread.sleep()` 而非 `LockSupport.park()`？**

```java
// ✅ 推荐：Thread.sleep()
Thread.startVirtualThread(() -> {
    Thread.sleep(1000);  // 精确、清晰、可中断
});

// ⚠️ 不推荐：LockSupport.park()（除非在做锁机制）
Thread.startVirtualThread(() -> {
    LockSupport.park();  // 需要配合 unpark()，不直观
});
```

### 其他限制

| 限制 | 说明 |
|------|------|
| 线程组 | 虚拟线程不属于任何线程组（`getThreadGroup()` 返回虚拟线程组） |
| 优先级 | `setPriority()` 不生效，虚拟线程无优先级概念 |
| 守护状态 | 虚拟线程始终是守护线程，`setDaemon(false)` 无效 |
| 栈大小 | 无法自定义虚拟线程栈大小 |
