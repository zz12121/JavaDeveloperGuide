
# 虚拟线程 vs 平台线程

## Q1：虚拟线程和平台线程有什么区别？

**A**：

| 维度 | 平台线程 | 虚拟线程 |
|------|---------|---------|
| 调度 | OS 内核调度 | JVM 用户态调度 |
| 栈大小 | 默认 1MB（可调 `-Xss`） | **最大 1MB**，初始 ~512B（按需增长） |
| 数量 | 几千（受 OS 限制） | 百万级 |
| 创建成本 | 高（系统调用） | 极低（JVM 内存分配） |
| 阻塞 | 阻塞 OS 线程 | 挂起，释放载体线程 |
| 上下文切换 | 内核态（慢，约 μs 级） | 用户态（快，约 ns 级） |

一句话：虚拟线程更轻、更多、更快，由 JVM 管理而非 OS。

---

## Q2：虚拟线程能完全替代平台线程吗？

**A**：

不能。虚拟线程适用于 IO 密集型场景，CPU 密集型任务仍然应该用平台线程。

原因：虚拟线程数量远超 CPU 核心数，CPU 密集任务会导致大量虚拟线程争抢 CPU，频繁上下文切换反而降低性能。平台线程数量 = CPU 核心数，不会过度竞争。

---

## Q3：迁移到虚拟线程需要改代码吗？

**A**：

API 层面几乎无需改动，只需修改线程创建方式：

```java
// 旧：new Thread(task).start()
Thread.startVirtualThread(task);

// 旧：Executors.newFixedThreadPool(10)
Executors.newVirtualThreadPerTaskExecutor();
```

`Runnable`、`Callable`、`synchronized`、`ReentrantLock` 等都正常工作。但需要注意 `ThreadLocal` 在百万虚拟线程下的内存问题和 `synchronized` 的 pinning 问题。

## Q4：M:N 调度模型是什么意思？载体线程数是怎么确定的？

**A**：

**M:N 模型**：M 个虚拟线程映射到 N 个载体线程（Carrier Thread）。N 远小于 M。

```
M:N 调度示意（8核 CPU）：
  VT-1    ─┐
  VT-2    │
  VT-3    ├──→ Carrier-1 ─→ CPU-1
  ...     │
  VT-8    ─┘
  VT-9    ─┐
  VT-10   ├──→ Carrier-2 ─→ CPU-2
  ...     │
  VT-16   ─┘
  ...
  VT-1000000（全部排队等待 mount）
```

**载体线程数 = `Runtime.getRuntime().availableProcessors()`**

即：CPU 核心数。例如 8 核机器 → 8 个载体线程。

**为什么这样设计？**

- CPU 密集任务：8 个载体线程正好 100% 占用 8 核 CPU，没有浪费
- IO 密集任务：100 万个虚拟线程轮流用 8 个载体线程，IO 等待时载体线程去服务其他虚拟线程

**可以通过 JVM 参数调整**：
```bash
# 默认：载体线程数 = CPU 核数
# 注意：JDK 官方不建议修改此值
-Djdk.virtualThreadScheduler.parallelism=8
```

## Q5：虚拟线程的栈大小可以手动设置吗？最大能到多少？

**A**：

**可以设置最大栈大小，但不能设置初始栈大小**。

```bash
# 设置虚拟线程最大栈大小（默认 1MB）
-Djdk.virtualThreadStackSize=2m
```

**关键特性：按需增长**

| 线程类型 | 初始大小 | 增长方式 | 最大大小 |
|---------|---------|---------|---------|
| 平台线程 | 固定（如 1MB） | 预分配，固定不变 | 由 `-Xss` 决定 |
| 虚拟线程 | **~512 字节** | 按需增长（按页分配，4KB/页） | 由 `-Djdk.virtualThreadStackSize` 决定（默认 1MB） |

**为什么虚拟线程初始只有 ~512 字节？**

因为虚拟线程的栈在堆上分配，不占用 OS 内存，可以"先用再扩展"。大部分虚拟线程只需要几十到几百字节的栈空间就够用了，只有深层递归才需要接近最大栈大小。

```java
// 实际内存占用示例
Thread.startVirtualThread(() -> {
    // 初始栈：~512 字节
    callMethod1(); // +4KB
    callMethod2(); // +4KB
    callMethod3(); // +4KB
    // 当前栈：~12KB
    // 最多增长到 1MB
});
// 方法返回后，未使用的栈空间可以被 GC 回收
```

**内存计算示例**：
```
100 万虚拟线程，每个平均使用 10KB 栈空间
  = 100万 × 10KB = ~10GB 堆内存（虚拟线程栈）
  vs 平台线程：100万 × 1MB = 1TB（不可能）

实际：大部分虚拟线程在 IO 阻塞时栈空间不活跃，可被 GC 压缩
```
