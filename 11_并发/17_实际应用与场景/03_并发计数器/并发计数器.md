
# 并发计数器

## 核心结论

并发计数器在多线程下保证计数操作的原子性和一致性。根据并发量选择：低并发用 `AtomicLong`，高并发用 `LongAdder`，需要精确快照用 `synchronized` 或分段锁。

## 深度解析

### 方案对比

| 方案 | 原理 | 并发性能 | 精确计数 | 适用场景 |
|------|------|---------|---------|---------|
| `synchronized` | 互斥锁 | 低 | ✅ 精确 | 并发量小，需要精确值 |
| `AtomicLong` | CAS 自旋 | 中（高并发退化为自旋） | ✅ 精确 | 中等并发 |
| `LongAdder` | 分段 CAS + Cells 数组 | 高 | ⚠️ 近似 | 高并发统计 |
| `LongAccumulator` | 分段 + 自定义函数 | 高 | ⚠️ 近似 | 高并发自定义运算 |

### AtomicLong（中等并发）

```java
AtomicLong counter = new AtomicLong();

// 线程安全计数
counter.incrementAndGet();    // +1
counter.addAndGet(10);        // +10
counter.getAndIncrement();    // +1，返回旧值

// 底层 CAS
public final long incrementAndGet() {
    return U.getAndAddLong(this, VALUE, 1L) + 1L;
}
```

**问题**：高并发下多线程竞争同一个 value，CAS 失败后不断自旋，性能下降。

### LongAdder（高并发推荐）

```java
LongAdder counter = new LongAdder();
counter.increment();    // +1
counter.add(10);        // +10
counter.sum();          // 获取总和（近似值）

// 高并发下性能是 AtomicLong 的 5~10 倍
```

**原理**：分段 CAS，减少竞争
```
LongAdder
├── base（无竞争时直接 CAS）
└── Cell[] cells（有竞争时，每个线程 hash 到不同 Cell）
    ├── Cell[0] → 1000
    ├── Cell[1] → 2000
    ├── Cell[2] → 1500
    └── Cell[3] → 800
    sum() = base + 所有 Cell 值
```

### 选择指南

```
并发量 < 100 线程    → AtomicLong（精确、简单）
并发量 ≥ 100 线程    → LongAdder（高性能、统计场景）
需要实时精确计数      → synchronized 或 AtomicLong
需要自定义累加逻辑    → LongAccumulator
```

### Striped64 父类原理（LongAdder 底层）

```
LongAdder 继承 Striped64：
┌────────────────────────────────────────┐
│ Striped64                              │
│   long base           // 无竞争时使用   │
│   Cell[] cells        // 有竞争时扩展   │
│   int cellsBusy       // cells 初始化锁 │
└────────────────────────────────────────┘

Cell 扩容策略：
- 初始大小：2
- 扩容时机：CAS 失败（竞争激烈）
- 扩容上限：CPU 核数（cells.length ≤ NCPU）
- 每次扩容：×2（始终是 2 的幂次，方便 hash 取模）

线程映射 Cell 的方式：
- 每个线程有 threadLocalRandomProbe（探针哈希值）
- index = probe & (cells.length - 1)
- CAS 失败时重新计算 probe，换到另一个 Cell
```

### @Contended 注解消除 Cell 伪共享

```java
// LongAdder 的 Cell 使用 @Contended 防伪共享
@sun.misc.Contended
static final class Cell {
    volatile long value;
    // @Contended 让 JVM 在 value 前后各填充约 128 字节
    // 确保每个 Cell 独占一条 CPU 缓存行（通常 64 字节）
}
```

**伪共享问题**：不同线程的两个变量恰好在同一缓存行，一个线程修改导致另一个线程缓存失效，频繁 MESI 协议同步。

**@Contended 解决**：让 JVM 自动在对象前后填充字节，使每个 Cell 独占缓存行，彻底消除伪共享。

> 使用 @Contended 需要 JVM 参数 `-XX:-RestrictContended`（JDK 内部类默认允许）。
