
# 连接池管理

## 核心结论

连接池通过**复用连接**减少创建/销毁开销，使用 `Semaphore` 控制最大连接数，`BlockingQueue` 存储空闲连接。

## 深度解析

### 简易连接池实现

```java
public class SimpleConnectionPool {
    private final BlockingQueue<Connection> idlePool;
    private final Semaphore semaphore;
    private final int maxTotal;

    public SimpleConnectionPool(int maxTotal, String url) {
        this.maxTotal = maxTotal;
        this.semaphore = new Semaphore(maxTotal);
        this.idlePool = new LinkedBlockingQueue<>(maxTotal);
        // 预创建连接
        for (int i = 0; i < maxTotal; i++) {
            idlePool.offer(createConnection(url));
        }
    }

    public Connection borrowObject() throws Exception {
        semaphore.acquire(); // 获取许可，控制最大并发
        Connection conn = idlePool.poll(); // 空时返回 null
        if (conn == null) conn = createConnection(url);
        return conn;
    }

    public void returnObject(Connection conn) {
        if (conn != null) {
            if (idlePool.size() < maxTotal) {
                idlePool.offer(conn);
            } else {
                close(conn); // 超过最大数量则关闭
            }
        }
        semaphore.release(); // 释放许可
    }
}
```

### 连接池关键参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| maxTotal | 最大连接数 | 根据 DB 和应用规模设置 |
| maxIdle | 最大空闲连接 | 与 maxTotal 相同 |
| minIdle | 最小空闲连接 | 5~10 |
| maxWaitMillis | 获取连接最大等待时间 | 3000~5000ms |
| testOnBorrow | 借出时验证 | true（开发）/ false（生产用 validationQuery） |
| testWhileIdle | 空闲时验证 | true |

### HikariCP 核心原理

**为什么 HikariCP 是最快的连接池？**

```
1. ConcurrentBag（核心数据结构）
   - 专为连接池设计的无锁并发集合
   - ThreadLocal 优先：每个线程优先从自己的"私有"连接列表取
   - 避免线程间的连接竞争

2. FastList（替代 ArrayList）
   - 自定义 List，删除操作从尾部倒序扫描（连接通常后借先还）
   - 减少数组移动开销

3. JavassistProxyFactory（字节码代理）
   - 编译期生成 Connection 代理类，避免反射
   - 比 JDK 动态代理快约 28 倍
```

### 连接池大小调优公式

```java
// HikariCP 官方推荐（PostgreSQL 团队验证）：
// pool_size = CPU核数 × 2 + 有效磁盘数（通常 = CPU核数 × 2 + 1）
// 经验值：10~20 个连接对大多数场景足够

HikariConfig config = new HikariConfig();
config.setMaximumPoolSize(20);        // 核心参数，通常 10~20
config.setMinimumIdle(10);            // 建议等于 maximumPoolSize
config.setConnectionTimeout(3000);    // 获取连接超时（建议 3~5s，而非默认 30s）
config.setIdleTimeout(300000);        // 空闲连接存活 5 分钟
config.setMaxLifetime(1800000);       // 连接最大存活 30 分钟（需 < DB wait_timeout）
config.setKeepaliveTime(120000);      // 2 分钟 ping 保活（避免防火墙断开空闲连接）
config.setLeakDetectionThreshold(60000); // 60s 未归还告警（生产建议开启）
```

**连接池过大的危害**：过多连接导致 DB 上下文切换开销反而增大。数据库的处理是串行的，更多连接 ≠ 更快，10~20 个连接通常优于 100 个连接。
