
# 并发限流

## 核心结论

并发限流是保护系统在高并发下不被击垮的关键手段。常见方案：`Semaphore`（信号量限流）、令牌桶、滑动窗口、Redis 分布式限流。

## 深度解析

### 方案一：Semaphore（JDK 原生）

```java
// 限制最多 100 个并发请求
private final Semaphore semaphore = new Semaphore(100);

public void handleRequest() {
    if (!semaphore.tryAcquire()) {
        throw new RuntimeException("系统繁忙，请稍后重试");
    }
    try {
        // 处理请求
    } finally {
        semaphore.release();
    }
}
```

### 方案二：令牌桶（Guava RateLimiter）

```java
// 每秒生成 100 个令牌
private final RateLimiter rateLimiter = RateLimiter.create(100);

public void handleRequest() {
    if (!rateLimiter.tryAcquire(500, TimeUnit.MILLISECONDS)) {
        throw new RuntimeException("请求超限");
    }
    // 处理请求
}
```

令牌桶特性：
- 以固定速率向桶中放令牌
- 请求获取令牌，桶中有令牌则通过
- 桶满则丢弃新令牌
- 允许突发流量（桶中积累的令牌）

### 方案三：滑动窗口（Redis + Lua）

```java
// Redis 滑动窗口限流
public boolean isAllowed(String userId, int limit, int windowSeconds) {
    String key = "rate_limit:" + userId;
    long now = System.currentTimeMillis();
    long windowStart = now - windowSeconds * 1000L;

    // Lua 脚本原子操作
    pipeline.zremrangeByScore(key, 0, windowStart);
    long count = pipeline.zcard(key);
    if (count < limit) {
        pipeline.zadd(key, now, UUID.randomUUID().toString());
        pipeline.expire(key, windowSeconds);
        return true;
    }
    return false;
}
```

### 方案四：漏桶算法

```java
// 漏桶：请求以任意速率进入，以固定速率流出（强制匀速）
public class LeakyBucket {
    private final long capacity;        // 桶容量
    private final long leakRate;        // 漏出速率（每秒）
    private long water = 0;             // 当前水量
    private long lastLeakTime = System.currentTimeMillis();

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        long elapsed = now - lastLeakTime;
        // 先漏水（按时间比例减少水量）
        water = Math.max(0, water - elapsed * leakRate / 1000);
        lastLeakTime = now;

        if (water < capacity) {
            water++;      // 进水
            return true;  // 允许通过
        }
        return false;     // 桶满，拒绝
    }
}
```

**漏桶的本质**：无论请求速率多高，输出速率永远固定。适合**流量整形**场景（如短信发送、资金转账）。

### 令牌桶 vs 漏桶 深度对比

| 维度 | 令牌桶（Token Bucket） | 漏桶（Leaky Bucket） |
|------|---------------------|-------------------|
| **突发流量** | ✅ 允许（桶中积累的令牌） | ❌ 强制匀速，不允许突发 |
| **输出速率** | 最大不超过生成速率，但可突发 | 严格固定速率输出 |
| **适用场景** | API 限流、允许瞬时突发 | 流量整形、均匀消费 |
| **JDK 实现** | Guava `RateLimiter` | 无原生，需手动实现 |
| **典型场景** | 秒杀限流（允许瞬时高峰） | 短信发送（严格均匀） |

**核心区别**：
- 令牌桶限制**平均速率**，允许短时突发（如积累了10个令牌，可以瞬间处理10个请求）
- 漏桶限制**瞬时速率**，强制输出匀速（无论多少请求进来，始终每秒处理固定数量）

### 方案对比

| 方案 | 类型 | 精度 | 分布式支持 | 实现复杂度 |
|------|------|------|-----------|-----------| 
| Semaphore | 并发数限制 | 精确 | ❌ 单机 | 低 |
| RateLimiter | QPS 限制（令牌桶） | 精确 | ❌ 单机 | 低 |
| 漏桶 | QPS 限制（匀速） | 精确 | ❌ 单机 | 中 |
| 滑动窗口（Redis） | QPS 限制 | 较精确 | ✅ 分布式 | 中 |
| Sentinel（阿里） | 综合 | 精确 | ✅ 分布式 | 低（框架） |

### 限流策略

| 策略 | 说明 |
|------|------|
| **快速失败** | 超过阈值直接拒绝 |
| **排队等待** | 超出部分进入队列 |
| **预热** | 冷启动时逐步放开流量 |
