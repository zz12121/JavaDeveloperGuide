
# 并发日志

## 核心结论

高并发日志系统面临的核心挑战是**日志写入不能成为性能瓶颈**。方案：异步日志（Log4j2 AsyncLogger）、无锁队列（Disruptor）、批量写入、MDC 链路追踪。

## 深度解析

### 方案一：Log4j2 AsyncLogger（推荐）

```xml
<!-- log4j2.xml -->
<Configuration>
    <Appenders>
        <Console name="Console" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss.SSS} [%t] %-5level %logger{36} - %msg%n"/>
        </Console>
    </Appenders>
    <Loggers>
        <!-- AsyncLogger 使用 Disruptor 实现 -->
        <AsyncLogger name="com.example" level="info" additivity="false">
            <AppenderRef ref="Console"/>
        </AsyncLogger>
    </Loggers>
</Configuration>
```

Log4j2 AsyncLogger 基于 **Disruptor** 环形队列，单线程写入日志，吞吐量可达每秒千万条。

### 方案二：Disruptor 无锁队列

```java
// Disruptor 核心概念
Disruptor<LogEvent> disruptor = new Disruptor<>(
    LogEvent::new,                    // EventFactory
    1024 * 64,                        // Ring Buffer 大小（2的幂次）
    Executors.defaultThreadFactory(),  // 线程工厂
    ProducerType.MULTI,               // 多生产者
    new BlockingWaitStrategy()        // 等待策略
);

// 消费者：批量写入文件
disruptor.handleEventsWith(new LogEventHandler());
disruptor.start();

// 生产者：发布日志事件
long sequence = ringBuffer.next();
try {
    LogEvent event = ringBuffer.get(sequence);
    event.setValue(logMessage);
} finally {
    ringBuffer.publish(sequence);
}
```

### Disruptor 等待策略

| 策略 | 吞吐量 | CPU 占用 | 适用场景 |
|------|--------|---------|---------|
| Blocking | 低 | 低 | 日志非关键路径 |
| Sleeping | 中 | 中 | 平衡方案 |
| Yielding | 高 | 高 | 低延迟场景 |
| BusySpin | 最高 | 100% | 极致性能 |

### MDC 链路追踪

```java
// Filter 中设置 TraceId
public void doFilter(request, response, chain) {
    String traceId = request.getHeader("X-Trace-Id");
    MDC.put("traceId", traceId != null ? traceId : UUID.randomUUID().toString());
    try {
        chain.doFilter(request, response);
    } finally {
        MDC.clear(); // 线程池中必须清理！
    }
}

// logback.xml 中使用
<pattern>%d{HH:mm:ss} [%X{traceId}] %-5level %logger - %msg%n</pattern>
```

**注意**：MDC 基于 `ThreadLocal`，线程池中必须清理，否则 TraceId 串线程。

### AsyncAppender vs AsyncLogger 区别

| 维度 | AsyncAppender（Logback） | AsyncLogger（Log4j2） |
|------|------------------------|---------------------|
| 实现原理 | `BlockingQueue` + 独立线程 | **Disruptor** 无锁环形队列 |
| 吞吐量 | 中（受 BlockingQueue 锁竞争） | 极高（无锁，Log4j2 全异步模式） |
| 配置方式 | 包裹同步 Appender | 直接替换 Logger 类型 |
| 队列满策略 | 丢弃（可配置）| Disruptor WaitStrategy |
| 适用框架 | Logback | Log4j2 |

```xml
<!-- Logback：AsyncAppender 包裹 FileAppender -->
<appender name="ASYNC" class="ch.qos.logback.classic.AsyncAppender">
    <queueSize>512</queueSize>
    <discardingThreshold>0</discardingThreshold>  <!-- 0=不丢弃，默认20%时丢TRACE/DEBUG -->
    <includeCallerData>false</includeCallerData>   <!-- 不记录调用位置（性能优化） -->
    <appender-ref ref="FILE" />
</appender>
```

```xml
<!-- Log4j2：AsyncLogger（全局异步，比 AsyncAppender 性能高 10 倍+） -->
<Configuration>
  <Loggers>
    <AsyncRoot level="info">          <!-- 全局异步 Root Logger -->
      <AppenderRef ref="File"/>
    </AsyncRoot>
  </Loggers>
</Configuration>
```

或通过系统属性开启全局异步：`-Dlog4j2.contextSelector=org.apache.logging.log4j.core.async.AsyncLoggerContextSelector`

### 日志队列溢出策略

当生产速度 > 消费速度时，队列可能溢出：

| 框架 | 溢出时默认策略 | 配置建议 |
|------|-------------|---------|
| Logback AsyncAppender | 丢弃 TRACE/DEBUG/INFO（保留 WARN/ERROR） | `discardingThreshold=0` 关闭丢弃 |
| Log4j2 AsyncLogger | 阻塞（BlockingWaitStrategy） | 生产环境改用 `SleepingWaitStrategy` |

```java
// Log4j2 队列满时阻塞 vs 丢弃
// 阻塞模式（默认）：业务线程等待，日志不丢但影响性能
// 丢弃模式：设置 log4j2.asyncQueueFullPolicy=Discard
```

**最佳实践**：
1. 队列大小设为 2 的幂次（如 512/1024/4096）
2. 生产环境不要 `includeCallerData=true`（获取调用栈很慢）
3. 使用 `%notEmpty{[%X{traceId}]}` 避免空 TraceId 打印
