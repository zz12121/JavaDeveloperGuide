
# 死锁 vs 活锁 vs 饥饿

## 核心结论

三者都是并发问题，但表现不同：**死锁是互相等待不动，活锁是不断重试无效，饥饿是永远轮不到**。

## 深度解析

### 全面对比

| 维度 | 死锁 | 活锁 | 饥饿 |
|------|------|------|------|
| **定义** | 线程互相等待对方释放锁 | 线程不断重试但无进展 | 线程长期获取不到资源 |
| **线程状态** | BLOCKED/WAITING | RUNNABLE | RUNNABLE/BLOCKED |
| **CPU 占用** | 低（线程空闲） | 高（不停重试） | 不定 |
| **能否自行恢复** | ❌ 永远不能 | 理论上能，实际很难 | 可能，取决于调度 |
| **检测难度** | 容易（jstack） | 较难 | 中等 |
| **典型原因** | 循环等待获取多个锁 | CAS 重试/消息重发 | 优先级/非公平锁 |
| **解决方案** | 预防/避免/检测+恢复 | 随机退避/有界重试 | 公平锁/合理优先级 |

### 图解

```
死锁：
  T1: 持有锁A → 等锁B ──┐
                        ├── 永远僵住
  T2: 持有锁B → 等锁A ──┘

活锁：
  T1: 获取锁失败 → 释放 → 重试 → 获取锁失败 → 释放 → 重试 → ...
  T2: 获取锁失败 → 释放 → 重试 → 获取锁失败 → 释放 → 重试 → ...

饥饿：
  T1(高优先级): 一直在执行 → ← → ← → ← → ←
  T2(低优先级): 等待...等待...等待... → ← ←（很难轮到）
```

### 解决方案对照

| 问题 | 预防 | 检测 | 恢复 |
|------|------|------|------|
| 死锁 | 按顺序加锁/tryLock | jstack/ThreadMXBean | 重启/中断/kill |
| 活锁 | 随机退避/有界重试 | CPU 高+无进展 | 自动恢复（退避有效时） |
| 饥饿 | 公平锁/合理优先级 | 线程等待时间监控 | 调整优先级/切换公平锁 |

## 易错点与踩坑

### 1. 混淆三者的表现形式

```java
// ❌ 误以为三者都是"线程不动"
// 死锁：线程 BLOCKED，完全不动
// 活锁：线程 RUNNABLE，但做无效工作
// 饥饿：线程 RUNNABLE，但很难轮到

// ✅ 诊断方法：
// 1. jstack 查看线程状态
//    - BLOCKED → 可能是死锁
//    - RUNNABLE + CPU 高 → 活锁
//    - RUNNABLE + CPU 低 → 饥饿
// 2. jstack 输出 "deadlock" → 死锁
// 3. 日志显示线程一直在重试 → 活锁
```

### 2. 认为活锁能自动恢复

```java
// ❌ 误以为活锁不需要干预，会自动解除
// 实际情况：
// 1. 如果不退避，活锁会一直持续
// 2. 即使退避，如果两个线程节奏相同，仍可能持续
// 3. 高并发下，活锁可能演变成性能问题

// ✅ 正确认识：
// 1. 活锁是性能问题，不是功能问题（线程在工作）
// 2. 需要通过退避算法解决
// 3. 监控重试次数，设置告警阈值
```

### 3. 认为饥饿不是严重问题

```java
// ❌ 误以为饥饿只是"慢一点"
// 实际情况：
// 1. 饥饿线程的任务永远无法完成
// 2. 业务上可能导致功能不可用
// 3. 可能演变成更严重的问题（如超时、服务降级）

// ✅ 饥饿需要处理的场景：
// 1. 写线程在读锁下饥饿 → 功能不完整
// 2. 后台任务被前台任务饿死 → 日志不完整/监控缺失
// 3. 低优先级线程无法完成关键操作 → 数据不一致
```

### 4. 面试中的典型混淆

```java
// ❌ 面试题：死锁和活锁的区别？
// 错误回答："死锁是互相等待，活锁是同时做一件事"
// ✅ 正确回答：
// "死锁是线程完全阻塞，无法继续执行；活锁是线程没有阻塞，但做的是无效工作，一直重试"

// ❌ 面试题：如何解决死锁？
// 错误回答："用 tryLock 重试"
// ✅ 正确回答：
// "预防为主：1) 固定加锁顺序 2) tryLock 超时 3) 一次性获取所有锁 4) 减少持锁时间"
```

### 5. 混淆三者的解决策略

```java
// ❌ 死锁用活锁的解决方案
// 死锁不能通过退避解决，必须预防或检测恢复

// ❌ 活锁用死锁的解决方案
// 活锁不是等待锁的问题，不能通过改变加锁顺序解决

// ✅ 正确对应：
// 问题      根因            解决方案
// 死锁      循环等待         预防（固定顺序/tryLock）或检测恢复
// 活锁      无效重试         退避算法（随机退避/指数退避）
// 饥饿      资源竞争不公平    公平锁/合理优先级/设计优化
```
