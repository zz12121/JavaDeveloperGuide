
# 死锁预防

## 核心结论

死锁预防通过**破坏死锁四个必要条件中的一个或多个**来避免死锁。实际开发中最常用的三种策略：**按顺序加锁、一次性获取所有锁、tryLock 超时**。

## 深度解析

### 策略一：破坏循环等待——按固定顺序加锁

```java
// 错误：加锁顺序不一致
void transfer(Account from, Account to, int amount) {
    synchronized (from) {
        synchronized (to) { // 可能死锁
            from.debit(amount);
            to.credit(amount);
        }
    }
}

// 正确：按固定顺序加锁（如按 hashCode）
void transfer(Account from, Account to, int amount) {
    Account first = from.hashCode() < to.hashCode() ? from : to;
    Account second = from.hashCode() < to.hashCode() ? to : from;
    synchronized (first) {
        synchronized (second) {
            from.debit(amount);
            to.credit(amount);
        }
    }
}
```

### 策略二：破坏占有并等待——一次性获取所有锁

```java
// 使用固定顺序获取多个锁的辅助方法
void transferWithAllLocks(Account from, Account to, int amount) {
    synchronized (Account.LOCK) { // 先获取全局锁
        synchronized (from) {
            synchronized (to) {
                from.debit(amount);
                to.credit(amount);
            }
        }
    }
}
```

缺点：降低了并发度。

### 策略三：破坏不可抢占——tryLock + 超时

```java
void transferWithTryLock(Account from, Account to, int amount) throws InterruptedException {
    while (true) {
        if (from.lock.tryLock()) {
            try {
                if (to.lock.tryLock(100, TimeUnit.MILLISECONDS)) {
                    try {
                        from.debit(amount);
                        to.credit(amount);
                        return; // 成功
                    } finally {
                        to.lock.unlock();
                    }
                }
            } finally {
                from.lock.unlock();
            }
        }
        Thread.sleep(100); // 短暂等待后重试
    }
}
```

### 策略四：减少锁的粒度

```java
// 粗粒度：整个方法一把锁
synchronized (this) {
    readData();
    processData();
    writeResult();
}

// 细粒度：只锁必要部分
List<Data> data;
synchronized (this) {
    data = readData();
}
processData(data); // 无锁处理
synchronized (this) {
    writeResult();
}
```

### 各策略对比

| 策略 | 优点 | 缺点 |
|------|------|------|
| 按顺序加锁 | 简单有效 | 需要全局统一的排序规则 |
| 一次性获取 | 不会死锁 | 并发度低 |
| tryLock 超时 | 灵活 | 需要处理重试逻辑 |
| 减少锁粒度 | 降低死锁概率 | 不保证完全避免 |

## 易错点与踩坑

### 1. hashCode 排序在并发下不可靠

```java
// ❌ 误以为 hashCode 排序是固定顺序
// 实际上：String 的 hashCode 不是 final，可能被覆写
// 而且对象引用可能为 null

// ❌ 错误代码：Account 可能为 null
void transfer(Account from, Account to, int amount) {
    Account first = from.hashCode() < to.hashCode() ? from : to;  // NPE！
}

// ✅ 正确做法：处理 null + 使用 System.identityHashCode
void transfer(Account from, Account to, int amount) {
    if (from == null || to == null) throw new IllegalArgumentException();
    Account first = System.identityHashCode(from) < System.identityHashCode(to) ? from : to;
}

// ✅ 更可靠：使用全局排序器
private static final Comparator<Account> COMPARATOR =
    Comparator.comparing(Account::getId);
Account first = COMPARATOR.compare(from, to) < 0 ? from : to;
```

### 2. tryLock 空转导致活锁

```java
// ❌ 错误的无限重试
while (!lock.tryLock()) {
    // ❌ CPU 空转，100% 占用
}

// ✅ 正确做法：带退避的重试
int retries = 0;
while (!lock.tryLock()) {
    Thread.sleep(50 * (retries + 1));  // 递增退避
    retries++;
    if (retries > 10) throw new RuntimeException("获取锁超时");
}

// ✅ 或者使用带超时的 tryLock
lock.tryLock(5, TimeUnit.SECONDS);
```

### 3. 一次性获取锁降低并发度

```java
// ❌ 过度同步：用一个全局锁保护所有操作
synchronized (ALL_OPERATIONS_LOCK) {
    transfer(a, b, 100);
    transfer(c, d, 200);
}
// 两个转账操作完全串行，并发度为零

// ✅ 正确做法：按需加锁，粒度细化
// A→B 和 C→D 如果账户不重叠，可以并行
transfer(a, b, 100);  // 锁住 a 和 b
transfer(c, d, 200);  // 锁住 c 和 d（无冲突）
```

### 4. 忽略锁的边界和持锁时间

```java
// ❌ 持锁期间做 IO 操作
synchronized (account) {
    account.debit(amount);
    saveToDatabase();  // IO 操作，持锁时间长
    sendNotification();  // 网络操作
}

// ✅ 正确做法：缩小锁范围
synchronized (account) {
    account.debit(amount);
}
saveToDatabase();  // 锁外操作
sendNotification();
```

### 5. 过度预防导致代码复杂

```java
// ❌ 过度工程：为了防死锁引入复杂逻辑
ReentrantLock lock1 = new ReentrantLock();
ReentrantLock lock2 = new ReentrantLock();
Map<Object, Lock> locks = new ConcurrentHashMap<>();

Object getLock(Object key) {
    return locks.computeIfAbsent(key, k -> new ReentrantLock());
}

// ⚠️ 实际上：大部分死锁来自不正确的加锁顺序
// 简单场景用固定顺序就够了
```
