
# 活锁

## 核心结论

活锁是指线程**没有阻塞**，但一直在做无效工作，无法取得进展。与死锁不同，活锁的线程状态是 RUNNABLE，CPU 可能很高。常见原因是**不断重试失败**（如 CAS 重试、消息重发）。

## 深度解析

### 活锁 vs 死锁

| 对比 | 死锁 | 活锁 |
|------|------|------|
| 线程状态 | BLOCKED/WAITING | RUNNABLE |
| CPU | 低（线程空闲） | 高（线程在重试） |
| 进展 | 完全无进展 | 表面有进展，实际无效 |
| 检测 | jstack 可见 deadlock | 不容易检测 |
| 恢复 | 重启/中断 | 引入随机退避 |

### 活锁示例

#### 示例一：CAS 无限重试

```java
AtomicInteger counter = new AtomicInteger(0);

// 两个线程同时 CAS，永远冲突
// Thread-1: CAS(0→1) 失败 → 重试 CAS(0→1) 失败 → ...
// Thread-2: CAS(0→1) 失败 → 重试 CAS(0→1) 失败 → ...
```

#### 示例二：消息重发

```java
// 两个线程互相让路
void moveForward() {
    while (true) {
        if (tryMoveRight()) return; // 失败
        Thread.yield(); // 让出 CPU
        if (tryMoveLeft()) return; // 失败
        Thread.yield(); // 让出 CPU
    }
}
// Thread-1: 右→左→右→左...
// Thread-2: 左→右→左→右...
// 互相让路，永远过不去
```

#### 示例三：事务冲突重试

```java
// 乐观锁冲突，不断重试
while (true) {
    VersionedData data = read();
    if (casUpdate(data, newData)) break; // 冲突则重试
}
```

### 解决方案

#### 1. 随机退避（最常用）

```java
Random rand = new Random();
while (true) {
    if (tryLock()) break;
    Thread.sleep(rand.nextInt(100) + 50); // 随机等待
}
```

#### 2. 指数退避

```java
int backoff = 100;
while (true) {
    if (tryLock()) break;
    Thread.sleep(backoff);
    backoff = Math.min(backoff * 2, 5000); // 100ms → 200ms → 400ms → ...
}
```

#### 3. 有界重试

```java
int retries = 0;
while (retries < MAX_RETRIES) {
    if (tryLock()) break;
    retries++;
    Thread.sleep(100 * retries);
}
if (retries >= MAX_RETRIES) {
    throw new RuntimeException("获取锁失败，已达最大重试次数");
}
```

## 易错点与踩坑

### 1. Thread.yield() 不能解决活锁

```java
// ❌ 误以为 Thread.yield() 能解决活锁
while (true) {
    if (tryLock()) break;
    Thread.yield();  // ❌ yield 只是建议让出 CPU，不保证其他线程会执行
}

// ✅ 正确做法：使用退避等待
Random rand = new Random();
while (true) {
    if (tryLock()) break;
    Thread.sleep(rand.nextInt(100));  // ✅ 强制等待，其他线程有机会获取锁
}

// ⚠️ yield 的问题：
// 1. 只让出当前时间片，不保证让出后其他线程会执行
// 2. 高优先级线程可能立即抢回 CPU
// 3. 无法保证活锁解除
```

### 2. 随机退避不是万能的

```java
// ❌ 过度依赖随机退避
while (true) {
    if (tryLock()) break;
    Thread.sleep(new Random().nextInt(1000));  // ❌ 随机可能仍然冲突
}

// ✅ 正确做法：结合退避 + 最大重试次数
int retries = 0;
while (retries < MAX_RETRIES) {
    if (tryLock()) break;
    Thread.sleep(calculateBackoff(retries));
    retries++;
}
if (retries >= MAX_RETRIES) {
    throw new RuntimeException("活锁：无法获取资源");
}
```

### 3. 活锁比死锁更难检测

```java
// ❌ 误以为活锁和死锁一样容易被发现
// 死锁：线程 BLOCKED → CPU 低 → 容易发现
// 活锁：线程 RUNNABLE → CPU 高 → 容易被忽略

// ✅ 活锁的特征：
// 1. CPU 使用率异常高（但不是真正的计算负载）
// 2. 接口响应时间长
// 3. 日志显示线程一直在工作，但没有任何进展

// ✅ 检测方法：
// 1. 监控关键操作的完成时间
// 2. 记录重试次数，超过阈值告警
// 3. 使用 Arthas trace 观察方法执行路径
```

### 4. CAS 重试的活锁问题

```java
// ❌ 高并发 CAS 操作可能活锁
AtomicInteger counter = new AtomicInteger(0);

// 1000 个线程同时执行
counter.incrementAndGet();  // CAS 失败 → 重试 → 失败 → ...

// ✅ 解决方案：分段锁 + 减少竞争
// 1. 使用 LongAdder（内部分段）
LongAdder adder = new LongAdder();
adder.increment();  // 分段 CAS，减少冲突

// 2. 减少并发度
Semaphore semaphore = new Semaphore(10);
semaphore.acquire();
try {
    counter.incrementAndGet();
} finally {
    semaphore.release();
}
```

### 5. 消息队列重发导致活锁

```java
// ❌ 无限重试消息
while (true) {
    if (sendMessage(msg)) break;
    // ❌ 如果一直失败，线程空转
}

// ✅ 正确做法：指数退避 + 最大重试
int backoff = 100;
for (int i = 0; i < MAX_RETRIES; i++) {
    if (sendMessage(msg)) return;
    Thread.sleep(backoff);
    backoff = Math.min(backoff * 2, 5000);  // 最大 5 秒
}
throw new RuntimeException("消息发送失败");
```
