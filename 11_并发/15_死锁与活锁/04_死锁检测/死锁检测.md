
# 死锁检测

## 核心结论

死锁检测是**事后**策略，允许死锁发生，通过工具定期检测并定位死锁线程。Java 提供了多种检测工具：`jstack`、`jConsole`、`JProfiler`、`VisualVM`、`Arthas`。

## 深度解析

### 检测工具

#### jstack（最常用）

```bash
# 1. 找到 Java 进程 PID
jps -l

# 2. 打印线程栈（含死锁信息）
jstack <pid>

# 3. 输出中会看到：
# Found one Java-level deadlock:
# =============================
# "Thread-1":
#   waiting to lock monitor 0x... (object 0x..., a java.lang.Object),
#   which is held by "Thread-2"
# "Thread-2":
#   waiting to lock monitor 0x... (object 0x..., a java.lang.Object),
#   which is held by "Thread-1"
```

#### jConsole

1. 启动 `jconsole <pid>`
2. 切换到"线程"选项卡
3. 点击"检测死锁"按钮
4. 显示死锁线程及等待链

#### JMX 编程式检测

```java
ThreadMXBean threadMXBean = ManagementFactory.getThreadMXBean();
long[] deadlockedThreads = threadMXBean.findDeadlockedThreads();
if (deadlockedThreads != null) {
    ThreadInfo[] infos = threadMXBean.getThreadInfo(deadlockedThreads);
    for (ThreadInfo info : infos) {
        log.error("死锁线程: {}", info);
    }
}
```

### 死锁检测原理

**资源分配图**：
- 节点：线程（矩形）+ 资源（圆形）
- 边：线程→资源（等待）、资源→线程（持有）
- 死锁 = 图中存在**环**

如果资源分配图中存在环，则存在死锁。

### 在线检测方案

```java
// 定时任务检测死锁
@Scheduled(fixedRate = 60_000)
public void detectDeadlock() {
    long[] deadlocked = threadMXBean.findDeadlockedThreads();
    if (deadlocked != null) {
        // 告警 + 打印线程栈
        alertService.notify("检测到死锁！");
        ThreadInfo[] infos = threadMXBean.getThreadInfo(deadlocked, true);
        for (ThreadInfo info : infos) {
            log.error("死锁线程: {}\n{}", info.getThreadName(), info.getStackTrace());
        }
    }
}
```

# 死锁定位步骤

## 核心结论

死锁定位的标准流程：**jps 找进程 → jstack 打印线程栈 → 查找死锁信息和 BLOCKED 线程**。

## 深度解析

### 标准流程

```
步骤一：发现异常
  → 系统无响应/接口超时/CPU 正常但线程阻塞

步骤二：找到 Java 进程
  → jps -l 或 ps -ef | grep java

步骤三：打印线程栈
  → jstack <pid> > thread_dump.txt

步骤四：分析线程栈
  → 搜索 "deadlock" / "BLOCKED" / "waiting to lock"
```

### 详细步骤

#### Step 1：发现死锁迹象

- 接口响应超时或无响应
- 线程数不增长（不是线程泄漏）
- CPU 利用率低（线程在等待锁）
- 应用日志停止输出

#### Step 2：找到进程 PID

```bash
jps -l
# 或
ps -ef | grep java

# 输出：12345 com.example.MyApplication
```

#### Step 3：打印线程栈

```bash
jstack 12345 > thread_dump.txt
```

多次打印间隔 3-5 秒，对比两次输出确认是持续死锁而非瞬时。

#### Step 4：分析线程栈

关键搜索词：

```bash
# 搜索死锁信息
grep -A 10 "deadlock" thread_dump.txt

# 搜索阻塞线程
grep -B 5 "BLOCKED" thread_dump.txt

# 搜索等待锁
grep -B 5 "waiting to lock" thread_dump.txt
```

#### Step 5：定位代码

从线程栈中找到具体的类名和方法行号：

```
"Thread-1" #12 prio=5 os_prio=0 tid=0x... nid=0x... waiting for monitor entry [0x...]
   java.lang.Thread.State: BLOCKED (on object monitor)
        at com.example.service.TransferService.transfer(TransferService.java:45)
        - waiting to lock <0x000000076ab4f7a0> (a java.lang.Object)
        at com.example.controller.PayController.pay(PayController.java:28)
```

### Arthas 快速定位

```bash
# 连接到目标进程
java -jar arthas-boot.jar

# 一键检测死锁
thread -b

# 输出死锁线程及其调用栈
```
## 易错点与踩坑

### 1. jstack 输出中的常见误读

```bash
# ❌ 误以为所有 BLOCKED 线程都是死锁
"Thread-1" prio=5 tid=0x... state=BLOCKED
   - waiting to lock <0x000000076ab4f7a0>

# ✅ 实际上：这是正常的锁等待
# BLOCKED 只是说明线程在等锁，不一定是死锁

# ❌ 死锁特征：
# Found one Java-level deadlock:
# 多个线程互相等待对方的锁
```

### 2. 检测频率与性能开销的矛盾

```java
// ❌ 检测死锁过于频繁
@Scheduled(fixedRate = 100)  // 每100ms检测一次
public void detectDeadlock() {
    threadMXBean.findDeadlockedThreads();  // 性能开销大！
}

// ✅ 合理设置检测频率
@Scheduled(fixedRate = 60_000)  // 每分钟检测一次
public void detectDeadlock() {
    // 检测逻辑
}

// ⚠️ 注意：findDeadlockedThreads() 需要遍历线程栈，开销不小
```

### 3. Object.wait 不是死锁

```java
// ❌ 误以为 Object.wait 的线程是死锁
synchronized (obj) {
    while (condition not met) {
        obj.wait();  // 线程状态是 WAITING，不是死锁！
    }
}

// ✅ 区分：
// - BLOCKED：等待锁（synchronized）
// - WAITING：等待特定条件（wait/await）
// - TIMED_WAITING：限时等待（sleep/join/awaitNanos）
```

### 4. Arthas thread -b 只检测 Object 锁死锁

```bash
# ❌ 误以为 Arthas 能检测所有类型的死锁
thread -b

# ✅ 实际上：thread -b 只检测 synchronized 关键字的死锁
# ReentrantLock 的死锁检测不到！

# ✅ 检测 ReentrantLock 死锁：
# 1. thread 查看 BLOCKED 线程
# 2. thread <tid> 查看线程栈
# 3. 分析 Lock 等待链
```

### 5. 死锁检测是事后手段，不是预防手段

```java
// ❌ 依赖检测来"解决"死锁
@Scheduled(fixedRate = 60_000)
public void detectAndRecover() {
    if (hasDeadlock()) {
        restartService();  // ❌ 不好：不应该让死锁发生
    }
}

// ✅ 正确：死锁检测 + 告警，人工介入
@Scheduled(fixedRate = 60_000)
public void detectAndAlert() {
    if (hasDeadlock()) {
        alertOps();  // ✅ 告警，让开发人员修复代码
        log.error("检测到死锁，请检查代码");
    }
}

// ✅ 根本解法：预防
// 1. 代码审查
// 2. 固定加锁顺序
// 3. 使用 tryLock 替代 synchronized
```
