
# 饥饿

## 核心结论

饥饿是指某个线程**长期无法获得所需资源**，虽然线程没有被阻塞，但始终无法执行或执行极度缓慢。常见原因是线程优先级设置不当、锁的不公平竞争。

## 深度解析

### 饥饿的原因

| 原因 | 说明 |
|------|------|
| 高优先级线程抢占 | 低优先级线程永远得不到 CPU |
| 非公平锁 | 后续线程不断插队，排队的线程永远等不到 |
| 写锁饥饿 | 大量读线程持续持有读锁，写线程无法获取 |

### 饥饿 vs 死锁 vs 活锁

| 对比 | 死锁 | 活锁 | 饥饿 |
|------|------|------|------|
| 线程状态 | BLOCKED | RUNNABLE | RUNNABLE/BLOCKED |
| 能否完成 | 永远不能 | 理论上能，实际很难 | 需要极长时间 |
| CPU | 低 | 高 | 不定 |
| 根因 | 循环等待 | 无效重试 | 被其他线程抢占 |

### Java 中的饥饿场景

#### 1. 非公平锁饥饿

```java
// ReentrantLock 非公平模式：新线程不断 CAS 插队
// 排队中的线程可能长时间等不到锁
ReentrantLock lock = new ReentrantLock(false); // 非公平（默认）
```

#### 2. 读写锁写饥饿

```java
// 大量读线程持有读锁，写线程永远获取不到
ReentrantReadWriteLock rwl = new ReentrantReadWriteLock();
// 100个读线程持续读取 → 写线程饥饿
```

#### 3. 线程优先级

```java
Thread low = new Thread(task);  // priority=1
Thread high = new Thread(task); // priority=10
// 高优先级线程可能完全抢占低优先级线程
```

### 解决方案

| 方案 | 说明 |
|------|------|
| 公平锁 | `new ReentrantLock(true)`，按 FIFO 顺序获取 |
| 合理优先级 | 不要设置极端的线程优先级 |
| 读写锁公平 | `new ReentrantReadWriteLock(true)` |
| 避免持锁时间过长 | 缩小锁范围 |

## 易错点与踩坑

### 1. 公平锁的性能陷阱

```java
// ❌ 误以为公平锁是解决饥饿的银弹
ReentrantLock fairLock = new ReentrantLock(true);  // 公平模式

// ⚠️ 公平锁的问题：
// 1. 需要维护等待队列，每次获取锁都要检查队列
// 2. 吞吐量比非公平锁低 10-100 倍
// 3. 如果等待时间长，公平锁反而更慢

// ✅ 正确选择：
// - 高并发场景 → 非公平锁（吞吐量优先）
// - 业务必须保证公平 → 公平锁
// - 更好的方案 → 合理设计减少持锁时间
```

### 2. 线程优先级在 JVM 中不保证

```java
// ❌ 过度依赖线程优先级
Thread low = new Thread(task);
low.setPriority(Thread.MIN_PRIORITY);  // 1
Thread high = new Thread(task);
high.setPriority(Thread.MAX_PRIORITY); // 10

// ⚠️ 问题：
// 1. JVM 线程优先级是"建议性"的，不是强制的
// 2. 操作系统可能忽略 JVM 设置的优先级
// 3. 不同 OS 对优先级的实现不同

// ✅ 正确做法：
// 1. 不要依赖线程优先级来控制业务
// 2. 使用信号量/队列等并发工具控制资源分配
// 3. 合理设计避免资源竞争
```

### 3. 读写锁的写饥饿问题

```java
// ❌ 误以为 ReentrantReadWriteLock 解决了读写冲突
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();

// 场景：大量读线程
for (int i = 0; i < 100; i++) {
    new Thread(() -> {
        rwLock.readLock().lock();
        try {
            // 长时间读操作
            Thread.sleep(1000);
        } finally {
            rwLock.readLock().unlock();
        }
    }).start();
}

// ❌ 写线程可能永远获取不到锁
// 因为读锁是共享的，新读线程可以随时加入
// 写线程永远等不到"没有读线程"的时刻

// ✅ 解决方案：StampedLock 或写优先
ReentrantReadWriteLock fairRwLock =
    new ReentrantReadWriteLock(true);  // 公平模式，写优先
```

### 4. 饥饿不一定能被自动恢复

```java
// ❌ 误以为饥饿的线程最终会自动恢复执行
// 实际情况：
// 1. 高优先级线程持续占用 CPU，低优先级线程永远得不到调度
// 2. 非公平锁场景下，新线程可能不断插队

// ✅ 饥饿的典型表现：
// - 线程日志显示"等待资源 XXX"
// - 接口超时，但资源使用率低
// - 特定线程（低优先级/后加入）始终得不到执行

// ✅ 解决方案：
// 1. 监控线程等待时间
// 2. 设置等待超时告警
// 3. 强制调整线程优先级或切换锁模式
```

### 5. 单例模式中的饥饿

```java
// ❌ 双重检查锁定的有序性问题
class Singleton {
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {  // 线程 A、B 同时通过
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();  // 线程 A 初始化
                }
            }
        }
        return instance;  // 线程 B 可能返回未完成初始化的对象！
    }
}

// ⚠️ 注意：这里主要是可见性问题，不是饥饿
// 但在高并发下，可能导致某些线程一直获取到错误的对象
```
