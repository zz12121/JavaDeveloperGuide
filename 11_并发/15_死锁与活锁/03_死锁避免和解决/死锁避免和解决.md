
# 死锁避免

## 核心结论

死锁避免与死锁预防不同，它**不事先限制**，而是在资源分配时**动态检查**是否会导致死锁。经典算法是**银行家算法**。

## 深度解析

### 预防 vs 避免 vs 检测

| 策略 | 思路 | 时机 |
|------|------|------|
| 死锁预防 | 破坏必要条件，限制线程行为 | 事前 |
| 死锁避免 | 动态检查资源分配是否安全 | 事中 |
| 死锁检测 | 允许死锁发生，定期检测并恢复 | 事后 |

### 银行家算法

模拟银行放贷：检查分配资源后系统是否仍处于"安全状态"。

```
概念：
- Available：系统可用资源
- Max：每个线程的最大需求
- Allocation：已分配资源
- Need = Max - Allocation：还需资源

安全状态：存在一个安全序列，所有线程都能按顺序完成
不安全状态：找不到安全序列，可能导致死锁
```

**分配前检查**：如果分配后系统仍处于安全状态，则分配；否则让线程等待。

### 银行家算法示例

```
可用资源：A=3, B=3, C=2

已分配：
P0: A=0, B=1, C=0  | Max: A=7, B=5, C=3  | Need: A=7, B=4, C=3
P1: A=2, B=0, C=0  | Max: A=3, B=2, C=2  | Need: A=1, B=2, C=2
P2: A=3, B=0, C=2  | Max: A=9, B=0, C=2  | Need: A=6, B=0, C=0
P3: A=2, B=1, C=1  | Max: A=2, B=2, C=2  | Need: A=0, B=1, C=1
P4: A=0, B=0, C=2  | Max: A=4, B=3, C=3  | Need: A=4, B=3, C=1

安全序列：P1 → P3 → P4 → P2 → P0（或 P3 → P1 → ...等）
```

### Java 中的实际应用

银行家算法在 Java 开发中几乎**不直接使用**，因为：
1. 性能开销大，每次加锁都要检查
2. 难以预知每个线程的最大资源需求
3. Java 提供了更简单的预防手段（按顺序加锁、tryLock）

实际中用**死锁预防**而非**死锁避免**。

# 解决死锁

## 核心结论

死锁发生后，只能通过**外部干预**恢复：重启进程、中断线程、或强制释放锁。但根本解法是**预防**，在编码阶段就避免死锁的产生。

## 深度解析

### 死锁恢复方案

|方案|操作|影响|
|---|---|---|
|**重启进程**|kill + 重启|最简单，但有服务中断|
|**中断线程**|`Thread.interrupt()`|被中断线程抛异常释放锁|
|**强制释放**|无直接 API|synchronized 无法强制释放|
|**jstack + kill**|定位后重启|适合线上应急|

### 编码阶段预防（核心）

#### 1. 固定加锁顺序

```java
// 所有线程都按 A→B→C 的顺序加锁
synchronized (lockA) {
    synchronized (lockB) {
        // 安全
    }
}
```

#### 2. tryLock + 超时 + 释放

```java
Lock lockA = new ReentrantLock();
Lock lockB = new ReentrantLock();

try {
    if (lockA.tryLock(100, TimeUnit.MILLISECONDS)) {
        try {
            if (lockB.tryLock(100, TimeUnit.MILLISECONDS)) {
                try {
                    // 执行业务
                } finally {
                    lockB.unlock();
                }
            }
        } finally {
            lockA.unlock();
        }
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}
```

#### 3. 使用并发工具替代显式锁

```java
// 用 ConcurrentHashMap 替代 synchronized HashMap
// 用 AtomicInteger 替代 synchronized int
// 用 CountDownLatch 替代 wait/notify
```

#### 4. 减少锁的持有时间

```java
// 错误：持锁期间做 IO
synchronized (lock) {
    data = readFile();   // IO 操作
    result = process(data);
}

// 正确：缩小锁范围
data = readFile();  // 锁外 IO
synchronized (lock) {
    result = process(data);  // 只锁必要操作
}
```

#### 5. 使用 open call（开放调用）

```java
// 不持锁调用外部方法，避免外部方法再获取锁形成死锁
synchronized (lock) {
    prepareData();
}
// 锁外调用外部方法
externalService.notify(prepareData());
```

### 线上应急流程

```
1. 发现异常（接口无响应/超时）
2. jstack 定位死锁线程
3. 评估影响：是否需要立即恢复？
4. 短期：重启进程
5. 长期：分析代码，修复加锁顺序
6. 添加死锁检测告警
```
## 易错点与踩坑

### 1. 银行家算法在 Java 开发中几乎不用

```java
// ❌ 误以为银行家算法是解决死锁的银弹
// 实际上：银行家算法在 Java 中几乎不适用

// 原因：
// 1. 每次资源分配都要检查，性能开销大
// 2. 需要预先知道每个线程的最大资源需求（难以预知）
// 3. Java 的锁机制不支持"试探性分配"

// ✅ 实际开发中：死锁预防（固定顺序/tryLock）比银行家算法更实用
```

### 2. tryLock 必须配合 finally 释放

```java
// ❌ tryLock 后忘记释放
Lock lock = new ReentrantLock();
public void doSomething() {
    if (lock.tryLock()) {
        // ... 抛异常了？
        // 锁永远不会释放！
    }
}

// ✅ 正确做法：try-finally
public void doSomething() {
    try {
        if (lock.tryLock(5, TimeUnit.SECONDS)) {
            try {
                // 业务逻辑
            } finally {
                lock.unlock();  // 必须在 finally 中释放
            }
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}
```

### 3. 重启不能根本解决问题

```java
// ❌ 误以为重启 JVM 能彻底解决死锁问题
// 实际上：重启只是临时恢复，死锁代码还在

// ✅ 正确做法：
// 1. 短期：重启恢复服务
// 2. 长期：修复代码中的加锁顺序问题
// 3. 预防：添加死锁检测告警
// 4. 上线前：代码审查检查多锁场景
```

### 4. jstack 定位死锁的常见误读

```java
// ❌ 误读 jstack 输出
// Found one Java-level deadlock:
// "Thread-1" waiting to lock <0x000000076ab4f7a0>
// which is held by "Thread-2"

// ❌ 误以为死锁线程一定在代码当前行
// 实际上：锁对象地址只是一个引用，要结合代码分析

// ✅ 正确解读：
// 1. 找到 "waiting to lock" 的对象地址
// 2. 找到 "held by" 的对象地址
// 3. 回到代码中找到对应的 synchronized 块
// 4. 分析加锁顺序
```

### 5. 忽略 open call 的重要性

```java
// ❌ 在持锁状态下调用外部方法（可能形成更复杂的死锁）
synchronized (account) {
    // 持锁状态下调用外部服务
    externalService.process(account);  // 外部服务可能反过来调用我们的方法
    // 如果外部服务也需要这个锁 → 死锁！
}

// ✅ 正确做法：open call
synchronized (account) {
    prepareData(account);  // 只做必要的准备工作
}
// 锁外调用外部方法
externalService.process(preparedData);
```
