# 泛型擦除后的类型桥接（Bridge Method）

## 问题背景

泛型擦除后，子类重写泛型父类方法时，类型签名会不一致，编译器需要生成 **桥接方法(Bridge Method)** 来维护多态。

## 问题示例

```java
// 父类（泛型）
public class Comparable<T> {
    public int compareTo(T other) { return 0; }
}

// 子类（指定具体类型）
public class Integer extends Comparable<Integer> {
    @Override
    public int compareTo(Integer other) { ... }
}
```

**擦除后，父类变为：**
```java
public class Comparable {
    public int compareTo(Object other) { return 0; }
}
```

**问题**：子类重写的是 `compareTo(Integer)`，而父类擦除后要求 `compareTo(Object)`，签名不匹配，多态无法正常工作。

## 编译器生成桥接方法

编译器自动在 `Integer` 类中生成桥接方法：

```java
// 编译器自动生成（不会出现在源码中）
public int compareTo(Object other) {
    return this.compareTo((Integer) other);  // 委托给实际方法
}
```

这样通过父类类型调用 `compareTo` 时，会先走桥接方法，再调用真正的实现。

## 查看桥接方法

```java
for (Method m : Integer.class.getDeclaredMethods()) {
    if (m.isBridge()) {
        System.out.println("Bridge: " + m);
    }
}
```

## 桥接方法的特征

| 特征 | 说明 |
|------|------|
| 来源 | 编译器自动生成，不在源码中 |
| 访问修饰符 | `synthetic` + `bridge`（通过反射可查 `isBridge()`） |
| 签名 | 与擦除后的父类方法签名一致 |
| 实现 | 内部调用真正的重写方法，加入强转 |

## 更复杂的场景

```java
public interface Converter<F, T> {
    T convert(F from);
}

public class StringToInteger implements Converter<String, Integer> {
    @Override
    public Integer convert(String from) {
        return Integer.parseInt(from);
    }
    // 编译器还会生成：
    // Object convert(Object from) { return convert((String) from); }
}
```
