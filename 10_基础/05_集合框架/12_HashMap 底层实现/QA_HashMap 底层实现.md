# HashMap 底层实现

## Q1：HashMap 的底层数据结构是什么？

**A：**
JDK 8 中 HashMap 底层是 **数组 + 链表 + 红黑树**。数组存桶，每个桶中可能是链表或红黑树。当链表长度达到 8 且数组长度 ≥ 64 时，链表转为红黑树。

---

## Q2：HashMap 的哈希函数是怎么设计的？

**A：**
```java
static final int hash(Object key) {
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```
将 `hashCode()` 的高 16 位与低 16 位异或，让高位特征参与低位运算，减少碰撞。桶定位用 `(n - 1) & hash` 位运算替代取模。

---

## Q3：为什么 HashMap 的数组长度必须是 2 的幂次？

**A：**
1. `hash % n` 可以用位运算 `hash & (n - 1)` 代替，位运算更快
2. `n - 1` 的二进制全为 1，能保证 hash 的低位信息全部参与桶定位
3. 扩容时元素的新位置 = 旧位置 或 旧位置 + 旧容量（只需判断 hash 新增的高位是 0 还是 1）

---

## Q4：JDK 7 和 JDK 8 的 HashMap 有什么区别？

**A：**

| 维度 | JDK 7 | JDK 8 |
|------|-------|-------|
| 结构 | 数组 + 链表 | 数组 + 链表 + 红黑树 |
| 链表插入 | 头插法 | 尾插法 |
| 多线程扩容 | 可能死循环 | 不会 |
| 哈希扰动 | 9 次扰动 | 1 次异或 |

---

## Q5：HashMap 什么时候扩容？

**A：**  
当 `size > capacity × loadFactor` 时扩容。默认初始容量 16，负载因子 0.75，即第 13 个元素插入时触发首次扩容。扩容后容量变为原来的 **2 倍**。

---

## Q6：为什么负载因子默认是 0.75？

**A：**  
0.75 是时间和空间的平衡点：
- 太大：哈希碰撞增多，链表变长，O(1) 退化为 O(n)
- 太小：空间浪费严重
- 0.75 时，根据泊松分布，链表长度达到 8 的概率约为 0.00000006，几乎不会发生

---

## Q7：JDK 8 扩容时元素是怎么迁移的？

**A：**  
JDK 8 扩容不需要重新计算 hash，而是通过 `e.hash & oldCapacity` 判断：
- 结果为 0 → 新位置不变
- 结果为 1 → 新位置 = 旧位置 + 旧容量
元素被分成低位链表和高位链表，分别放到新数组的两个桶中，保持原有顺序。

---

## Q8：JDK 7 的 HashMap 多线程扩容为什么会死循环？

**A：**  
JDK 7 使用头插法迁移链表，多线程并发扩容时，头插法会导致链表节点顺序反转，可能形成**环形链表**。后续 `get()` 操作遍历环形链表时进入死循环。JDK 8 改为尾插法解决了此问题。

---

## Q9：描述一下 HashMap 的 put 流程？

**A：**
1. 计算 key 的 hash 值：`(h = key.hashCode()) ^ (h >>> 16)`
2. 定位桶：`index = (table.length - 1) & hash`
3. 如果桶为空，直接创建 Node 放入
4. 如果桶不为空，遍历链表或红黑树：
    - 如果 key 已存在（hash 相同且 equals 为 true），替换 value
    - 如果是红黑树节点，调用 `putTreeVal()` 插入
    - 否则尾插到链表末尾
5. 如果链表长度 ≥ 8 且数组长度 ≥ 64，链表转为红黑树
6. `size++`，如果 `size > threshold`，触发扩容

---

## Q10：HashMap 判断 key 相等的条件是什么？

**A：**  
两个条件**同时满足**：
1. `hash` 值相同
2. `key == k`（引用相同）或 `key.equals(k)` 返回 true
```java
if (p.hash == hash &&
    ((k = p.key) == key || (key != null && key.equals(k))))
```

---

## Q11：链表转红黑树的条件是什么？

**A：**  
两个条件**同时满足**：
1. 链表长度 ≥ 8（`TREEIFY_THRESHOLD`）
2. 数组长度 ≥ 64（`MIN_TREEIFY_CAPACITY`）
如果链表长度 ≥ 8 但数组长度 < 64，会**优先扩容**而不是树化。

---

## Q12：HashMap 为什么在链表长度达到 8 时转为红黑树？

**A：**
1. 链表过长时查询退化为 O(n)，红黑树可以保证 O(log n)
2. 根据泊松分布，负载因子 0.75 时链表长度达到 8 的概率仅千万分之六，说明遇到了极端哈希碰撞
3. 此时转为红黑树是性能保底策略

---

## Q13：为什么不一开始就用红黑树？

**A：**
1. 红黑树节点占用内存约为链表节点的 **2 倍**（多 parent 指针和 color 字段）
2. 当链表很短时（大多数情况），链表的 O(n) 和红黑树的 O(log n) 差异不大
3. 红黑树的插入/删除有旋转开销，链表的尾插 O(1) 更快
4. 综合考虑：正常情况用链表（省内存），极端情况自动转红黑树（保性能）

---

## Q14：为什么退化阈值是 6 而不是 8？

**A：**  
避免频繁转换。如果进（8）和出（8）的阈值一样，元素增减时会在链表和红黑树间来回转换。设为 6 留出缓冲区，减少不必要的转换开销。

---

## Q15：为什么选红黑树而不是 AVL 树？

**A：**  
AVL 树严格平衡，查找稍快，但插入/删除需要更多旋转操作。HashMap 是频繁增删的场景，红黑树的旋转次数不超过 3 次，维护成本更低，是更合适的折中。

---

## Q16：HashMap 的 key 为什么推荐用 String？

**A：**
1. **不可变**：String 创建后内容不可变，`hashCode()` 值不会变化
2. **缓存 hashCode**：String 内部缓存了 hash 值，首次计算后直接返回，性能好
3. **equals 完善**：按内容比较，能正确判断相等

---

## Q17：如果用自定义对象作为 HashMap 的 key，需要注意什么？

**A：**
1. **重写 `hashCode()` 和 `equals()`**，且必须一致（`equals` 为 true 时 hashCode 必须相同）
2. **保证不可变**：用作 key 的对象属性不应该被修改，否则 hashCode 变化后找不到原来存储的值
3. 最好将所有字段声明为 **final**

---

## Q18：为什么可变对象不适合做 key？

**A：**  
如果 key 对象的属性被修改，`hashCode()` 会返回不同的值，导致无法在原来的桶中找到该 key，造成"存得进、取不出"的问题。

---

## Q19：HashMap 有几种遍历方式？

**A：**  
三种：
1. **entrySet**：遍历键值对 `for (Entry e : map.entrySet())`
2. **keySet**：遍历 key，再 `map.get(key)` 获取 value
3. **values**：只遍历 value

---

## Q20：entrySet 和 keySet 哪个效率更高？

**A：**  
**entrySet 更高**。
- `entrySet`：直接遍历 Entry 对象，已经包含 key 和 value，无需额外查找
- `keySet`：遍历 key 后，每次 `map.get(key)` 需要重新计算 hash + 定位桶 + 遍历链表，多了一次查找开销
如果只需要 value，用 `values()`；如果需要 key 和 value，用 `entrySet()`。

---

## Q21：遍历时能删除元素吗？

**A：**  
直接用 `map.remove()` 会抛 `ConcurrentModificationException`。安全删除方式：
- `Iterator.remove()`
- `entrySet().removeIf(predicate)`（Java 8+）
