# 反射的性能开销

## 核心结论

反射的性能比直接调用慢约 **10~100 倍**（取决于 JVM 优化程度和调用频率），主要开销来自安全检查、参数装箱、方法查找等。通过缓存 Method/Field 对象、setAccessible 预设、JIT 热点优化可以大幅缩小差距，但在性能敏感场景仍应避免反射。

## 深度解析

### 性能开销来源

| 开销来源 | 说明 |
|---------|------|
| **安全检查** | 每次调用检查访问权限（setAccessible 可关闭） |
| **方法查找** | 通过方法名和参数类型查找 Method 对象（可缓存） |
| **参数装箱** | `invoke(Object, Object...)` 导致基本类型装箱拆箱 |
| **方法分发** | 需要动态解析调用目标 |
| **JIT 优化限制** | 反射调用难以被 JIT 充分内联优化 |

### 性能对比

```
直接调用：     ~1x（基准）
反射调用：     ~10-100x（首次调用更慢）
缓存反射调用： ~2-10x（Method 对象缓存后）
MethodHandle： ~1.5-5x（JDK 7+，接近直接调用）
```

> 实际性能因 JVM 版本、调用频率、JIT 优化程度而异，现代 JVM 对反射的优化已显著改善。

### 性能优化方案

#### 1. 缓存 Method/Field/Constructor 对象

```java
// ❌ 每次都查找（慢）
Method method = clazz.getMethod("getName");
method.invoke(target);

// ✅ 初始化时查找并缓存（快）
private static final Method GET_NAME;
static {
    GET_NAME = User.class.getMethod("getName");
}
// 后续直接使用 GET_NAME.invoke(target)
```

#### 2. 使用 setAccessible(true)

```java
// 跳过权限检查，减少一次安全校验开销
Method method = clazz.getDeclaredMethod("privateMethod");
method.setAccessible(true);
```

#### 3. 使用 MethodHandle（JDK 7+）

```java
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle handle = lookup.findVirtual(User.class, "getName", MethodType.methodType(String.class));
String name = (String) handle.invoke(user); // 比 Method.invoke 更快
```

#### 4. 避免在热路径中使用反射

```java
// ❌ 循环中频繁反射调用
for (Object item : list) {
    Method method = item.getClass().getMethod("getValue");
    method.invoke(item);
}

// ✅ 提取到循环外，缓存 Method
Method method = Item.class.getMethod("getValue");
for (Object item : list) {
    method.invoke(item);
}
```

### 框架如何处理反射性能

- **Spring**：启动时扫描并缓存所有 Bean 的元数据（Method、Field），运行时直接使用缓存
- **MyBatis**：首次执行 SQL 时解析映射，缓存反射操作后的 Setter/Getter 方法
- **Jackson**：序列化时缓存类的字段反射信息（`SerializerCache`）
