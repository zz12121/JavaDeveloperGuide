# static 关键字

## Q1：static 修饰的成员有哪四种？分别是什么？

**A**：
1. **静态属性**：类级别变量，所有对象共享同一份，存储在方法区
2. **静态方法**：无需实例即可调用，不能访问实例变量和实例方法
3. **静态代码块**：类加载时执行一次，用于静态数据初始化
4. **静态内部类**：不持有外部类引用，只能访问外部类静态成员

---

## Q2：静态属性和实例属性的区别？

**A**：

| 对比维度 | 静态属性 | 实例属性 |
|---------|---------|---------|
| 归属 | 属于类 | 属于对象 |
| 内存位置 | 方法区（元空间） | 堆（对象实例中） |
| 共享性 | 所有对象共享 | 每个对象独立一份 |
| 访问方式 | `类名.属性`（推荐） | `对象.属性` |
| 生命周期 | 类加载 ~ 类卸载 | 对象创建 ~ 对象回收 |

---

## Q3：静态方法有哪些限制？为什么？

**A**：
- **不能使用 `this`**：静态方法通过类名调用，调用时可能没有任何对象实例，故没有 this 上下文
- **不能使用 `super`**：同理，super 也依赖对象上下文
- **不能直接访问实例变量和实例方法**：实例成员必须依附于某个对象才存在，静态方法无法确定是哪个对象
- **可以被子类继承，但不能被重写（Override）**：子类定义同名静态方法是"隐藏（hide）"，而非多态重写

---

## Q4：静态代码块的执行时机和执行次数？

**A**：
- **时机**：类**首次被加载**时（第一次使用该类时）执行，发生在类初始化阶段
- **次数**：整个 JVM 生命周期内**只执行一次**
- **顺序**：同一个类中多个静态代码块**从上到下**依次执行；先于静态属性的显式赋值语句……实际上静态属性的定义和静态代码块**按代码书写顺序**依次执行
- **与构造方法的区别**：静态代码块只执行一次，构造方法每次 new 都执行

---

## Q5：类中各种初始化代码的执行顺序？

**A**：完整顺序如下（父类优先于子类）：

```
1. 父类静态属性 & 父类静态代码块（按书写顺序）
2. 子类静态属性 & 子类静态代码块（按书写顺序）
3. 父类实例属性 & 父类普通代码块（按书写顺序）
4. 父类构造方法
5. 子类实例属性 & 子类普通代码块（按书写顺序）
6. 子类构造方法
```

静态相关的（1、2）只在类加载时执行一次；每次 new 对象都会执行 3~6。

---

## Q6：静态内部类和普通内部类的区别？

**A**：

| 特性 | 普通内部类 | 静态内部类 |
|------|-----------|-----------|
| 创建方式 | `new Outer().new Inner()` | `new Outer.Inner()` |
| 外部类引用 | 隐式持有（可能导致内存泄漏） | 不持有 |
| 访问外部成员 | 可访问外部类所有成员（静态+实例） | 只能访问外部类静态成员 |
| 应用场景 | 与外部类强耦合的场景 | Builder、Entry、Holder 单例等 |

---

## Q7：为什么 HashMap.Entry 设计为静态内部类？

**A**：`Map.Entry` 设计为静态内部类（`static class Entry`），原因是：
1. `Entry` 对象只需要持有 key/value/hash 等数据，**不需要访问外部 HashMap 的实例成员**
2. 若设计为普通内部类，每个 `Entry` 对象都会持有 `HashMap` 的引用（8字节指针），造成**内存浪费**
3. 静态内部类可以**独立创建**，不强依赖 HashMap 实例，设计更合理

---

## Q8：静态内部类实现懒加载单例的原理？

**A**：
```java
public class Singleton {
    private Singleton() {}

    private static class Holder {
        // 类加载时初始化，JVM 保证线程安全
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

**原理**：
- `Holder` 类只在第一次调用 `getInstance()` 时才被加载（懒加载）
- JVM 类加载的**初始化阶段由 JVM 保证线程安全**（加锁），所以天然线程安全
- 相比 `synchronized` 方法，没有锁开销，性能更好

---

## Q9：能用对象实例访问静态属性/方法吗？

**A**：语法上**可以**，但**强烈不推荐**，IDE 通常会给出警告。  
原因：通过实例访问静态成员会产生误导，让人以为这是实例级别的数据，降低代码可读性。  
**正确做法**：始终用 `类名.静态属性` 或 `类名.静态方法()` 的方式访问。

---

## Q10：static 和 final 组合使用的场景？

**A**：`public static final` 常用于定义**编译期常量**（常量池优化）：
```java
public static final int MAX_SIZE = 100;       // 基本类型常量
public static final String APP_NAME = "Demo"; // 字符串常量
```
- `static`：属于类，无需实例访问
- `final`：值不可改变
- 两者组合 = 全局唯一的不可变常量，命名约定为**全大写+下划线**
- 若值为基本类型或 String，编译器会做**常量折叠（内联）**优化，调用方直接用值而非引用
