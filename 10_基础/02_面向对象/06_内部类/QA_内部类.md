# 内部类（成员内部类/静态内部类/局部内部类/匿名内部类）

## Q1：四种内部类？

**A**： 
- **成员内部类**：定义在类内方法外，隐式持有外部类引用，可访问所有成员
- **静态内部类**：static 修饰，不持有外部类引用，只能访问静态成员
- **局部内部类**：定义在方法内，只能访问 effectively final 的局部变量
- **匿名内部类**：没有名字，一次性使用，可实现接口或继承类

---

## Q2：核心区别

**A**： 
- 成员内部类/局部内部类/匿名内部类都**持有外部类引用**
- 静态内部类**不持有外部类引用**，更轻量

---

## Q3：实际应用？

**A**： 
- 静态内部类 → Builder 模式、Map.Entry
- 匿名内部类 → 事件回调、线程 Runnable
- Lambda → 匿名内部类的简化替代

---

## Q4：为什么局部内部类只能访问 final 的局部变量？

**A**：局部内部类访问的局部变量会被 **值拷贝**​ 到内部类对象中。
如果允许修改局部变量，那么在不同时间点创建的内部类对象可能持有不同的值，导致逻辑混乱和不可预测的行为。
Java 通过强制要求变量是 `final`或 **effectively final**，来保证值拷贝的确定性和一致性。
Java 8 引入 effectively final，是为了让 Lambda 和局部内部类在使用局部变量时，不必显式声明 `final`。
它的本质是：变量在初始化后不能再被赋值，从而保证值拷贝的确定性。**​这说明 Java 关注的是“不可变性”这一事实，而不是 `final`关键字本身。

---

## Q5：静态内部类有什么实际用途？

**A**：最常见的用途是 **Builder 模式**（如 `StringBuilder`、`Request.Builder`）和 **辅助类**（如 `Map.Entry`、`LinkedList.Node`）。静态内部类不依赖外部类实例，更独立，创建不需要外部类对象，内存开销更小。

---

## Q6：内部类会导致内存泄漏吗？

**A**：会。非静态内部类（成员/局部/匿名）隐式持有外部类引用。如果内部类对象的生命周期长于外部类（如将匿名内部类注册到全局监听器），外部类就无法被 GC 回收。解决方法：使用**静态内部类**或及时**注销监听器**。
