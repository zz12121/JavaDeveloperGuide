# 异常体系

## 先说结论

Java 异常体系以 `Throwable` 为根，分为 `Error`（JVM 级别不可恢复错误）和 `Exception`（程序可处理的异常）。`Exception` 再分为**受检异常（Checked）**和**非受检异常（Unchecked）**。受检异常必须显式处理，非受检异常（即 `RuntimeException` 及其子类）编译器不强制要求处理。

---

## 深度解析

### 异常体系类图

```
Throwable
├── Error                         （JVM 级别，程序无法处理）
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   ├── VirtualMachineError
│   └── ...
└── Exception                     （程序可处理）
    ├── IOException               （受检异常 Checked）
    │   ├── FileNotFoundException
    │   └── SocketException
    ├── SQLException              （受检异常 Checked）
    ├── CloneNotSupportedException（受检异常 Checked）
    └── RuntimeException          （非受检异常 Unchecked）
        ├── NullPointerException
        ├── ArrayIndexOutOfBoundsException
        ├── ClassCastException
        ├── IllegalArgumentException
        ├── IllegalStateException
        ├── NumberFormatException
        ├── ArithmeticException
        └── UnsupportedOperationException
```

---

### Error vs Exception

| 对比维度 | Error | Exception |
|---------|-------|-----------|
| 含义 | JVM 或系统级别的严重错误 | 程序逻辑或运行时可处理的异常 |
| 是否可恢复 | 通常不可恢复 | 大多数可以捕获处理 |
| 是否需要处理 | 不需要（也无意义） | 受检异常必须处理 |
| 典型代表 | `OutOfMemoryError`、`StackOverflowError` | `IOException`、`NullPointerException` |
| 触发原因 | 内存溢出、栈溢出、类加载失败 | 空指针、越界、类型转换错误等 |

---

### 受检异常（Checked Exception）vs 非受检异常（Unchecked Exception）

| 对比维度 | 受检异常 | 非受检异常 |
|---------|---------|----------|
| 定义 | `Exception` 的子类（除 RuntimeException 分支） | `RuntimeException` 及其子类 |
| 编译器要求 | **必须** `try-catch` 或 `throws` 声明，否则编译错误 | 不强制要求处理 |
| 触发场景 | 外部资源操作（文件、网络、数据库） | 编程错误（空指针、越界、类型错误） |
| 典型代表 | `IOException`、`SQLException`、`CloneNotSupportedException` | `NullPointerException`、`ArrayIndexOutOfBoundsException` |
| 设计意图 | 提醒调用者：这里可能出错，请处理 | 应通过修复代码逻辑来消除，而非捕获 |

---

### Throwable 核心方法

```java
Throwable e = new RuntimeException("出错了");

e.getMessage();          // 返回异常信息字符串："出错了"
e.getCause();            // 返回导致此异常的原因（Throwable），无则返回 null
e.toString();            // 返回 "java.lang.RuntimeException: 出错了"
e.printStackTrace();     // 打印完整异常堆栈到标准错误流（调试用）
e.getStackTrace();       // 返回 StackTraceElement[] 数组（可编程访问堆栈）

// 异常链（Chained Exception）
try {
    // 原始操作
} catch (IOException e) {
    throw new RuntimeException("业务层包装异常", e);  // 将原因 e 传入新异常
}
```

---

### 常见 RuntimeException 及触发场景

| 异常类 | 典型触发场景 |
|--------|------------|
| `NullPointerException` | 对 null 对象调用方法或访问字段 |
| `ArrayIndexOutOfBoundsException` | 数组下标越界（负数或 >= length） |
| `StringIndexOutOfBoundsException` | String 下标越界 |
| `ClassCastException` | 强制类型转换失败（非该类型的对象） |
| `NumberFormatException` | 字符串转数字格式错误（`Integer.parseInt("abc")`） |
| `ArithmeticException` | 整数除以零（`10 / 0`） |
| `IllegalArgumentException` | 传入方法的参数不合法 |
| `IllegalStateException` | 对象当前状态不允许执行该操作 |
| `StackOverflowError` | 递归过深导致栈帧耗尽（注意：这是 Error） |
| `UnsupportedOperationException` | 调用了不支持的操作（如不可修改集合的 add） |

### 异常处理的两种方式

#### 方式一：try-catch 捕获处理
```java
try {
    FileInputStream fis = new FileInputStream("file.txt");
} catch (FileNotFoundException e) {
    // 处理异常：记录日志、返回默认值、重试等
    System.err.println("文件未找到：" + e.getMessage());
} finally {
    // 无论是否发生异常都会执行（资源清理）
    System.out.println("finally 执行");
}
```

#### 方式二：throws 声明向上抛出
```java
public void readFile(String path) throws IOException {
    FileInputStream fis = new FileInputStream(path);
    // 不在这里处理，交给调用者处理
}
```

### 自定义异常

```java
// 自定义受检异常：继承 Exception
public class BusinessException extends Exception {
    private final int code;

    public BusinessException(int code, String message) {
        super(message);
        this.code = code;
    }

    public BusinessException(int code, String message, Throwable cause) {
        super(message, cause);  // 保留异常链
        this.code = code;
    }

    public int getCode() { return code; }
}

// 自定义非受检异常：继承 RuntimeException
public class DataNotFoundException extends RuntimeException {
    public DataNotFoundException(String message) {
        super(message);
    }
}
```

**自定义异常设计建议**：
- 业务异常优先继承 `RuntimeException`（避免强迫调用方处理）
- 保留异常链（传入 `cause`），便于排查根本原因
- 提供有意义的错误码（code）和消息（message）

## 易错点/踩坑

- ❌ "Error 也是 Exception"：Error 和 Exception 是 Throwable 的两个**并列**子类，Error 不是 Exception
- ❌ "RuntimeException 是受检异常"：RuntimeException 及其子类是**非受检**异常，不需要强制处理
- ❌ "catch(Exception e) 能捕获所有异常"：能捕获所有 Exception，但**不能捕获 Error**（如 OOM）；`catch(Throwable t)` 才能全捕
- ❌ "finally 一定会执行"：`System.exit()` 或 JVM 崩溃时 finally 不执行；正常情况下（包括有 return）finally 都会执行
- ❌ "子类重写方法可以抛出更大范围的受检异常"：不能，子类重写方法只能抛出**相同或更小范围**的受检异常，或不抛出
- ❌ "异常被 catch 后就消失了"：若 catch 块中又抛出异常，原异常会丢失（除非用异常链保留）

## 图解/流程

```
Java 异常处理流程：

代码抛出异常
     │
     ▼
JVM 查找调用栈中最近的匹配 catch 块
     │
  找到了？
  ├── 是 → 执行 catch 块 → 执行 finally（若有）→ 继续执行后续代码
  └── 否 → 向上层调用方传播 → 直到找到或到 main()
                                          │
                                     最终未处理
                                          │
                                    JVM 打印堆栈
                                    终止当前线程


受检 vs 非受检 编译器行为：

  方法内出现受检异常
        │
  编译器检查：是否 catch 或 throws？
  ├── 有处理 → ✅ 编译通过
  └── 无处理 → ❌ 编译错误

  方法内出现非受检异常（RuntimeException）
        │
  编译器不检查 → ✅ 编译通过（运行时才暴露）
```
