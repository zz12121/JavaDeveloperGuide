# Stream惰性求值
## Q1：什么是 Stream 的惰性求值？

**A**：惰性求值是指中间操作（filter、map 等）不会立即执行，而是等到终端操作（collect、count 等）被调用时，整个流水线才一次性执行。
**执行方式**是逐元素流过整个流水线：每个元素依次通过所有中间操作到达终端操作，而非先完成一个中间操作再开始下一个。
```java
// 不加终端操作，什么都不执行
stream.filter(...).map(...); // 无输出

// 加上终端操作，逐元素流过
stream.filter(...).map(...).collect(...); // 真正执行
```

---

## Q2：惰性求值有什么优势？

**A**：
1. **性能优化**：不需要创建中间集合存储中间结果，减少内存分配和 GC 压力
2. **短路执行**：配合 `findFirst`、`anyMatch` 等短路操作，处理完必要元素后立即停止
3. **管道融合**：JVM 可以对操作链进行优化（如合并相邻的 map 操作）
4. **按需计算**：无限流只有在 limit 等短路操作限制后才可使用
```java
// 百万数据只需要第一个匹配项，后续元素不会被处理
Optional<User> found = millionUsers.stream()
    .filter(u -> u.getId().equals(targetId))
    .findFirst(); // 找到就停，不会遍历全部
```

---

## Q3：Stream 操作的执行顺序是"从左到右"的吗？

**A**：**是的，对于每个元素来说是从左到右的**。但**不是**"先对所有元素执行第一个操作，再对所有元素执行第二个操作"。
正确理解：**横向逐元素**，而非纵向逐操作。
```
filter → map → collect
  ↓       ↓      ↓
元素1: [不通过]  → 不进入 map
元素2: [不通过]  → 不进入 map
元素3: [通过]   → [执行map] → [收集]
元素4: [通过]   → [执行map] → [收集]
```
这也是为什么在 Stream 中应该**先 filter 再 map**——过滤在前可以减少 map 的调用次数。
