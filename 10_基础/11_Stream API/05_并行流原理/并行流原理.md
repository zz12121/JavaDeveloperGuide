## 核心结论

并行流（parallelStream）利用 JDK 7 引入的 **ForkJoinPool** 将任务拆分为子任务并行执行，最后合并结果。并行流能提升处理大数据集的速度，但并不总是更快——线程切换、合并开销、线程安全等因素可能使其适得其反。

---

## 深度解析

### 1. 工作原理
```
数据源 → Spliterator 拆分 → ForkJoinPool 并行处理 → 合并结果
```

1. **Spliterator** 将数据源拆分为多个小块
2. 每个小块由 ForkJoinPool 中的工作线程处理
3. 各线程的局部结果通过指定的 combiner 合并为最终结果
```java
// 并行流创建
list.parallelStream();        // 从集合直接创建
list.stream().parallel();     // 顺序流转并行流

// 顺序流创建
parallelStream.sequential();  // 并行流转顺序流
```

### 2. ForkJoinPool

- 并行流默认使用 `ForkJoinPool.commonPool()`
- 默认线程数 = `Runtime.getRuntime().availableProcessors() - 1`（CPU 核心数 - 1）
- 所有并行流共享同一个公共线程池，可能互相影响
```java
// 查看默认并行度
System.out.println(ForkJoinPool.commonPool().getParallelism()); // 通常是 CPU 核数-1

// 自定义线程池（不推荐随意修改）
System.setProperty("java.util.concurrent.ForkJoinPool.common.parallelism", "8");
```

### 3. 并行流适用与不适用场景

#### ✅ 适合并行流
- **大数据集**（通常 > 10000 个元素）
- **无状态操作**（filter、map、flatMap）
- **计算密集型**（非 IO 密集型）
- **数据可均匀拆分**（ArrayList 优于 LinkedList）

#### ❌ 不适合并行流
- **小数据集**（线程调度开销 > 并行收益）
- **有状态操作**（sorted、distinct、limit）
- **IO 密集型**（线程被 IO 阻塞，线程池耗尽）
- **顺序敏感操作**（forEach 的执行顺序不保证）
- **需要线程安全的操作**（ArrayList 非线程安全）

### 4. 数据源对并行性能的影响

| 数据源 | 拆分效率 | 说明 |
|--------|----------|------|
| `ArrayList` | ⭐⭐⭐⭐⭐ | 数组结构，可精确拆分 |
| `IntStream.range()` | ⭐⭐⭐⭐⭐ | 范围精确拆分 |
| `HashSet` / `HashMap` | ⭐⭐⭐ | 取决于桶的分布 |
| `LinkedList` | ⭐ | 链表必须遍历才能拆分，性能差 |
| `Stream.iterate()` | ⭐ | 无法预测长度，性能最差 |

### 5. 并行流注意事项

```java
// ⚠️ 并行流中共享状态的陷阱
List<Integer> results = new ArrayList<>(); // 非线程安全！
list.parallelStream().forEach(results::add); // 可能丢失数据或越界

// ✅ 使用 collect 保证线程安全
List<Integer> results = list.parallelStream()
    .collect(Collectors.toList()); // collect 内部处理线程安全

// ⚠️ forEach 不保证顺序
list.parallelStream().forEach(System.out::println); // 输出顺序随机

// ✅ 需要顺序时使用 forEachOrdered
list.parallelStream().forEachOrdered(System.out::println);
```
