## 标准回答思路（STAR）

### S — 现象描述

> "在我们项目中，有一次订单服务的接口响应时间从 200ms 飙升到 3s，TPS 从 500 降到 50。通过监控看到 CPU 使用率不高（20%），但线程数接近上限。"

**关键信号**：
- 响应变慢 + CPU不高 → **大概率是锁等待/IO阻塞**（不是CPU密集）
- 响应变慢 + CPU很高 → **死循环/CAS自旋/GC频繁**
- OOM → **内存泄漏/线程池队列打满**

### T — 定位过程

**Step 1：快速定位阻塞线程**

```bash
# 看线程状态分布
jstack <pid> | grep -c "RUNNABLE"     # 运行中
jstack <pid> | grep -c "BLOCKED"       # 阻塞等待锁
jstack <pid> | grep -c "WAITING"       # 等待条件
jstack <pid> | grep -c "TIMED_WAITING" # 定时等待

# 用 Arthas 在线诊断
thread -n 5           # CPU最高的5个线程
thread -b             # 找出当前阻塞其他线程的"罪魁祸首"
thread --state BLOCKED  # 所有BLOCKED线程
```

**Step 2：分析阻塞原因**

```
jstack输出中重点关注：
├── "waiting to lock <0x...>"   → 等待获取锁，看锁对象是哪个类的哪个实例
├── "locked <0x...>"            → 持有锁，看这个线程在干什么（是否持有时间过长）
├── 死锁特征："waiting to lock" 形成环  → 用 jstack 自带的死锁检测
└── 大量线程WAITING on condition → 可能是线程池任务队列满，线程在等任务
```

**Step 3：确认是否线程安全**

```bash
# 看锁升级状态
# -XX:+PrintSafepointStatistics
# JOL查看对象头：ClassLayout.parseInstance(obj).toPrintable()
```

### A — 根因分析

这次问题的根因是 **synchronized 锁粒度太粗**：

```
// 问题代码：整个方法加锁
public synchronized Order createOrder(OrderRequest req) {
    // 1. 参数校验（不需要锁）
    // 2. 库存扣减（需要锁）
    // 3. 创建订单（需要锁）
    // 4. 发送消息通知（不需要锁，IO操作）
    // 5. 记录日志（不需要锁）
}
```

锁粒度分析：
- 临界区实际只需要保护 **步骤2+3**（约5ms）
- 但步骤4的MQ发送可能耗时 100-500ms（IO）
- 整个方法加锁 → 每个请求白白多等 100-500ms
- 高并发下形成排队 → 响应时间线性增长

### R — 解决方案

```java
// 方案1：缩小锁范围（最小改动）
public Order createOrder(OrderRequest req) {
    validate(req);           // 不加锁
    
    synchronized(lock) {     // 只锁必要部分
        deductStock(req);
        Order order = buildOrder(req);
        return order;
    }
    // notify 和 log 放到锁外面
}

// 方案2：用 ReentrantLock + tryLock 实现超时（推荐）
private ReentrantLock orderLock = new ReentrantLock();

public Order createOrder(OrderRequest req) {
    validate(req);
    if (!orderLock.tryLock(3, TimeUnit.SECONDS)) {
        throw new BizException("系统繁忙，请稍后重试");
    }
    try {
        deductStock(req);
        return buildOrder(req);
    } finally {
        orderLock.unlock();
    }
}

// 方案3：用 ConcurrentHashMap + AtomicLong 做分段（数据量大时）
// 方案4：用分布式锁（多实例部署时）— Redis/Zookeeper
```

**优化效果**：单次请求锁持有时间从 ~500ms 降到 ~5ms，TPS 从 50 恢复到 800+。

## 可能的追问链

### 追问1：你怎么判断是锁竞争而不是GC导致的？

> "首先看 GC 日志（`-XX:+PrintGCDetails`），如果 Full GC 频率正常（比如几小时一次），说明不是 GC 问题。然后再看 jstack 输出中 BLOCKED 线程数量，如果大量线程都在 `waiting to lock <同一个对象>`，基本可以确认是锁竞争。另外也可以用 Arthas 的 `thread -b` 直接找出阻塞源头。"

→ 相关：[[JVM-GC分析]]

### 追问2：锁升级到重量级锁后，对性能影响有多大？

> "主要影响是线程上下文切换。轻量级锁竞争时线程在用户态自旋，不涉及内核态切换。升级到重量级锁后，竞争失败的线程进入 BLOCKED 状态，需要从用户态切换到内核态，一次切换大约消耗 1-10 微秒。如果竞争激烈、频繁切换，开销会非常大。"

→ 相关：[[锁升级全过程]]

### 追问3：用ReentrantLock替代synchronized后，有什么需要注意的？

> "三点：第一，必须在 finally 里 unlock，否则异常时锁不释放；第二，ReentrantLock 支持公平锁和非公平锁，默认非公平，性能更好但可能饥饿；第三，ReentrantLock 支持 tryLock 超时获取，可以避免死等，这是 synchronized 做不到的。"

→ 相关：[[Lock vs synchronized]]

### 追问4：如果线程池满了会怎样？

> "如果用的是 ThreadPoolExecutor，任务会被拒绝策略处理：默认 AbortPolicy 抛异常，生产环境建议用 CallerRunsPolicy 让调用线程自己执行，起到限流效果。如果用的是 Executors 工厂方法，newFixedThreadPool 和 newSingleThreadExecutor 的队列是 LinkedBlockingQueue(Integer.MAX_VALUE)，基本不会满，但会导致任务堆积、OOM。所以生产环境禁止用 Executors。"

→ 相关：[[线程池工作流程]]、[[线程池拒绝策略]]

### 追问5：CopyOnWriteArrayList 能解决这个场景吗？

> "要看读写比例。CopyOnWriteArrayList 写操作会复制整个数组，写多读少场景下性能很差。我们这个订单场景是写多读少（创建订单多，查询少），不适合。适合的场景比如白名单列表、配置项列表这种读多写极少的情况。"

→ 相关：[[CopyOnWriteArrayList]]
